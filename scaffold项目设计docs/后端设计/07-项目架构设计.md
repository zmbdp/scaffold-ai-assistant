### 7.0.4 关于 ToolConfig 改名为 ToolRegistryService 的说明

> **设计决策**：原设计中的 `ToolConfig` 类放在 `config/` 包下，标注 `@Configuration`。经审视，此设计不合理，改为 `ToolRegistryService` 放在 `service/` 包下，标注 `@Service`。

**原因**：
1. **`@Configuration` 的语义是"Bean 定义类"**——通常用于 `@Bean` 方法声明、第三方框架集成配置等**静态** Bean 定义，不应包含业务逻辑。
2. **`ToolRegistryService` 的职责是"通过 Feign 调用 `ArgumentServiceApi` 查 `sys_argument` 表动态过滤启用工具"**——这是典型的业务逻辑（Feign 调用 + 过滤 + 注册），属于 Service 层职责。
3. **业内主流做法**（如 Spring AI 官方示例、LangChain4j）都将工具注册/发现逻辑放在 Service 层，因为：
   - 工具启停需要动态调整（管理员通过 B端配置）
   - 工具注册依赖 Feign 调用结果
   - 需要支持运行时热加载（不重启应用更新工具列表）
4. **改名 `ToolRegistryService` 更符合命名语义**——"Registry" 表达"注册中心"含义，比 "Config" 更准确。

**改动点**：
- 文件位置：`config/ToolConfig.java` → `service/ToolRegistryService.java`
- 注解：`@Configuration` → `@Service`
- 依赖注入：通过 Feign 注入 `ArgumentServiceApi` 查询 `sys_argument` 表的 `ai.tool.*.enabled` 配置项

### 7.0.5 portal-service 和 admin-service 需新增的组件

> **重要说明**：portal-service 和 admin-service 是**脚手架已有服务**（非新增服务），但为支持 AI 功能，需在这两个服务中**新增**以下组件。7.0.2 节列出的是这两个服务可**复用**的脚手架组件，本节列出的是**需要新增**的 AI 相关组件。

#### 7.0.5.1 portal-service 需新增的组件

> **职责定位**：portal-service 作为 C端统一入口，**只做透传和业务编排**（如：先调用 RAG 检索接口获取 context，再拼接 Prompt，最后 WebClient 调用 chat-service 的 SSE 端点）。**RAG 检索本身在 chat-service 做**，portal-service 不实现 RAG。

| 类型 | 组件 | 说明 |
|-----|------|------|
| **Controller** | `ChatPortalController` | C端对话入口（`/chat/completions/stream`、`/chat/image/completions/stream`），透传 SSE 流 |
| | `HistoryPortalController` | C端对话历史入口（`/history`、`/history/{sessionId}`） |
| | `FeedbackPortalController` | C端回答反馈入口（`/feedback`、`/feedback/{conversationId}`） |
| **Service** | `IChatPortalService` / `ChatPortalServiceImpl` | C端对话业务编排：①Feign 调用 `ChatApi.retrieveContext()` 获取 RAG context → ②拼接 Prompt → ③WebClient 调用 chat-service SSE 端点 → ④透传 `Flux<String>` 给前端 |
| | `IHistoryPortalService` / `HistoryPortalServiceImpl` | C端对话历史业务编排：Feign 调用 `HistoryApi` |
| | `IFeedbackPortalService` / `FeedbackPortalServiceImpl` | C端反馈业务编排：Feign 调用 `FeedbackApi` |
| **Config** | `WebClientConfig` | WebClient 配置（连接池、超时、traceId 传递过滤器），通过 ExchangeFilterFunction 从 MDC 提取 traceId 写入 X-Trace-Id 请求头，实现跨服务链路追踪（详见 03-C端功能设计.md 3.0.1 节） |

#### 7.0.5.2 admin-service 需新增的组件

| 类型 | 组件 | 说明 |
|-----|------|------|
| **Controller** | `KnowledgeController` | B端知识库管理入口（`/knowledge/sources/**`、`/knowledge/documents/**`） |
| | `AiConfigController` | B端 AI 配置管理入口（`/ai/config`、`/ai/models`） |

### 7.1.2 模块职责说明

| 模块 | 职责 | 包含内容 |
|-----|------|---------|
| **zmbdp-chat-api** | 定义AI核心能力的Feign接口和数据结构，供portal-service和admin-service调用 | DTO（请求对象）、VO（响应对象）、Feign接口定义（非流式）、枚举类 |
| **zmbdp-chat-service** | AI核心能力实现（RAG检索、Agent工具调用、对话管理），提供Feign接口（同步）和HTTP SSE端点（流式） | Controller（实现Feign接口+SSE端点）、Service（业务逻辑）、Mapper（数据访问）、Entity（数据库实体）、Config（配置类）、Tool（Agent工具） |
| **zmbdp-portal-service** | C端用户入口，提供统一对话接口，承担业务编排职责（RAG检索→Prompt拼接→流式调用） | Controller（HTTP入口）、Service（业务编排，包含WebClient调用）、Feign调用（chat-api，非流式） |
| **zmbdp-admin-service** | B端管理入口，代理AI管理接口，复用现有用户管理体系 | Controller（HTTP入口）、Service（业务编排）、Feign调用（chat-api） |

**调用关系**：

> **架构原则**：AI对话**必须使用流式输出**（WebClient + SSE），**不使用Feign**。
> Feign仅用于同步的非对话接口（RAG检索、历史记录、知识库管理等）。

**流式对话流程**（核心流程，portal-service承担业务编排）：
```
前端 → Gateway → zmbdp-portal-service /chat/completions/stream
                    ↓
              1. 解析用户提问，获取sessionId
              2. Feign调用 → zmbdp-chat-api /chat/retrieve（RAG检索，获取context）
              3. 拼接Prompt（system prompt + context + 用户提问）
              4. WebClient调用 → zmbdp-chat-service /chat/completions/stream（SSE端点）
              5. 接收Flux<String>流数据，透传给前端
                    ↓
              返回SSE流数据
```

**管理接口流程**：
```
前端 → Gateway → zmbdp-admin-service /admin/knowledge/**
                    ↓
              Feign调用 → zmbdp-chat-api → zmbdp-chat-service
                    ↓
              返回管理结果
```

**核心设计原则**：
1. **职责单一**：chat-service只做AI相关的核心能力，提供Feign接口（同步）和HTTP SSE端点（流式）
2. **分层调用**：portal-service作为C端统一入口，负责认证、业务编排（RAG检索→Prompt拼接）和流透传；admin-service作为B端统一入口，负责权限控制和管理功能
3. **多模型扩展**：通过ModelType枚举和ModelService接口预留多模型切换能力
4. **流式支持**：流式对话使用HTTP SSE端点 + WebClient调用，避免Feign的流式限制

### 7.2.13 IChatService / ChatServiceImpl

**类名**：`IChatService`（接口） / `ChatServiceImpl`（实现类）

**职责**：AI对话核心逻辑，负责流式调用大模型、工具调用编排、对话记忆管理和对话记录持久化

**路径**：
- 接口：`com.zmbdp.chat.service.service.IChatService`
- 实现：`com.zmbdp.chat.service.service.ChatServiceImpl`

> **重要**：ChatService 仅负责**流式对话**（返回 `Flux<String>`），不提供同步对话方法。
> 同步的 RAG 检索由 `ChatController.retrieveContext()` 直接委托给 `VectorStoreService` 完成。
> Prompt 拼接由 `portal-service` 的 `IChatPortalService` 完成，chat-service 接收已拼接好的 `prompt`。

**依赖注入**：
- `IModelService` - 获取 ChatClient 和 EmbeddingModel
- `IChatMemoryService` - 对话记忆管理（Redis）
- `IHistoryService` - 对话记录持久化（MySQL）
- `ToolRegistryService` - 获取已启用的 Agent 工具列表（`@Service`，通过 Feign 调用 `ArgumentServiceApi` 查 `sys_argument` 表的 `ai.tool.*.enabled` 配置项）
- `ArgumentServiceApi`（Feign） - 读取 AI 运行时配置（`ai.temperature`、`ai.enable_tools` 等，复用脚手架 `sys_argument` 表）
- `SnowflakeIdService` - 生成对话记录 ID

**核心方法**：

**streamChat()**：
```java
Flux<String> streamChat(ChatStreamReqDTO request);
```
- **参数**：`ChatStreamReqDTO` - 包含已拼接好的完整 prompt、sessionId、model、temperature
- **返回值**：`Flux<String>` - SSE 流式响应，每帧为 JSON 格式
- **说明**：流式文本对话，由 ChatStreamController 调用

**streamChatWithImage()**：
```java
Flux<String> streamChatWithImage(ChatWithImageStreamReqDTO request);
```
- **参数**：`ChatWithImageStreamReqDTO` - 包含 prompt、images、sessionId、model
- **返回值**：`Flux<String>` - SSE 流式响应
- **说明**：流式图文对话，自动选择支持图片的模型

**streamChat() 方法执行流程**：
```
输入：ChatStreamReqDTO（已包含 portal-service 拼接好的完整 prompt）
  │
  ├─ Step 1: 解析参数，确定使用的模型
  │   ├─ 如果 request.model 为空，从 Nacos 配置读取默认模型（模型名称属基础设施配置，由 Nacos 管理）
  │   └─ 通过 ModelService.getChatClient(model) 获取 ChatClient
  │
  ├─ Step 2: 获取对话历史（如果 sessionId 不为空）
  │   └─ ChatMemoryService.getHistory(sessionId) → List<ChatMessage>
  │
  ├─ Step 3: 构建消息列表
  │   ├─ 将历史消息添加到上下文
  │   └─ 使用 request.prompt 作为当前用户消息
  │
  ├─ Step 4: 判断是否需要工具调用（通过 ChatClient 的函数调用能力）
  │   ├─ 如果 enable_tools=true 且模型支持 FUNCTION_CALL
  │   │   └─ 配置工具列表（从 ToolRegistryService 获取已启用的工具）
  │   └─ 否则跳过工具调用
  │
  ├─ Step 5: 调用 ChatClient.stream() 流式生成
  │   ├─ chatClient.prompt(messages).stream().content()
  │   └─ 返回 Flux<String>（每个元素为一个 token 片段）
  │
  ├─ Step 6: 流式输出完成后，异步执行后处理
  │   ├─ 拼接完整回答内容
  │   ├─ 保存对话历史到 Redis：ChatMemoryService.addMessage(sessionId, ...)
  │   ├─ 记录对话到 MySQL：sys_ai_conversation 表（含 model、response_time、token_usage）
  │   └─ 如果使用了工具，记录 tool_usage 信息
  │
  └─ 返回 Flux<String>（SSE 格式，每帧包含 chunk、done 等字段）
```

**SSE 响应帧格式**：
```json
// 内容帧
{"chunk": "三级缓存架构包含", "done": false}

// 工具调用开始帧（AI决定调用工具时发送）
{"toolCall": {"name": "ReadFileTool", "args": {"filePath": "/path/to/file"}}, "done": false}

// 工具调用结果帧（工具执行完成后发送，与03-C端功能设计.md 3.2.1节、05-Agent工具设计.md 5.3.3节一致）
{"toolResult": {"name": "ReadFileTool", "success": true, "summary": "读取文件成功，共123行", "duration": 45}, "done": false}

// 结束帧
{"chunk": "", "done": true, "sessionId": "abc123", "sources": [...], "model": "deepseek-v4-flash"}
```

**错误处理**：
- 模型调用失败：返回错误帧 `{"error": "AI服务连接失败", "code": 500023, "done": true}`
- 工具调用失败：记录错误但继续生成回答（不中断流）
- 超时（60秒无响应）：返回超时帧 `{"error": "AI服务响应超时", "code": 500024, "done": true}`

**syncKnowledge执行流程**：
```
1. 从 sys_ai_knowledge_source 表查询所有 enabled=1 的知识源
2. 遍历每个知识源（knowledgeSourceId 为表的主键 id，雪花算法生成）：
   ├─ 获取分布式锁 redissonLockService.lock("knowledge:sync:" + knowledgeSourceId)
   │   └─ 按知识源粒度加锁，防止同一知识源的并发同步冲突（不同知识源可并行同步）
   │   └─ 获取失败 → 跳过该知识源（说明已有同步任务在执行），继续处理下一个
   ├─ 遍历该知识源目录，收集当前所有文件路径
   ├─ 从 sys_ai_document 表查询该知识源的已有文档列表
   ├─ 对比文件路径，识别三种状态：
   │   ├─ 新增文件：当前目录有，MySQL 没有
   │   ├─ 更新文件：当前目录有，MySQL 有，但 SHA-256 哈希不同
   │   └─ 删除文件：当前目录没有，MySQL 有
   ├─ 处理新增文件：
   │   ├─ 读取文件内容，计算 SHA-256 哈希
   │   ├─ 调用 chunkDocument() 分块
   │   ├─ 插入 sys_ai_document 表，获取文档ID
   │   └─ 调用 vectorStoreService.addDocuments() 写入 Milvus
   ├─ 处理更新文件：
   │   ├─ 读取文件内容，计算 SHA-256 哈希
   │   ├─ 更新 sys_ai_document 表，version+1
   │   ├─ 调用 vectorStoreService.deleteByDocumentId() 删除旧分块
   │   ├─ 调用 chunkDocument() 重新分块
   │   └─ 调用 vectorStoreService.addDocuments() 写入新分块
   ├─ 处理删除文件：
   │   ├─ 调用 vectorStoreService.deleteByDocumentId() 删除 Milvus 分块
   │   └─ 更新 sys_ai_document 表 status='DELETED'（软删除）
   ├─ 若 force=true：跳过哈希检查，所有文件视为更新文件重新处理
   └─ 释放分布式锁
3. 记录操作日志（通过 `@LogAction` 注解自动记录到脚手架 `operation_log` 表）
4. 返回 SyncResultVO（含各项统计数量）
```

```
   ├─ 返回 false → document_id 不存在，直接返回（避免无效的 Milvus 查询）
```

> **不涉及缓存失效**：向量检索结果不缓存（详见 7.12.1 节），因此删除文档时无需失效任何缓存。

> **布隆过滤器说明**（复用 `zmbdp-common-filter` 的 `BloomFilterService`）：
> - 推荐使用 `RedisBloomFilterService` 实现（配置 `bloom.filter.type=redis`），支持分布式
> - 知识同步时 `addDocuments` 将 document_id 加入布隆过滤器
> - 删除文档时先 `mightContain` 快速过滤，避免对不存在的 document_id 执行无效的 Milvus 删除操作
> - 布隆过滤器有假阳性（可能误判存在），但不会假阴性（说不存在就一定不存在），适合做前置过滤
> - **Redis key 命名**：布隆过滤器在 Redis 中的 key 由 `zmbdp-common-filter` 模块内部约定（通常为 `bloom:filter:{filterName}`），AI 模块通过 `BloomFilterService` API 调用，不直接感知 key 格式。AI 模块创建的布隆过滤器建议命名为 `document_ids`（即完整 key 为 `bloom:filter:document_ids`）

- **参数**：无
- **返回值**：`long` - Milvus 中分块总数
- **说明**：获取知识库文档分块数量

**initCollection()**：
```java
void initCollection()
```
- **参数**：无
- **返回值**：无
- **说明**：幂等初始化 Milvus 集合和索引（集合不存在则创建，索引不存在则创建）
- **索引创建清单**（与 06-RAG流程设计.md 6.2.2 节一致）：
  - 向量索引：HNSW，metric_type=IP，efConstruction=200，M=16，efRuntime=100
  - 标量索引：对 `document_id`、`source_type`、`module`、`category` 建立标量索引（通过 Milvus 2.4+ JSON 字段部分键索引，支持 `source_type == "doc"` 风格的过滤表达式）

**addDocuments执行流程**：
```
1. 遍历传入的 documents 列表
2. 对每个 document：
   ├─ 调用 modelService.getEmbeddingModel().embed(content) 生成向量
   │   └─ 返回 float[]（维度1536，text-embedding-v1）
   ├─ 构建 Milvus InsertReq：
   │   ├─ id: snowflakeIdService.nextId()
   │   ├─ content: 分块内容
   │   ├─ document_id: 关联 MySQL 文档ID
   │   ├─ chunk_index: 分块序号
   │   ├─ metadata: 元数据 Map（source_type、source_path、module、category、title、create_time）
   │   ├─ embedding_model: "text-embedding-v1"
   │   └─ vector: float[] 向量
   └─ 执行 milvusClient.insert()
3. 将 document_id 加入布隆过滤器（bloomFilterService.put）
   └─ 后续检索/删除时可用 mightContain 快速判断 document_id 是否存在，避免无效的 Milvus 查询
4. 返回（不抛异常即视为成功）
```

**search执行流程**：
```
1. 调用 modelService.getEmbeddingModel().embed(query) 生成查询向量
2. 构建 Milvus SearchReq：
   ├─ collection_name: "scaffold_knowledge"
   ├─ vector field: 查询向量
   ├─ top_k: 请求的 topK * 扩大因子（如 topK*3，作为重排序候选集）
   ├─ metric_type: IP（内积）
   └─ params: {"ef": 100}（HNSW查询参数）
3. 执行 milvusClient.search()
4. 解析返回结果：
   ├─ 提取 id、content、document_id、chunk_index、metadata
   ├─ 计算 score（相似度分数）
   └─ 构建 DocumentVO 候选列表
5. Reranking（重排序）：
   ├─ 调用 RerankModel 对候选文档重新打分（基于 query 与文档语义相关性）
   └─ 按 rerank score 降序排序，截取最终 topK 条
6. 按 score 降序排序
7. 返回结果
```

> **不经过二级缓存**：向量检索结果不缓存（详见 7.12.1 节"缓存范围决策"），每次都实时查询 Milvus，保证检索结果基于最新的向量库数据。

### 7.2.18 IChatMemoryService / ChatMemoryServiceImpl

**类名**：`IChatMemoryService`（接口） / `ChatMemoryServiceImpl`（实现类）

**职责**：对话记忆管理，基于 Redis List 存储；实现类 `ChatMemoryServiceImpl` 实现 Spring AI 的 `ChatMemory` 接口以支持流式对话的上下文记忆

> **Spring AI ChatMemory 接口方法映射**：
> - `ChatMemory.add(conversationId, messages)` → 内部调用 `addMessages(sessionId, messages)` 写入 Redis List + invalidateL1Only
> - `ChatMemory.get(conversationId, lastN)` → 内部调用 `getHistory(sessionId)` 获取完整列表，截取最后 N 条返回
> - `ChatMemory.clear(conversationId)` → 内部调用 `clearHistory(sessionId)` 删 Redis List + invalidateL1Only
>
> `IChatMemoryService` 接口对外暴露 `addMessage`/`addMessages`/`getHistory`/`clearHistory` 自定义方法，实现类同时覆盖 ChatMemory 接口的 `add`/`get`/`clear` 方法（内部委托给自定义方法），供 Spring AI ChatClient 自动注入使用。

**路径**：
- 接口：`com.zmbdp.chat.service.service.IChatMemoryService`
- 实现：`com.zmbdp.chat.service.service.ChatMemoryServiceImpl`

**依赖注入**：
- `RedisService` - Redis 操作封装（数据存储 List 读写、TTL 管理）
- `JsonUtil` - 序列化工具（ChatMessage ↔ JSON）
- `SysAiConversationMapper` - DB 降级时直接查询 MySQL（不经过 IHistoryService 避免循环依赖）
- `CacheInvalidateProducer` - 缓存失效广播（invalidateL1Only）
- `Cache<String, Object> caffeineCache` - L1 Caffeine 缓存实例（由 `CaffeineConfig` 注册）

**Redis 存储结构**：
```
Key: ai:chat:memory:{sessionId}    （数据存储，List 结构）
Type: List
Value: JSON 序列化的 ChatMessage 列表（lPush 头部插入，lRange 读取后反转）
TTL: 7天（604800秒）
```

> **关键设计——数据存储与 L1 缓存 key 分离**：
> - **数据存储 key**：`ai:chat:memory:{sessionId}`（Redis List，TTL=7天，长期保存对话数据）
> - **L1 Caffeine 缓存 key**：`CHAT_MEMORY:ai:chat:memory:{sessionId}`（内存对象副本，TTL=5分钟）
> - **CHAT_MEMORY 不走 L2 缓存失效流程**：Redis List 本身是数据存储而非缓存，`addMessage` 通过 `lPush` 直接更新它（数据已是最新），只需 MQ 广播删 L1 Caffeine 即可。这与 `AI_CONFIG`（MySQL 为数据源、Redis 为 L2 缓存）的失效语义不同。

**核心方法**：

**getHistory()**：
```java
List<ChatMessage> getHistory(String sessionId)
```
- **参数**：`sessionId` - 会话ID
- **返回值**：`List<ChatMessage>` - 对话历史列表（按时间正序）
- **说明**：获取会话的对话历史

**addMessage()**：
```java
void addMessage(String sessionId, ChatMessage message)
```
- **参数**：`sessionId` - 会话ID，`message` - 消息对象（含 role、content、timestamp）
- **返回值**：无
- **说明**：添加一条消息到对话历史

**addMessages()**：
```java
void addMessages(String sessionId, List<ChatMessage> messages)
```
- **参数**：`sessionId` - 会话ID，`messages` - 消息列表
- **返回值**：无
- **说明**：批量添加消息到对话历史

**clearHistory()**：
```java
void clearHistory(String sessionId)
```
- **参数**：`sessionId` - 会话ID
- **返回值**：无
- **说明**：清除会话的对话历史（删除 Redis List 数据存储 + MQ 广播删 L1 Caffeine）

**缓存策略**（详见 7.12 节）：
- **数据存储（Redis List）**：key=`ai:chat:memory:{sessionId}`，TTL=7 天，每会话最多 100 条消息
- **L1 Caffeine 缓存**：key=`CHAT_MEMORY:ai:chat:memory:{sessionId}`，缓存反序列化后的 `List<ChatMessage>` 对象，TTL=5 分钟，最大 1000 条
- **无独立 L2 缓存**：Redis List 本身即数据存储，不另设 L2 缓存层（避免"缓存"与"数据存储"key 混淆）
- **序列化**：使用 `JsonUtil.toJson()` / `JsonUtil.fromJson()` 序列化 ChatMessage 对象
- **DB 降级**：Redis 不可用时降级查 MySQL `sys_ai_conversation` 表（直接查 `SysAiConversationMapper`，不经过 IHistoryService 避免循环依赖）
- **缓存失效**：`addMessage`/`addMessages` 通过 MQ 广播删除所有实例的 L1 Caffeine 缓存；`clearHistory` 删除 Redis List + MQ 广播删 L1

**addMessage执行流程**：
```
1. 构建 Redis 数据存储 key: AiCacheConstants.CHAT_MEMORY_REDIS_PREFIX + sessionId（ai:chat:memory:{sessionId}）
2. 使用 JsonUtil.toJson(message) 序列化消息对象
3. 调用 redisService.lPush(key, jsonMessage) 添加到列表头部
4. 检查列表长度：redisService.lLen(key)
   └─ 若超过 100，使用 redisService.lTrim(key, 0, 99) 裁剪
5. 刷新数据存储 TTL：redisService.expire(key, AiCacheConstants.CHAT_MEMORY_REDIS_TTL_SECONDS)（7天）
6. 失效 L1 Caffeine（不删 Redis 数据存储）：
   cacheInvalidateProducer.invalidateL1Only(CHAT_MEMORY, key)
   └─ MQ 广播通知所有实例删除 L1 Caffeine 中的 "CHAT_MEMORY:{key}"
   注：CHAT_MEMORY 不调用 invalidate()，因为 Redis List 已通过 lPush 更新为最新数据，无需删除
```

**getHistory执行流程**：
```
1. 构建 Redis 数据存储 key: AiCacheConstants.CHAT_MEMORY_REDIS_PREFIX + sessionId
2. 构建 L1 Caffeine key: CHAT_MEMORY + ":" + key
3. 查 L1 Caffeine（getIfPresent）→ 命中则直接返回 List<ChatMessage>
4. 查 Redis 数据存储：
   ├─ 调用 redisService.lRange(key, 0, -1) 获取全部消息
   ├─ 若 List 非空：
   │   ├─ 遍历列表，使用 JsonUtil.fromJson(json, ChatMessage.class) 反序列化
   │   ├─ 反转列表（Redis List 是 lPush 反向存储，需转为时间正序）
   │   ├─ 回填 L1 Caffeine（put，TTL=5分钟）
   │   └─ 返回 List<ChatMessage>
   └─ 若 List 为空（Redis 未命中或已过期）：
       ├─ DB 降级：直接查 SysAiConversationMapper（从 MySQL sys_ai_conversation 表查询，不经过 IHistoryService 避免循环依赖）
       ├─ 若 DB 有数据：按时间正序逐条 lPush 回填 Redis List + 回填 L1 Caffeine，返回结果
       └─ 若 DB 也无数据：返回空列表（新会话）
5. 返回 List<ChatMessage>
```

**clearHistory执行流程**：
```
1. 构建 Redis 数据存储 key: AiCacheConstants.CHAT_MEMORY_REDIS_PREFIX + sessionId
2. 删除 Redis 数据存储：redisService.deleteObject(key)
3. 失效 L1 Caffeine：
   cacheInvalidateProducer.invalidateL1Only(CHAT_MEMORY, key)
   └─ MQ 广播通知所有实例删除 L1 Caffeine 中的 "CHAT_MEMORY:{key}"
```

| getToolUsageStats | `stats:tools` | 300秒 |
| getAiCallMetrics | `stats:ai_metrics` | 300秒 |

```
3. 缓存未命中，执行数据库聚合查询：
   ├─ 总对话数：SELECT COUNT(*) FROM sys_ai_conversation WHERE status='SUCCESS'
   ├─ 今日对话数：SELECT COUNT(*) FROM sys_ai_conversation WHERE status='SUCCESS' AND create_date=今日
   ├─ 平均响应时间：SELECT AVG(response_time) FROM sys_ai_conversation WHERE status='SUCCESS'
   ├─ 活跃用户数：SELECT COUNT(DISTINCT user_id) FROM sys_ai_conversation WHERE create_date>=近7天
   └─ 趋势数据：按日期分组统计近7天每日对话数
4. 构建 ConversationStatisticsVO 对象
5. 写入缓存（TTL=300秒）
6. 返回结果
```

```
6. 构建 KnowledgeSourceVO 返回
```

**deleteSource执行流程**：
```
1. 查询知识源下所有文档（sys_ai_document WHERE knowledge_source_id=#{id}）
2. 遍历每个文档：
   ├─ 调用 vectorStoreService.deleteByDocumentId(documentId) 删除 Milvus 分块
   └─ 更新 sys_ai_document status='DELETED'（软删除）
3. 删除知识源：UPDATE sys_ai_knowledge_source SET enabled=0 WHERE id=#{id}
4. 返回 void
```

**sync执行流程**：
```
1. 委托给 knowledgeLoaderService.syncKnowledge(dto.force)
2. 返回 SyncResultVO
```

### 7.2.24 chat-service 内部接口鉴权机制

**说明**：脚手架不使用 Spring Security 的 `SecurityConfig`，认证统一由 Gateway 层的 `AuthFilter` 完成。`zmbdp-chat-service` 不直接暴露给前端，不经过网关，因此**不需要 JWT 认证**。

**鉴权机制**：

| 接口类型 | 调用方 | 鉴权方式 | 说明 |
|---------|-------|---------|------|
| Feign 接口（同步） | portal-service、admin-service | 无需鉴权 | 依靠 Docker 内部网络隔离，仅同网络内的服务可调用 |
| HTTP SSE 端点（流式） | portal-service（WebClient） | 无需鉴权 | 同上，内部网络隔离 |
| `/actuator/**` | 运维监控 | 无需鉴权 | 健康检查接口，不暴露敏感信息 |

**安全保障**：
1. **网络隔离**：chat-service 仅在 Docker 内部网络中可见，外部无法直接访问 `http://{chat-service-ip}:18084`
2. **网关统一鉴权**：前端请求经 Gateway 时，由 AuthFilter 统一完成 JWT 校验和用户来源检查
3. **服务信任**：portal-service 和 admin-service 已通过 Gateway 鉴权，chat-service 信任内部调用方
4. **无需引入 `share-token` 配置**：chat-service 不做 Token 校验，但为保持与脚手架一致（`TokenService` 可能被其他组件依赖），仍建议引入 `share-token` 共享配置

**注意**：如果未来需要 chat-service 直接暴露给前端（不推荐），则需要新增独立的鉴权机制。当前架构下，所有前端请求必须经 Gateway → portal-service/admin-service → chat-service。

### 7.2.25 TokenService（使用项目现有）

**类名**：`TokenService`

**职责**：Token的创建、验证、刷新、删除，基于JWT和Redis实现

**路径**：`com.zmbdp.common.security.service.TokenService`

**说明**：AI模块直接使用项目现有的 `TokenService`，无需重新定义。

**核心方法**：

**createToken()**：
```java
public TokenDTO createToken(LoginUserDTO loginUserDTO, String secret)
```
- **参数**：`LoginUserDTO` 用户信息、`secret` JWT签名密钥
- **返回值**：`TokenDTO`（包含 `accessToken` 和 `expires`）
- **说明**：为用户创建登录Token，自动存储到Redis

**getLoginUser()**：
```java
public LoginUserDTO getLoginUser(String token, String secret)
```
- **参数**：JWT Token、签名密钥
- **返回值**：`LoginUserDTO` 用户信息
- **说明**：从Token中解析用户信息，需同时验证JWT和Redis

**delLoginUser()**：
```java
public void delLoginUser(String token, String secret)
```
- **参数**：JWT Token、签名密钥
- **说明**：删除用户登录状态（从Redis中移除）

**verifyToken()**：
```java
public void verifyToken(LoginUserDTO loginUserDTO)
```
- **参数**：`LoginUserDTO` 用户信息
- **说明**：验证并续期Token（剩余时间少于120分钟自动续期）

**Token配置**：

| 配置项 | 值 | 说明 |
|-------|-----|------|
| `expireTime` | 720分钟（12小时） | Token在Redis中的过期时间 |
| `refreshTime` | 120分钟 | 续期阈值，剩余时间少于此时自动续期 |
| `signingAlgorithm` | HS512 | 签名算法（与 `JwtUtil.java` 实际实现一致） |
| `signingKey` | 配置在Nacos | 签名密钥（`jwt.token.secret`，位于 `share-token-${env}.yaml`） |

**Redis Key格式**：`logintoken:{uuid}`

**使用示例（登录）**：
```java
LoginUserDTO loginUserDTO = new LoginUserDTO();
loginUserDTO.setUserId(user.getId());
loginUserDTO.setUserName(user.getUsername());
// userFrom 由后端根据登录入口硬编码设置（非前端传入）
// B端登录：loginUserDTO.setUserFrom(UserConstants.USER_FROM_TU_B);  // "sys"
// C端登录：loginUserDTO.setUserFrom(UserConstants.USER_FROM_TU_C);  // "app"

TokenDTO tokenDTO = tokenService.createToken(loginUserDTO, jwtSecret);
LoginVO loginVO = new LoginVO();
loginVO.setToken(tokenDTO.getAccessToken());
loginVO.setExpireTime(System.currentTimeMillis() + tokenDTO.getExpires() * 60 * 1000);
loginVO.setUser(userVO);
return Result.success(loginVO);
```

**UserConstants 用户来源常量**：
- `UserConstants.USER_FROM_TU_B = "sys"` - B端用户（管理员）
- `UserConstants.USER_FROM_TU_C = "app"` - C端用户（前端用户）

**说明**：userFrom 字段由后端根据登录入口硬编码设置（C端登录入口 `UserServiceImpl#login` 中固定 `setUserFrom(USER_FROM_TU_C)`，B端登录入口固定 `setUserFrom(USER_FROM_TU_B)`），前端不传入该字段。Gateway 的 AuthFilter 通过 JWT 中的 userFrom 字段做来源校验。

### 7.2.26 B端AI接口的权限控制（AuthFilter 统一校验）

> **重要说明**：脚手架代码库中**不存在** `@RequiresPermission` 注解，也未实现细粒度 RBAC 权限控制。sys_user 表的 `identity` 字段（`super_admin`/`platform_admin`）目前仅作标识存储，没有对应的权限校验逻辑。AI 模块的 B 端接口权限控制统一由 Gateway 的 AuthFilter 完成。

**v1.0 AuthFilter 的实际行为**（详见 03-C端功能设计.md）：

```java
String userFrom = JwtUtil.getUserFrom(claims);
// v1.0 已将 url.contains("sys_user") 扩展为统一对所有 /admin/** 路径做来源校验
if (url.startsWith("/admin/") && !isWhitelist(url) &&
    !UserConstants.USER_FROM_TU_B.equals(userFrom)) {    // 仅当 userFrom != "sys" 时拒绝
    return unauthorizedResponse(exchange, ResultCode.TOKEN_CHECK_FAILED);
}
```

- v1.0 AuthFilter 已将 `url.contains("sys_user")` 扩展为 `url.startsWith("/admin/") && !isWhitelist(url)`，统一对所有 `/admin/**` 路径（白名单除外）做 `userFrom="sys"` 校验
- AI 模块的 `/admin/knowledge/**`、`/admin/ai/**`、`/admin/statistics/**`、`/admin/tools/**` 路径已被 `/admin/` 前缀匹配覆盖，**无需额外扩展**
- 其他所有非 `/admin/**` 路径（如 `/portal/user/**`）只要有有效 Token 即可访问，不区分用户来源

**权限控制策略**：

| 接口路径 | C 端用户（userFrom=app） | B 端用户（userFrom=sys） | 实现方式 |
|---------|----------------------|----------------------|---------|
| `/portal/chat/**`、`/portal/history/**`（C端 AI 接口） | ✅ 放行 | ✅ 放行 | 非 `/admin/**` 路径，AuthFilter 不拦截 |
| `/admin/knowledge/**`、`/admin/ai/**`、`/admin/statistics/**`、`/admin/tools/**` | ❌ 拒绝 | ✅ 放行 | v1.0 AuthFilter 统一校验 |
| `/admin/sys_user/**`（现有 B 端用户管理） | ❌ 拒绝 | ✅ 放行 | v1.0 AuthFilter 统一校验 |
| 白名单路径（如 `/admin/codeLogin`） | ✅ 放行 | ✅ 放行 | `security.ignore.whites` |

**未来扩展**：若后续需要实现 `@RequiresPermission` 式的细粒度权限控制，需要**新增**该注解及对应的 AOP 切面（脚手架当前没有，需作为 v2.0 功能开发）。

## 7.4 数据库设计

> **说明**：用户管理相关表已在 `zmbdp-admin-service` 的 `sys_user` 表中实现（identity字段标识身份，权限通过Gateway的userFrom字段控制），C端用户使用 `app_user` 表，本服务直接复用，无需重复建表。本服务仅需创建AI业务相关的表。

> **时间字段设计原则**（各表统一遵循）：
> - **实体性表**（sys_ai_knowledge_source、sys_ai_document、sys_ai_permission）：含 `create_date` + `update_date`（BIGINT YYYYMMDD），用于记录创建和更新日期，支持按日期范围查询
> - **事件性表**（sys_ai_operation_log、sys_ai_conversation、sys_ai_feedback）：含 `create_date` + `create_time`（BIGINT YYYYMMDD + DATETIME），用于精确记录事件发生时间，支持时间统计和排序
> - 所有 `create_date`/`update_date` 字段均建立索引以优化时间范围查询性能
> - 该差异为设计上的合理区分：实体表需要记录更新日期，事件表只记录创建时间（事件不可更新）

### 7.4.3 AI调用链路日志表（sys_ai_operation_log）

**实体类**：`SysAiOperationLog`（继承 `BaseDO`）

**路径**：`com.zmbdp.chat.service.domain.entity.SysAiOperationLog`

**职责定位**：专用于 **AI 调用链路追踪**（记录每次 AI 调用的完整 Prompt、LLM 响应、工具调用链路、Token 消耗），用于 AI 调试、效果评估和调用统计（配合 Spring AI Micrometer 指标，**不使用 LangSmith**）。

> **与脚手架 operation_log 的职责切分**：
> - **B 端管理操作审计**（知识源 CRUD、配置修改、工具启停等）→ 复用脚手架 `@LogAction` 注解 + `operation_log` 表，由 `LogActionAspect` 切面自动记录，零代码侵入。
> - **AI 调用链路追踪**（Prompt/LLM响应/工具调用/Token）→ 本表 `sys_ai_operation_log`，在 AI 编排层手动埋点记录。
> - **与 sys_ai_conversation 的边界**：`sys_ai_conversation` 是用户视角的一问一答聚合（含 `token_usage`/`tool_usage`/`response_time`），`sys_ai_operation_log` 是开发/运维视角的 AI 调用明细链路（含完整 Prompt/响应/工具调用详情），粒度更细。

#### 7.4.3.1 操作日志职责边界对照表（开发必读）

> **核心原则**：人在管理端做的"管理动作"走 `operation_log`；AI 模型被调用的"执行动作"走 `sys_ai_operation_log`。两者不交叉、不重复记录。

**一、走 `operation_log` 表的操作（脚手架 `@LogAction` 注解自动记录，无需手写日志代码）**

| 操作场景 | @LogAction 参数示例 | 触发位置 | 说明 |
|---------|-------------------|---------|------|
| 知识源新增 | `@LogAction(operation = "CREATE", module = "knowledge")` | `KnowledgeController.createSource()` | B端管理员新增知识源 |
| 知识源修改 | `@LogAction(operation = "UPDATE", module = "knowledge")` | `KnowledgeController.updateSource()` | B端管理员修改知识源 |
| 知识源删除 | `@LogAction(operation = "DELETE", module = "knowledge")` | `KnowledgeController.deleteSource()` | B端管理员删除知识源 |
| 文档上传 | `@LogAction(operation = "UPLOAD", module = "knowledge")` | `KnowledgeController.uploadDocument()` | B端管理员上传文档 |
| 文档删除 | `@LogAction(operation = "DELETE", module = "knowledge")` | `KnowledgeController.deleteDocument()` | B端管理员删除文档 |
| 知识同步触发 | `@LogAction(operation = "SYNC", module = "knowledge")` | `KnowledgeController.syncKnowledge()` | B端管理员手动触发知识同步 |
| AI配置修改 | `@LogAction(operation = "UPDATE", module = "ai_config")` | `AiConfigController.updateConfig()` | B端管理员修改temperature/max_tokens等运行时参数 |
| 工具启用/禁用 | `@LogAction(operation = "UPDATE", module = "tools")` | `ToolController.updateToolConfig()` | B端管理员启停Agent工具 |
| 工具参数修改 | `@LogAction(operation = "UPDATE", module = "tools")` | `ToolController.updateToolConfig()` | B端管理员修改工具Nacos参数 |
| AI权限配置 | `@LogAction(operation = "UPDATE", module = "ai_permission")` | `PermissionController.updatePermission()` | B端管理员配置用户AI调用配额 |

> **开发方式**：在 admin-service 的 Controller 方法上加 `@LogAction` 注解即可，`LogActionAspect` 切面自动拦截并记录到 `operation_log` 表（含入参、返回值、耗时、操作人、IP等），**不要手写日志插入代码**。

**二、走 `sys_ai_operation_log` 表的操作（AI 编排层手动埋点，记录模型调用链路）**

| operation_type | 触发场景 | 记录内容要点 | 埋点位置 |
|---------------|---------|------------|---------|
| `CHAT` | C端/B端发起AI对话（流式或非流式） | 完整Prompt、LLM响应、工具调用链路、Token消耗、响应耗时 | `IChatService` 调用 `ChatClient` 前后 |
| `RETRIEVE` | RAG检索阶段（向量检索+重排序） | 检索Query、命中文档数、重排序耗时、检索模型名 | `IVectorStoreService.search()` 方法内 |
| `EMBEDDING` | 知识同步时批量生成向量嵌入 | 文档分块数、嵌入模型名、Token消耗、耗时 | `IVectorStoreService.addDocuments()` 方法内 |
| `RERANK` | RAG检索的重排序阶段 | 候选文档数、重排序模型名、耗时 | `IVectorStoreService.search()` 中的 Reranking 步骤 |

> **开发方式**：在 chat-service 的 AI 编排层（Service 方法内）手动构建 `SysAiOperationLog` 对象并调用 `SysAiOperationLogMapper.insert()`。每次 AI 模型调用（ChatClient/EmbeddingModel/RerankModel）都应记录一条，包含完整的输入输出和耗时。

**三、不走任何日志表的操作（无需记录）**

| 操作场景 | 原因 |
|---------|------|
| C端用户发起对话（HTTP请求本身） | 由Gateway网关访问日志记录，不需要业务日志 |
| C端用户点赞/点踩 | 记录在 `sys_ai_feedback` 业务表，不是操作日志 |
| B端查询列表/查看详情 | 只读操作，`@LogAction` 默认不记录查询（仅记录写操作） |
| 系统健康检查 | 由Spring Boot Actuator + Prometheus 监控，不需要业务日志 |

## 7.6 错误码定义

### 7.6.1 错误码枚举（ResultCode）

**类名**：`ResultCode`

**路径**：`com.zmbdp.common.domain.domain.ResultCode`

**说明**：AI模块直接使用项目现有的 `ResultCode` 枚举，遵循6位编码规则（前3位为HTTP状态码）。

**现有通用错误码**：

| 错误码 | 错误信息 | 说明 |
|-------|---------|------|
| `200000` | 操作成功 | 请求成功 |
| `400000` | 无效的参数 | 参数校验失败 |
| `401000` | 令牌不能为空 | Token缺失 |
| `401001` | 令牌已过期或验证不正确 | Token无效 |
| `401002` | 令牌已过期 | Token过期 |
| `404000` | 服务未找到 | 服务不存在 |
| `404001` | url未找到 | 路径不存在 |
| `500000` | 服务繁忙请稍后重试 | 系统内部异常 |
| `500001` | 操作失败 | 业务操作失败 |

### 7.6.2 AI模块新增业务错误码

需在 `ResultCode` 枚举中新增以下AI模块专属错误码：

| 错误码 | 错误信息 | 说明 |
|-------|---------|------|
| `403000` | 无权限访问此资源 | 用户来源与接口不匹配（Gateway层） |
| `403001` | 无模型调用权限 | 未配置该模型的调用权限 |
| `403002` | 模型调用配额已用尽 | 权限配额使用完毕 |
| `403003` | 权限已过期 | 权限有效期已过 |
| `404002` | 知识源不存在 | 知识源查询失败 |
| `404003` | 文档不存在 | 文档查询失败 |
| `500022` | AI配置未设置 | API Key等配置缺失 |
| `500023` | AI服务连接失败 | 无法连接到AI模型服务 |
| `500024` | AI服务响应超时 | AI模型响应超时 |
| `500025` | 工具调用失败 | Agent工具执行出错 |
| `500026` | 路径不在白名单 | 文件读取路径不在允许范围 |
| `500027` | 文件类型不支持 | 上传文件类型不在允许列表 |
| `500028` | 文件超过大小限制 | 上传文件超过最大限制 |

1. AI模块新增错误码从 `500022` 开始（现有最大业务码为 `500021` Excel文件写入失败）。
2. 错误码 `400020`（请求过于频繁）已被现有代码占用（频控），AI模块限流复用此错误码。
3. **用户管理相关错误码（如登录失败、用户已存在、用户已禁用等）复用脚手架现有定义**，AI模块不重复定义。例如 `400007`（账号已停用，登录失败）。

| `404000-404001` | 已占用 | 服务未找到、URL未找到 |
| `404002-404099` | **未占用** | AI模块用于资源不存在 |
| `405000` | 已占用 | 请求方法不支持 |
| `500000-500021` | 已占用 | 系统异常、操作失败、短信、OSS、地图、邮件、Excel等 |
| `500022+` | **未占用** | AI模块业务错误码 |

**路径**：`com.zmbdp.chat.service.config.ScheduledConfig`

**配置内容**：
```java
@Configuration
@EnableScheduling
public class ScheduledConfig {

    @Value("${scaffold.scheduler.pool-size:5}")
    private int poolSize;

    @Value("${scaffold.scheduler.thread-name-prefix:scheduled-}")
    private String threadNamePrefix;

    @Bean
    public ThreadPoolTaskScheduler taskScheduler() {
        ThreadPoolTaskScheduler scheduler = new ThreadPoolTaskScheduler();
        scheduler.setPoolSize(poolSize);
        scheduler.setThreadNamePrefix(threadNamePrefix);
        scheduler.setWaitForTasksToCompleteOnShutdown(true);
        scheduler.setAwaitTerminationSeconds(60);
        return scheduler;
    }
}
```

> **说明**：线程池参数通过 `@Value` 从 Nacos 读取（参照 `zmbdp-common-core` 的 `ThreadPoolConfig` 模式），支持 `@RefreshScope` 动态刷新。Nacos 配置项（`zmbdp-chat-service-{env}.yaml`）：
> ```yaml
> scaffold:
>   scheduler:
>     pool-size: 5              # 定时任务线程池大小
>     thread-name-prefix: "scheduled-"  # 线程名前缀
> ```

## 7.10 请求参数校验

### 7.10.1 校验注解使用规范

所有DTO字段必须添加校验注解，使用Jakarta Validation规范（JSR-380）。

**常用校验注解**：

| 注解 | 说明 | 示例 |
|-----|------|------|
| `@NotBlank` | 字符串不能为空 | `@NotBlank(message = "用户名不能为空")` |
| `@NotNull` | 对象不能为空 | `@NotNull(message = "知识源ID不能为空")` |
| `@Size` | 长度限制 | `@Size(min = 6, max = 20, message = "密码长度6-20位")` |
| `@Email` | 邮箱格式 | `@Email(message = "邮箱格式不正确")` |
| `@Min` | 最小值 | `@Min(value = 1, message = "页码最小为1")` |

### 7.10.3 校验执行机制（复用脚手架，禁止手动校验）

> **核心原则**：所有参数校验通过 Jakarta Validation 注解（`@NotBlank`/`@NotNull`/`@Size`/`@Min`/`@Max`/`@Pattern`/`@Email` 等）在 Controller 层完成，由 Spring MVC 自动触发，校验失败抛 `MethodArgumentNotValidException`，由 `GlobalExceptionHandler`（复用 `zmbdp-common-security`）统一处理返回 `400000` 错误码。**禁止在 Service 层手写 if 判断做参数校验。**

**Controller 层用法**：
```java
@PostMapping
public Result<Long> createSource(@Validated @RequestBody KnowledgeSourceReqDTO reqDTO) {
    // 进入方法体时参数已校验通过，无需手动校验
    return Result.success(knowledgeService.createSource(reqDTO));
}
```

**复杂业务规则校验**（如"feedbackType=DISLIKE 时 dislikeReason 必填"）：
- 在 DTO 中用 `@AssertTrue` 注解实现跨字段校验：
```java

    return feedbackType != FeedbackType.DISLIKE || StringUtils.isNotBlank(dislikeReason);
}
```
- 如需分组校验（新增/编辑不同校验规则），在 Controller 方法参数加 `@Validated(AddGroup.class)`，DTO 字段加 `@NotBlank(groups = AddGroup.class)`。
- 简单字段校验（非空、长度、格式）一律用注解，不要在 Service 层重复手动校验。

## 7.12 二级缓存设计（Caffeine + Redis + MQ 广播失效）

> **术语澄清（重要）——"二级缓存"与"三级缓存"的区别**：
>
> 本节描述的"**二级缓存**"专指 **AI 模块自身**为 `CHAT_MEMORY` 和 `AI_CONFIG` 两个缓存空间设计的 L1 Caffeine + L2 Redis + MQ 广播失效机制，是 AI 系统的**内部基础设施**。
>
> 文档其他章节（03/05/06/08/10）中出现的"**三级缓存**"指的是**脚手架对外暴露的缓存功能**（布隆过滤器 + Caffeine + Redis 三层），是 AI 助手要**回答的业务知识对象**，与本节的"二级缓存"是完全不同层面的概念。
>
> 两者的关系：
> - AI 模块的二级缓存**复用了**脚手架的 `CaffeineConfig`（L1 Bean 注册）和 `RedisService`（L2 读写），但**未直接复用** `CacheUtil`（因需 MQ 广播失效能力，CacheUtil 不支持）
> - 脚手架三级缓存中的布隆过滤器由 `BloomFilterService` 单独提供（AI 模块用于 document_id 存在性检查，见 7.2.16 节）
> - 简言之：**"三级缓存"是 AI 要讲解的脚手架功能；"二级缓存"是 AI 模块自己用的缓存机制**

### 7.12.1 设计背景

chat-service 部署多实例时（prd 环境双副本 vm1+vm2），Caffeine 本地缓存无法跨实例同步。当实例 A 更新了数据并删除了自己的 Caffeine 缓存，实例 B 的 Caffeine 缓存仍然存在旧数据，导致**脏读**。因此必须通过 MQ 广播通知所有实例清除本地 Caffeine 缓存。

> **缓存范围决策——为什么不缓存向量检索结果**：
>
> 业界主流 RAG 框架（LangChain `CacheBackedEmbeddings`、LlamaIndex、GPTCache 等）缓存的是 **embedding 结果**（文本→向量的映射）或 **LLM 最终响应**（语义缓存），而**不是向量检索结果**。原因如下：
>
> 1. **数据一致性要求高**：知识库持续更新（知识同步、文档增删），缓存的检索结果容易脏，命中旧结果会导致回答错误
> 2. **缓存命中率低**：用户提问千差万别，query 的 MD5 作为 key 时，即使语义相似的 query 也无法命中（语义缓存属于 GPTCache 范畴，超出二级缓存设计目标）
> 3. **向量库查询已足够快**：Milvus HNSW 索引检索为毫秒级，Reranking 也在秒级以内，缓存收益有限
> 4. **Reranking 结果会变**：重排序模型升级或参数调整后，缓存结果会过时
>
> 因此本设计**只缓存对话历史和 AI 配置**，向量检索每次都实时查询 Milvus + Reranking，保证检索可靠性。

### 7.12.2 缓存架构

```
读取流程：
┌──────────┐     未命中      ┌──────────┐     未命中      ┌──────────┐
│ L1 本地   │ ──────────→ │ L2 Redis  │ ──────────→ │ DB / 计算  │
│ Caffeine │              │ (共享)    │              │ (Milvus等) │
└──────────┘              └──────────┘              └──────────┘
     ↑ 回填L1+L2               ↑ 回填L2

失效流程（数据变更时）：
┌──────────┐   1.删L2 Redis   ┌──────────┐
│ DB 写入   │ ──────────────→ │ Redis    │ 删除缓存key
└──────────┘                  └──────────┘
     │
     ↓ 2.发MQ消息（fanout广播）
┌──────────────────────────────────────┐
│       cache.invalidate.exchange       │  Fanout 交换机
└──────────┬───────────────┬───────────┘
           ↓               ↓
   ┌───────────┐   ┌───────────┐
   │ 实例A队列  │   │ 实例B队列  │  每实例独占匿名队列
   │ (anonymous)│   │ (anonymous)│
   └─────┬─────┘   └─────┬─────┘
         ↓               ↓
   3.删L1 Caffeine  3.删L1 Caffeine
```

### 7.12.3 缓存空间设计

| 缓存空间 | cacheName 常量 | 用途 | L1(Caffeine) TTL | L1 最大条数 | L2(Redis) | 失效触发场景 | 失效方式 |
|---------|---------------|------|------------------|------------|-----------|------------|---------|
| 对话历史 | `CHAT_MEMORY` | 按 sessionId 缓存对话历史 List 副本 | 5 分钟 | 1000 | **无独立 L2**（Redis List 即数据存储，TTL=7天） | 新增消息/删除会话 | `invalidateL1Only`（只删 L1，Redis List 由 lPush/clearHistory 直接操作） |
| AI 配置 | `AI_CONFIG` | 按 configKey 缓存 sys_argument 配置值 | 30 分钟 | 100 | 60 分钟 | 管理员修改配置/工具状态 | `invalidate`/`invalidateAll`（删 L2 Redis + 广播删 L1） |

> **CHAT_MEMORY 设计特殊性**：Redis List 本身是对话历史的数据存储（不是缓存），通过 `lPush` 写入新消息时数据已是最新，无需删除。只需 MQ 广播删 L1 Caffeine 让其他实例下次读时从 Redis List 重新加载即可。这与 `AI_CONFIG`（MySQL 为数据源、Redis 为 L2 缓存）的失效语义不同。

> **向量检索结果不缓存**：详见 7.12.1 节"缓存范围决策"，向量检索每次都实时查询 Milvus + Reranking，保证检索可靠性。

### 7.12.6 CacheMqConfig（MQ 配置类）

> **路径**：`com.zmbdp.chat.service.config.CacheMqConfig`
>
> **职责**：声明 Fanout 交换机 Bean（队列和绑定关系由 `@RabbitListener(bindings=...)` 自动声明，见 7.12.8 节消费者）

```java
@Configuration
public class CacheMqConfig {

    /**
     * 缓存失效广播交换机（Fanout 类型，持久化，不自动删除）
     * 消费者端通过 @RabbitListener(bindings=@QueueBinding) 自动声明匿名队列并绑定到此交换机
     */
    @Bean
    public FanoutExchange cacheInvalidateExchange() {
        return new FanoutExchange(AiCacheConstants.CACHE_INVALIDATE_EXCHANGE, true, false);
    }
}
```

> **关于交换机重复声明**：Fanout 交换机在此处用 `@Bean` 声明一次，在 7.12.8 消费者的 `@RabbitListener(bindings=@QueueBinding(...))` 的 `@Exchange` 中又会声明一次。Spring AMQP 对此**幂等处理**（属性一致不会报错），这是 Spring AMQP 推荐的做法：生产者端通过 `@Bean` 确保交换机存在（生产者启动时即可声明），消费者端通过 `@QueueBinding` 确保绑定关系存在（消费者启动时声明匿名队列并绑定）。两处引用同一个常量 `AiCacheConstants.CACHE_INVALIDATE_EXCHANGE`，属性一致，无需担心冲突。

### 7.12.7 CacheInvalidateProducer（生产者）

> **路径**：`com.zmbdp.chat.service.mq.producer.CacheInvalidateProducer`
>
> **职责**：数据变更时删除 L2(Redis) 缓存（如有），并发送 MQ 消息广播通知所有实例删除 L1(Caffeine) 缓存
>
> **三个方法对应两种缓存空间的不同失效语义**：
> - `invalidate` / `invalidateAll`：用于 `AI_CONFIG`（MySQL 为数据源、Redis 为 L2 缓存），删 L2 Redis + 广播删 L1
> - `invalidateL1Only`：用于 `CHAT_MEMORY`（Redis List 即数据存储），只广播删 L1（Redis List 已通过 lPush 更新或由 clearHistory 删除，无需此方法再删）

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class CacheInvalidateProducer {

    private final RabbitTemplate rabbitTemplate;
    private final RedisService redisService;

    /**
     * 失效单个缓存条目（删 L2 Redis + 广播删 L1 Caffeine）
     * <p>适用场景：AI_CONFIG 缓存空间的单条配置失效（如管理员修改某个工具启用状态）
     * @param cacheName  缓存空间名（AiCacheConstants.CACHE_NAME_*）
     * @param redisKey   完整的 Redis key（含前缀）
     * @param cacheKey   Caffeine 中的 key（通常与 redisKey 相同）
     */
    public void invalidate(String cacheName, String redisKey, String cacheKey) {
        // 1. 删除 L2 Redis 缓存
        redisService.deleteObject(redisKey);
        // 2. 发送 MQ 消息广播删除 L1 Caffeine 缓存
        CacheInvalidateMessage message = new CacheInvalidateMessage(cacheName, cacheKey);
        rabbitTemplate.convertAndSend(AiCacheConstants.CACHE_INVALIDATE_EXCHANGE, "", message);
        log.info("缓存失效广播（L2+L1）：cacheName = {}, cacheKey = {}", cacheName, cacheKey);
    }

    /**
     * 失效整个缓存空间（删 L2 Redis 通配符 key + 广播删 L1 Caffeine 整空间）
     * <p>适用场景：AI_CONFIG 缓存空间的整空间失效（如管理员批量修改配置）
     * @param cacheName 缓存空间名（AiCacheConstants.CACHE_NAME_*）
     * @param redisKeyPattern Redis key 通配符（如 "ai:config:*"）
     */
    public void invalidateAll(String cacheName, String redisKeyPattern) {
        // 1. 删除 L2 Redis 中该空间的所有 key
        Collection<String> keys = redisService.keys(redisKeyPattern);
        if (keys != null && !keys.isEmpty()) {
            redisService.deleteObject(keys);
        }
        // 2. 发送 MQ 消息广播删除 L1 Caffeine 该空间的所有条目（cacheKey=null 表示清除整个空间）
        CacheInvalidateMessage message = new CacheInvalidateMessage(cacheName, null);
        rabbitTemplate.convertAndSend(AiCacheConstants.CACHE_INVALIDATE_EXCHANGE, "", message);
        log.info("缓存空间失效广播（L2+L1）：cacheName = {}, 删除Redis key数量 = {}", cacheName, keys == null ? 0 : keys.size());
    }

    /**
     * 仅失效 L1 Caffeine（不删 L2 Redis）
     * <p>适用场景：CHAT_MEMORY 缓存空间（Redis List 即数据存储，已通过 lPush 更新或由 clearHistory 删除，
     * 无需再删 Redis，只需让其他实例的 L1 Caffeine 失效，下次读时从 Redis List 重新加载）
     * @param cacheName 缓存空间名（AiCacheConstants.CACHE_NAME_CHAT_MEMORY）
     * @param cacheKey  Caffeine 中的 key（通常是完整的 Redis 数据存储 key）
     */
    public void invalidateL1Only(String cacheName, String cacheKey) {
        // 只发送 MQ 消息广播删除 L1 Caffeine 缓存，不删除 Redis
        CacheInvalidateMessage message = new CacheInvalidateMessage(cacheName, cacheKey);
        rabbitTemplate.convertAndSend(AiCacheConstants.CACHE_INVALIDATE_EXCHANGE, "", message);
        log.info("缓存失效广播（仅L1）：cacheName = {}, cacheKey = {}", cacheName, cacheKey);
    }
}
```

### 7.12.8 CacheInvalidateConsumer（消费者）

> **路径**：`com.zmbdp.chat.service.mq.consumer.CacheInvalidateConsumer`
>
> **职责**：接收 MQ 广播消息，删除本地 Caffeine 缓存
>
> **关键设计**：使用 `@RabbitListener(bindings = @QueueBinding(...))` 在代码中自动声明交换机、匿名队列和绑定关系，**无需在 RabbitMQ 管理页面手动创建**。每个 chat-service 实例启动时自动创建独占的匿名队列，绑定到 Fanout 交换机，实现广播。

```java
@Slf4j
@Component
public class CacheInvalidateConsumer {

    /** 注入 Caffeine 缓存（复用 zmbdp-common-cache 的 CaffeineConfig 注册的 Bean） */
    @Autowired
    @Qualifier("caffeineCache")
    private Cache<String, Object> caffeineCache;

    /**
     * 处理缓存失效消息
     *
     * @QueueBinding 说明：
     * - exchange: Fanout 类型交换机，名字提取到 AiCacheConstants.CACHE_INVALIDATE_EXCHANGE 常量
     * - queue: @Queue 不写 value 属性 → Spring AMQP 自动生成匿名队列（exclusive=true, autoDelete=true）
     *          每个实例独占一个队列，实现 Fanout 广播（消息会推送到所有绑定的队列）
     * - key: 路由 key 提取到 AiCacheConstants.CACHE_INVALIDATE_ROUTING_KEY 常量（Fanout 类型会忽略此值）
     *
     * 交换机名、队列绑定关系由 Spring AMQP 在应用启动时自动声明到 RabbitMQ，
     * 无需在 RabbitMQ 管理页面手动创建交换机和队列。
     */
    @RabbitListener(bindings = @QueueBinding(
        exchange = @Exchange(value = AiCacheConstants.CACHE_INVALIDATE_EXCHANGE, type = ExchangeTypes.FANOUT),
        queue = @Queue,  // 匿名队列，每实例独占，实现广播
        key = AiCacheConstants.CACHE_INVALIDATE_ROUTING_KEY
    ))
    public void handleCacheInvalidate(CacheInvalidateMessage message) {
        String cacheName = message.getCacheName();
        String cacheKey = message.getCacheKey();

        if (cacheKey == null) {
            // cacheKey 为空 → 清除整个缓存空间的所有条目
            // Caffeine 不支持按前缀删除，需要遍历删除（缓存空间名作为 key 前缀）
            caffeineCache.asMap().keySet()
                .stream()
                .filter(key -> key.startsWith(cacheName + ":"))
                .forEach(caffeineCache::invalidate);
            log.info("清除 Caffeine 缓存空间：{}", cacheName);
        } else {
            // 删除单个缓存条目（Caffeine key 格式：{cacheName}:{cacheKey}）
            caffeineCache.invalidate(cacheName + ":" + cacheKey);
            log.info("清除 Caffeine 缓存条目：{}:{}", cacheName, cacheKey);
        }
    }
}
```

### 7.12.9 缓存读写流程

**读取流程一：CHAT_MEMORY（对话历史，Redis List 为数据存储）**：

```java
public List<ChatMessage> getHistory(String sessionId) {
    String redisKey = AiCacheConstants.CHAT_MEMORY_REDIS_PREFIX + sessionId;
    String caffeineKey = AiCacheConstants.CACHE_NAME_CHAT_MEMORY + ":" + redisKey;

    // 1. 查 L1 Caffeine
    Object cached = caffeineCache.getIfPresent(caffeineKey);
    if (cached != null) {
        return (List<ChatMessage>) cached;
    }

    // 2. 查 Redis 数据存储（List 结构，lRange 读取后反序列化）
    List<String> jsonList = redisService.lRange(redisKey, 0, -1);
    if (jsonList != null && !jsonList.isEmpty()) {
        List<ChatMessage> result = jsonList.stream()
            .map(json -> JsonUtil.fromJson(json, ChatMessage.class))
            .collect(Collectors.toList());
        Collections.reverse(result);  // lPush 头部插入，需反转为时间正序
        // 回填 L1 Caffeine（不回填 Redis，Redis List 本身是数据存储）
        caffeineCache.put(caffeineKey, result);
        return result;
    }

    // 3. DB 降级（Redis 未命中或已过期，直接查 MySQL sys_ai_conversation 表，不经过 IHistoryService 避免循环依赖）
    List<ChatMessage> result = sysAiConversationMapper.selectBySessionId(sessionId);
    if (result != null && !result.isEmpty()) {
        // 回填 Redis List：按时间正序逐条 lPush（lPush 头部插入，读取时 lRange+reverse 即恢复正序，与 addMessage 语义一致）
        result.forEach(msg -> redisService.lPush(redisKey, JsonUtil.toJson(msg)));
        redisService.expire(redisKey, AiCacheConstants.CHAT_MEMORY_REDIS_TTL_SECONDS);
        caffeineCache.put(caffeineKey, result);
    }
    return result != null ? result : Collections.emptyList();
}
```

**读取流程二：AI_CONFIG（AI 配置，MySQL 为数据源、Redis 为 L2 缓存）**：

```java
public String getConfigValue(String configKey) {
    String redisKey = AiCacheConstants.AI_CONFIG_REDIS_PREFIX + configKey;
    String caffeineKey = AiCacheConstants.CACHE_NAME_AI_CONFIG + ":" + redisKey;

    // 1. 查 L1 Caffeine
    Object cached = caffeineCache.getIfPresent(caffeineKey);
    if (cached != null) {
        return (String) cached;
    }

    // 2. 查 L2 Redis
    cached = redisService.getCacheObject(redisKey);
    if (cached != null) {
        caffeineCache.put(caffeineKey, cached);
        return (String) cached;
    }

    // 3. DB 查询（通过 ArgumentServiceApi Feign 调用读 sys_argument 表）
    String result = argumentServiceApi.getValue(configKey).getData();

    // 4. 回填 L1 + L2
    caffeineCache.put(caffeineKey, result);
    redisService.setCacheObject(redisKey, result,
        AiCacheConstants.AI_CONFIG_L2_TTL_MINUTES, TimeUnit.MINUTES);
    return result;
}
```

**失效流程一：CHAT_MEMORY（新增消息，只删 L1）**：

```java
// 新增对话消息 → Redis List 已通过 lPush 更新为最新，只需广播删 L1 Caffeine
String redisKey = AiCacheConstants.CHAT_MEMORY_REDIS_PREFIX + sessionId;
cacheInvalidateProducer.invalidateL1Only(
    AiCacheConstants.CACHE_NAME_CHAT_MEMORY,
    redisKey  // Caffeine key 格式为 "CHAT_MEMORY:{redisKey}"
);
```

**失效流程二：AI_CONFIG（管理员修改配置，删 L2 + 广播删 L1）**：

```java
// 管理员修改配置 → 清除整个 AI_CONFIG 缓存空间
cacheInvalidateProducer.invalidateAll(
    AiCacheConstants.CACHE_NAME_AI_CONFIG,
    AiCacheConstants.AI_CONFIG_REDIS_PREFIX + "*"
);
```

> **向量检索不经过缓存**：`IVectorStoreService.search()` 每次都实时执行 Embedding → Milvus → Reranking 流程，不查询也不回填二级缓存，保证检索结果始终基于最新的向量库数据。

### 7.12.10 各缓存空间的失效触发时机

| 缓存空间 | 失效触发时机 | 调用方式 | 埋点位置 |
|---------|------------|---------|---------|
| `CHAT_MEMORY` | 新增对话消息 | `invalidateL1Only(CHAT_MEMORY, redisKey)` | `IChatMemoryService.addMessage()` |
| `CHAT_MEMORY` | 批量新增消息 | `invalidateL1Only(CHAT_MEMORY, redisKey)` | `IChatMemoryService.addMessages()` |
| `CHAT_MEMORY` | 清除会话历史 | `invalidateL1Only(CHAT_MEMORY, redisKey)`（Redis List 由 clearHistory 删除） | `IChatMemoryService.clearHistory()` |
| `CHAT_MEMORY` | 删除会话 | `invalidateL1Only(CHAT_MEMORY, redisKey)`（Redis List 由 clearHistory 删除，deleteSession 调用 clearHistory） | `IHistoryService.deleteSession()` → `IChatMemoryService.clearHistory()` |
| `AI_CONFIG` | 管理员修改配置 | `invalidateAll(AI_CONFIG, "ai:config:*")` | `IAdminService.updateAiConfig()` |
| `AI_CONFIG` | 管理员修改工具状态 | `invalidate(AI_CONFIG, redisKey, redisKey)` | `IAdminService.updateToolConfig()` |

### 7.12.11 Nacos 配置项

在 `zmbdp-chat-service-{env}.yaml` 中新增 Caffeine 配置（复用 `zmbdp-common-cache` 的 `CaffeineConfig`）：

```yaml
# ==================== Caffeine 本地缓存配置（L1，详见 7.12 节） ====================
caffeine:
  build:
    initial-capacity: 100        # 初始容量
    maximum-size: 1000           # 最大条数（两个缓存空间共享：CHAT_MEMORY、AI_CONFIG）
    expire: 30                   # 全局默认过期时间（秒），各缓存空间在代码中通过 put 时指定独立 TTL 覆盖

# ==================== 布隆过滤器配置（文档ID存在性检查，详见 7.2.16 节） ====================
bloom:
  filter:
    type: redis                  # 推荐redis实现，支持分布式（fast/safe/redis）

# ==================== 定时任务线程池配置（详见 7.9.1 节） ====================
scaffold:
  scheduler:
    pool-size: 5
    thread-name-prefix: "scheduled-"
```

> **配置项说明**：
> - `caffeine.build.maximum-size`：Caffeine 全局最大条数（脚手架 `CaffeineConfig` 只支持单一全局 `maximumSize`，两个缓存空间共享此容量上限）
> - `caffeine.build.expire`：Caffeine 全局默认 TTL（脚手架 `CaffeineConfig` 只支持单一全局 `expireAfterWrite`）。**各缓存空间在代码中 `put` 时通过 Caffeine 的 `expiringAfter()` 或自定义 `Expiry` 覆盖**：CHAT_MEMORY=5分钟、AI_CONFIG=30分钟（对应 `AiCacheConstants.CHAT_MEMORY_L1_TTL_MINUTES` 和 `AI_CONFIG_L1_TTL_MINUTES`）
> - `bloom.filter.type=redis`：**必需配置**，`RedisBloomFilterService` 通过 `@ConditionalOnProperty(value = "bloom.filter.type", havingValue = "redis")` 激活，未配置则 Bean 不创建，`IVectorStoreService` 注入 `BloomFilterService` 会失败
