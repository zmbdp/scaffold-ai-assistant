# 📄 B端功能设计文档

> **文档说明**：本文档详细描述 Scaffold AI Assistant B端（管理端）的功能设计。
> **复用说明**：用户管理（sys_user 表 CRUD）复用现有 `zmbdp-admin-service` 的实现。操作日志使用现有 `@LogAction` 注解自动记录。
> **重要说明**：脚手架**未实现**角色管理（无 sys_role 表）和细粒度权限控制（无 sys_permission 表），sys_user.identity 字段仅作标识存储。B端接口的访问控制通过扩展 Gateway AuthFilter 实现（详见 07-项目架构设计.md 7.2.24节）。

```
│  │             │    │             │                             │
│  └─────────────┘    └─────────────┘                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

> **数据备份恢复说明**：数据备份恢复**不在应用层实现**，由 DevOps/DBA 工具负责：
> - **MySQL**：使用 `mysqldump`、MySQL Enterprise Backup 或云数据库备份服务
> - **Milvus**：使用 [milvus-backup](https://github.com/zilliztech/milvus-backup) 工具
> - **知识文件**：使用 `rsync`、`tar` 等标准文件系统备份工具
>
> **原因**：应用层备份恢复存在一致性风险（跨 MySQL + Milvus 双库）、影响生产环境稳定性，且非 AI 助手核心功能。数据库备份应由专业工具在运维层面统一管理。

## 4.2 功能模块详细设计

### 4.2.1 知识库管理模块

**功能描述**：管理知识源、上传文档、触发知识同步、查看文档状态

**子功能**：

| 子功能 | 描述 |
|-------|------|
| **知识源管理** | 查看、新增、编辑、删除知识源（目录路径、类型等） |
| **文档上传** | 上传新文档到知识库 |
| **知识同步** | 手动触发知识同步，更新向量数据库 |
| **文档状态** | 查看文档索引状态、更新时间、大小等 |
| **文档搜索** | 搜索知识库中的文档 |
| **召回测试** | 输入测试问题，验证RAG检索效果（检索到哪些文档、相似度多少） |

> **召回测试说明**（参考 Dify、FastGPT 等主流 RAG 产品的召回测试功能）：
> - 管理员输入一个测试问题，系统执行完整的 RAG 检索流程（Embedding → Milvus 检索 → Reranking），返回命中的文档分块列表及相似度分数
> - 用于验证知识库质量：新增知识源并同步后，测试"用户问某个问题能不能检索到对应文档"
> - 复用 C 端已有的 `/chat/retrieve` 接口能力（通过 Feign 调用 chat-service）

**API接口**：

| HTTP方法 | 路径 | 说明 |
|---------|------|------|
| GET | `/knowledge/sources` | 获取知识源列表 |
| POST | `/knowledge/sources` | 新增知识源 |
| PUT | `/knowledge/sources/{id}` | 更新知识源 |
| DELETE | `/knowledge/sources/{id}` | 删除知识源 |
| POST | `/knowledge/sync` | 触发知识同步 |
| GET | `/knowledge/documents` | 获取文档列表 |
| GET | `/knowledge/documents/{id}` | 获取文档详情 |
| DELETE | `/knowledge/documents/{id}` | 删除文档 |
| POST | `/knowledge/retrieve-test` | 召回测试（输入测试问题，返回检索到的文档分块及相似度） |

### 4.2.2 AI配置管理模块

**功能描述**：配置AI模型参数、API Key等

**子功能**：

| 子功能 | 描述 | 配置来源 |
|-------|------|---------|
| **模型配置** | 选择AI模型（通义千问/OpenAI） | Nacos（改了需重启） |
| **API Key管理** | API Key 统一在 Nacos 管理（环境变量注入），B端界面只读脱敏显示 | Nacos |
| **参数配置** | 调整temperature、max-tokens、topK、enableRag、enableTools等运行时参数 | sys_argument表（复用脚手架通用参数表，改了即时生效） |
| **Embedding配置** | 配置向量嵌入模型和参数 | Nacos（改了需重启） |

> **职责边界说明**：
> - **Nacos 管理**（基础设施配置）：api-key、模型名称、milvus连接、知识库路径等，改了需要重启或刷新配置
> - **sys_argument 表管理**（运行时可调参数，复用脚手架通用参数表）：config_key 为 `ai.temperature`、`ai.max_tokens`、`ai.top_k`、`ai.enable_rag`、`ai.enable_tools`，B端界面修改后即时生效

**配置项**：

| 配置项 | 说明 | 示例值 | 配置来源 |
|-------|------|--------|---------|
| `spring.ai.dashscope.api-key` | 通义千问API Key | - | Nacos |
| `spring.ai.dashscope.chat.model` | 聊天模型 | `qwen-max` | Nacos |
| `spring.ai.dashscope.chat.options.temperature` | 温度参数（Nacos兜底默认值） | `0.7` | Nacos + sys_argument表 |
| `spring.ai.dashscope.chat.options.max-tokens` | 最大token数（Nacos兜底默认值） | `4096` | Nacos + sys_argument表 |
| `spring.ai.dashscope.embedding.model` | 嵌入模型 | `text-embedding-v1` | Nacos |
| `spring.ai.rerank.model` | 重排序模型（详见 06-RAG流程设计.md 6.3.1节） | `gte-rerank` | Nacos |
| `scaffold.rag.topK` | RAG检索默认topK（范围1-20，运行时可被请求参数覆盖） | `5` | Nacos + sys_argument表 |
| `scaffold.rag.context-max-length` | Prompt中文档上下文最大长度（字符） | `8000` | Nacos |
| `scaffold.rag.memory-rounds` | 对话历史保留轮数 | `5` | Nacos |
| `scaffold.rag.rerank-timeout` | Reranking超时时间（秒），超时降级为Milvus原始排序 | `5` | Nacos |

**API接口**：

| HTTP方法 | 路径 | 说明 |
|---------|------|------|
| GET | `/ai/config` | 获取AI配置 |
| PUT | `/ai/config` | 更新AI配置 |
| GET | `/ai/models` | 获取可用模型列表 |
| POST | `/ai/test` | 测试AI连接 |

### 4.2.3 使用统计模块

**功能描述**：查看系统使用情况、对话统计等，结合 Grafana + `sys_ai_operation_log` 表（AI调用链路追踪）+ `sys_ai_conversation` 表（对话聚合）+ `sys_ai_feedback` 表（满意度）实现 AI 调用追踪和评估（**不使用 LangSmith**，详见 09-部署方案.md 9.9.3 节）

**子功能**：

| 子功能 | 描述 | 数据来源 |
|-------|------|---------|
| **对话统计** | 总对话次数、今日对话次数、趋势图 | 数据库聚合查询 |
| **热门问题** | 统计最常被问的问题TOP10 | 数据库聚合查询 |
| **用户活跃度** | 活跃用户数、使用频次 | 数据库聚合查询 |
| **工具使用统计** | 各Agent工具的调用次数 | 数据库聚合查询 |
| **回答满意度** | 点赞率、点踩率、点踩原因分布（反哺RAG优化） | `sys_ai_feedback` 表聚合查询 |
| **响应时间统计** | AI响应时间分布 | Grafana监控 |
| **AI调用追踪** | 查看每次AI调用的完整链路（Prompt、LLM响应、工具调用） | `sys_ai_operation_log` 表（AI调用链路日志） |
| **Token消耗统计** | AI调用的token消耗、成功率、平均耗时 | `sys_ai_operation_log` 表（按 `operation_type` 聚合） |

**统计指标**：

| 指标 | 说明 |
|-----|------|
| `totalConversations` | 总对话次数 |
| `todayConversations` | 今日对话次数 |
| `avgResponseTime` | 平均响应时间 |
| `topQuestions` | 热门问题列表 |
| `activeUsers` | 活跃用户数 |
| `tokenUsage` | Token消耗统计 |
| `successRate` | AI调用成功率 |
| `likeRate` | 点赞率（点赞数/已反馈对话数） |
| `dislikeRate` | 点踩率（点踩数/已反馈对话数） |
| `feedbackRate` | 反馈率（已反馈对话数/总对话数） |
| `dislikeReasonDistribution` | 点踩原因分布（outdated/irrelevant/code_error/other 各占比） |

**API接口**：

| HTTP方法 | 路径 | 说明 |
|---------|------|------|
| GET | `/statistics/conversations` | 获取对话统计 |
| GET | `/statistics/questions` | 获取热门问题 |
| GET | `/statistics/users` | 获取用户统计 |
| GET | `/statistics/tools` | 获取工具使用统计 |
| GET | `/statistics/feedback` | 获取回答满意度统计（点赞率、点踩率、原因分布） |
| GET | `/statistics/ai-metrics` | 获取AI调用聚合指标 |
| GET | `/statistics/ai-metrics/{operationId}` | 获取单次AI调用详情（完整Prompt、LLM响应、工具调用链路） |

**数据来源**：
- **API调用统计**：后端提供统计API接口，供前端调用展示（对话统计、热门问题、用户统计、工具使用统计、回答满意度）
- **Grafana监控**：通过Prometheus采集应用指标（请求量、响应时间、错误率），用于系统级监控告警
- **AI 调用追踪**：通过 `sys_ai_operation_log` 表记录每次 AI 调用的完整链路（Prompt、LLM 响应、工具调用、Token 消耗），配合 Spring AI Micrometer 指标实现调试和评估（**不使用 LangSmith**，详见 09-部署方案.md 9.9.3 节）。注：B端管理操作审计（知识源CRUD等）由脚手架 `@LogAction` 注解自动记录到 `operation_log` 表，不在 `sys_ai_operation_log` 重复
- **回答满意度**：通过 `sys_ai_feedback` 表（C端用户点赞/点踩反馈，详见 03-C端功能设计.md 3.2.5节）聚合统计，点踩原因分布可指导 RAG 检索策略优化和知识库内容更新

### 4.2.4 系统管理模块

**功能描述**：用户管理、操作日志、系统状态等

**子功能**：

| 子功能 | 描述 |
|-------|------|
| **用户管理** | 用户列表、新增、编辑、删除、禁用（复用脚手架 sys_user 表） |
| **操作日志** | 查看管理员操作记录（复用 `@LogAction` 注解）+ 查看AI调用链路日志（`sys_ai_operation_log` 表） |
| **系统状态** | 查看系统健康状态、组件状态 |

**用户身份标识**（sys_user.identity 字段，仅作标识，无细粒度权限校验）：

| identity 值 | 说明 | 当前权限控制 |
|------------|------|-------------|
| `super_admin` | 超级管理员 | 仅标识，无额外权限逻辑 |
| `platform_admin` | 平台管理员 | 仅标识，无额外权限逻辑 |

> **注意**：脚手架未实现角色体系（无 sys_role 表）和细粒度权限控制（无 sys_permission 表）。如需实现 RBAC，需作为 v2.0 功能新增开发。

> **操作日志职责边界**（开发必读，详见 07-项目架构设计.md 7.4.3.1 节）：
> - **B端管理操作**（知识源CRUD、配置修改、工具启停、同步触发等）→ 复用脚手架 `@LogAction` 注解自动记录到 `operation_log` 表，Controller 方法加注解即可，**不要手写日志代码**。
> - **AI调用链路**（对话CHAT/检索RETRIEVE/嵌入EMBEDDING/重排RERANK）→ 记录到 `sys_ai_operation_log` 表，AI 编排层手动 `insert`。
> - **C端对话记录** → 记录到 `sys_ai_conversation` 表。
> - **C端点赞/点踩** → 记录到 `sys_ai_feedback` 业务表，不是操作日志。
> - 核心原则：人在管理端做的"管理动作"走 `operation_log`；AI 模型被调用的"执行动作"走 `sys_ai_operation_log`。两者不交叉、不重复记录。

**系统健康检查详情**：

| 组件 | 检测方式 | 健康标准 |
|------|---------|---------|
| **MySQL** | Spring Boot Actuator `DataSourceHealthIndicator` | 能成功获取数据库连接并执行 `SELECT 1` |
| **Redis** | Spring Boot Actuator `RedisHealthIndicator` | 能成功执行 Redis `PING` 命令 |
| **Nacos** | 自定义 HealthIndicator（调用 Nacos OpenAPI `/ns/operator/metrics`） | Nacos Server 可访问且返回正常 |
| **Milvus** | 自定义 HealthIndicator（调用 Milvus SDK `getCollectionStats`） | Milvus Server 可访问且 `scaffold_knowledge` 集合存在 |
| **LLM API** | 自定义 HealthIndicator（调用通义千问 `/v1/models` 接口） | API 可访问且 API Key 有效 |

**系统健康状态响应结构**（`SystemHealthVO`）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `status` | String | 整体状态（UP/DOWN） |
| `components` | List | 各组件状态列表 |
| `components[].name` | String | 组件名称（mysql/redis/nacos/milvus/llm） |
| `components[].status` | String | 组件状态（UP/DOWN） |
| `components[].detail` | String | 状态详情（DOWN时记录错误信息） |
| `components[].responseTime` | Long | 响应时间（毫秒） |

**API接口**：

| HTTP方法 | 路径 | 说明 |
|---------|------|------|
| GET | `/knowledge/logs` | 获取知识库操作日志 |
| GET | `/system/health` | 获取系统健康状态（含5个组件检测） |

> **说明**：用户管理接口已在 `zmbdp-admin-service` 中实现，本模块不再重复开发。AI 模块仅新增知识库操作日志查询接口和系统健康状态查询接口。

### 4.2.5 工具管理模块

**功能描述**：管理Agent工具的启用/禁用、配置工具参数

**子功能**：

| 子功能 | 描述 |
|-------|------|
| **工具列表** | 查看所有Agent工具 |
| **工具状态** | 启用/禁用工具 |
| **工具配置** | 配置工具参数（如文件路径白名单、黑名单、文件大小限制等） |
| **工具测试** | 测试工具功能 |

**工具列表**：

| 工具名称 | 默认状态 | 描述 |
|---------|---------|------|
| ReadFileTool | 启用 | 读取指定文件内容 |
| SearchCodeTool | 启用 | 搜索代码中的类/方法 |
| ListDirTool | 启用 | 列出目录结构 |
| SearchInFileTool | 启用 | 在文件中搜索关键字 |
| NacosConfigTool | 启用 | 查询Nacos配置项 |
| CompareConfigTool | 启用 | 对比不同环境配置 |
| PreDeployCheckTool | 启用 | 部署前检查清单 |

**工具参数配置详情**：

> **存储位置**：所有工具配置统一存储在 Nacos 配置中心（dataId: `zmbdp-chat-service-${env}.yaml`），支持 `@RefreshScope` 动态刷新，修改后无需重启服务即可生效。
>
> **设计原因**：工具配置（如路径白名单）需要根据部署环境动态调整，存在 Nacos 便于运维人员直接修改，也便于不同环境（dev/test/prd）使用不同的配置。

**可配置工具及参数清单**：

| 工具名称 | 可配置参数 | 配置项 | 说明 |
|---------|-----------|--------|------|
| ReadFileTool | 路径白名单、路径黑名单、最大文件大小 | `knowledge.allowed-paths`、`knowledge.path-blacklist`、`knowledge.max-file-size`、`knowledge.base-path` | 控制可读取的文件范围 |
| SearchCodeTool | 路径白名单、搜索结果数量 | `knowledge.allowed-paths`、`knowledge.base-path` | 控制搜索范围（复用ReadFileTool白名单） |
| ListDirTool | 路径白名单、路径黑名单 | `knowledge.allowed-paths`、`knowledge.path-blacklist` | 控制可列出的目录范围（复用ReadFileTool白名单） |
| SearchInFileTool | 路径白名单、路径黑名单 | `knowledge.allowed-paths`、`knowledge.path-blacklist` | 控制可搜索的文件范围（复用ReadFileTool白名单） |
| NacosConfigTool | 无独立配置项 | - | 直接通过 INacosConfigService 调用 Nacos OpenAPI |
| CompareConfigTool | 无独立配置项 | - | 复用 INacosConfigService |
| PreDeployCheckTool | 路径白名单（读取部署脚本） | `knowledge.allowed-paths` | 控制可读取的部署脚本路径（复用ReadFileTool白名单） |

> **说明**：`knowledge.allowed-paths` 和 `knowledge.path-blacklist` 为共享配置，ReadFileTool、SearchCodeTool、ListDirTool、SearchInFileTool、PreDeployCheckTool 共用同一组白名单/黑名单。工具启用/禁用状态复用脚手架 `sys_argument` 通用参数表（config_key 格式：`ai.tool.{toolName}.enabled`），具体参数配置存储在 Nacos。

**Nacos 完整配置示例**（dataId: `zmbdp-chat-service-dev.yaml`）：

```yaml
# ==================== 服务配置 ====================
server:
  port: 18084

spring:
  application:
    name: zmbdp-chat-service
  cloud:
    nacos:
      discovery:
        server-addr: ${NACOS_ADDR:nacos-server:8848}
        namespace: ${NACOS_NAMESPACE:public}
      config:
        server-addr: ${NACOS_ADDR:nacos-server:8848}
        namespace: ${NACOS_NAMESPACE:public}
        import-check:
          enabled: false
        import:
          - optional:nacos:share-token-dev.yaml
          - optional:nacos:share-redis-dev.yaml
          - optional:nacos:share-mybatis-dev.yaml
          - optional:nacos:share-logging-dev.yaml
    # Sentinel 限流配置
    sentinel:
      transport:
        dashboard: sentinel-dashboard:8858
      eager: true

  # ==================== AI 框架配置（Spring AI Alibaba 1.0.0 GA） ====================
  ai:
    dashscope:
      api-key: ${DASHSCOPE_API_KEY:sk-xxxxxxxxxxxx}  # 通义千问 API Key，从环境变量注入
      chat:
        model: qwen-max                              # 聊天模型
        options:
          temperature: 0.7                           # 温度参数
          max-tokens: 4096                           # 最大 token 数
      embedding:
        model: text-embedding-v1                     # 嵌入模型（1536维）
    rerank:
      model: gte-rerank                              # 重排序模型（阿里云 DashScope）
    # Milvus 向量数据库配置
    vectorstore:
      milvus:
        client:
          host: ${MILVUS_HOST:milvus-server}
          port: ${MILVUS_PORT:19530}
        database-name: default
        collection-name: scaffold_knowledge          # 向量集合名称
        embedding-dimension: 1536                    # 向量维度（text-embedding-v1 = 1536）
        index-type: HNSW                             # 索引类型
        metric-type: IP                              # 相似度度量（内积）
        index-parameters:
          efConstruction: 200
          M: 16
        query-parameters:
          efRuntime: 100                             # 查询时参数（与06-RAG流程设计.md 6.4节一致）

# ==================== RAG 检索配置 ====================
scaffold:
  rag:
    topK: 5                                          # RAG 检索默认 topK（范围 1-20，运行时可被请求参数覆盖）
    context-max-length: 8000                         # Prompt 中文档上下文最大长度（字符）
    memory-rounds: 5                                 # 对话历史保留轮数
```
