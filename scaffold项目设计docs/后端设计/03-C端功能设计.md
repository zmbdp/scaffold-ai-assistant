# 📄 C端功能设计文档

> **文档说明**：本文档详细描述 Scaffold AI Assistant C端（用户端）的功能设计。
> **复用说明**：C端用户认证由 `zmbdp-portal-service` 的 `UserController` 提供登录接口（手机号/邮箱/openId登录），通过 `AppUserApi` Feign接口查询/注册用户信息。
> **架构说明**：`zmbdp-portal-service` 承担业务编排职责，流式对话使用WebClient调用 `zmbdp-chat-service` 的SSE端点。

---

## 3.0 zmbdp-portal-service 业务编排职责

> **重要架构原则**：AI对话**必须使用流式输出**（WebClient + SSE），**不使用Feign**。
> 原因：Feign不支持SSE流式传输，AI响应较慢需要打字机效果提升用户体验。
> Feign仅用于同步的非对话接口（如RAG检索、历史记录查询、知识库管理）。

**流式对话流程**（核心编排逻辑，详细8步见 3.0.1 节）：
```
前端 → zmbdp-portal-service
         ├─ 步骤1: 解析请求（message, sessionId, model等），从 Header 获取 userId/userFrom
         ├─ 步骤2: 生成或复用 sessionId
         ├─ 步骤3: Feign调用 zmbdp-chat-service /chat/retrieve（RAG检索，获取上下文）
         ├─ 步骤4: 拼接完整Prompt（system prompt + 上下文 + 对话历史 + 用户提问）
         ├─ 步骤5: 构建 ChatStreamReqDTO
         ├─ 步骤6: WebClient调用 zmbdp-chat-service SSE端点 /chat/completions/stream
         ├─ 步骤7: 透传Flux流数据给前端
         └─ 步骤8: 流结束后异步保存对话历史（Feign 调用 HistoryApi）
```

**流式图文对话流程**（与上文类似，SSE端点为 `/chat/image/completions/stream`，请求体含 images 字段）：
```
前端 → zmbdp-portal-service
         ├─ 步骤1: 解析请求（message, images, sessionId, model等），从 Header 获取 userId/userFrom
         ├─ 步骤2: 生成或复用 sessionId
         ├─ 步骤3: Feign调用 zmbdp-chat-service /chat/retrieve（RAG检索，获取上下文）
         ├─ 步骤4: 拼接完整Prompt
         ├─ 步骤5: 构建 ChatWithImageStreamReqDTO（含 images）
         ├─ 步骤6: WebClient调用 zmbdp-chat-service SSE端点 /chat/image/completions/stream
         ├─ 步骤7: 透传Flux流数据给前端
         └─ 步骤8: 流结束后异步保存对话历史（Feign 调用 HistoryApi）
```

**关键设计**：
- **RAG 检索在 `zmbdp-chat-service` 完成**（portal-service 通过 Feign 调用 `ChatApi.retrieve()` 获取检索结果），portal-service 只做业务编排（调用检索接口 → 拼接 Prompt → WebClient 调用 SSE 端点 → 透传流数据）
- 流式调用使用WebClient而非Feign，避免Feign的流式限制
- `zmbdp-chat-service` 专注于 AI 核心能力实现（RAG 检索、模型调用、工具调用）
- **不提供同步对话接口**，所有C端AI对话均采用流式输出

### 3.0.1 PortalChatService 详细设计

**类名**：`PortalChatService`（无接口，直接实现）

**路径**：`com.zmbdp.portal.service.service.PortalChatService`

**依赖注入**：
- `ChatApi` - Feign 接口（RAG 检索）
- `HistoryApi` - Feign 接口（对话历史查询、对话memory获取）
- `WebClient.Builder chatWebClientBuilder` - 用于调用 chat-service 的 SSE 端点（通过 `@LoadBalanced` 注入，`build()` 后使用）
- `TokenService` - **可选注入**，仅当方式B（重新解析JWT）时使用（方法：`getLoginUser(token, secret)`，内部会解析JWT + 查Redis）

**核心方法**：

**streamChat()**：
```java
public Flux<String> streamChat(ChatReqDTO request, String token)
```
- **参数**：`request` - 对话请求（message、sessionId、model、temperature、topK），`token` - JWT Token
- **返回值**：`Flux<String>` - SSE 流数据
- **说明**：流式文本对话的核心编排方法

**streamChatWithImage()**：
```java
public Flux<String> streamChatWithImage(ChatWithImageReqDTO request, String token)
```
- **参数**：`request` - 图文对话请求（message、images、sessionId、model），`token` - JWT Token
- **返回值**：`Flux<String>` - SSE 流数据
- **说明**：流式图文对话的核心编排方法

**streamChat 执行流程**：
```
1. 解析 JWT Token，提取 userId、username、userFrom
   └─ 调用 tokenService.getLoginUser(token, secret)（注：TokenService 无 parseToken 方法，实际方法为 getLoginUser，需传入 jwt.token.secret 密钥，从 share-token 共享配置获取）
2. 生成或复用 sessionId
   ├─ 若 request.sessionId 为空，生成新 sessionId（UUID）
   └─ 若 request.sessionId 非空，校验归属当前用户
3. RAG 检索（Feign 调用）：
   ├─ 构建 RetrieveReqDTO（question=message, topK=request.topK或默认5）
   ├─ 调用 chatApi.retrieveContext(retrieveReqDTO)
   └─ 获取 Result<List<DocumentVO>>，提取 List<DocumentVO>
4. 拼接完整 Prompt（详见 06-RAG流程设计.md 6.8节）：
   ├─ System Prompt（从 resources/prompt/system-prompt.txt 加载）
   ├─ 文档上下文（按 score 降序，限制 8000 字符）
   ├─ 对话历史（最近5轮，通过 Feign 调用 HistoryApi.getSessionHistory 获取；该方法优先从 Redis（IChatMemoryService）读取对话memory，Redis 无数据时降级查 MySQL）
   └─ 用户提问
5. 构建 ChatStreamReqDTO：
   ├─ message: request.message
   ├─ sessionId: sessionId
   ├─ prompt: 完整 Prompt
   ├─ model: request.model（默认 qwen-max）
   └─ temperature: request.temperature（默认 0.7）
6. chatWebClientBuilder.build() 调用 chat-service SSE 端点：
   ├─ URL: lb://zmbdp-chat-service/chat/completions/stream（通过 @LoadBalanced 服务发现，**不硬编码 IP/端口**）
   ├─ Method: POST
   ├─ Content-Type: application/json
   ├─ Accept: text/event-stream
   └─ Body: ChatStreamReqDTO
7. 接收 Flux<String>，透传给前端
8. 流结束后异步保存对话历史（Feign 调用 HistoryApi）
```

**WebClient 配置**（位于 `portal-service` 的 `WebClientConfig`）：
```java
@Configuration
public class WebClientConfig {

    @Bean
    @LoadBalanced  // 关键：启用服务发现，通过服务名解析实际地址
    public WebClient.Builder chatWebClientBuilder() {
        // 注入方式：portal-service 中 @Autowired WebClient.Builder chatWebClientBuilder
        // 使用方式：chatWebClientBuilder.build().post().uri("lb://zmbdp-chat-service/...")
        return WebClient.builder()
            .defaultHeader(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE)
            .codecs(configurer -> configurer.defaultCodecs().maxInMemorySize(16 * 1024 * 1024))  // 16MB
            .filter(traceIdExchangeFilter());  // 传递 traceId，与 FeignTraceInterceptor 保持一致
    }

    /**
     * traceId 传递过滤器（参照 zmbdp-common-log 的 FeignTraceInterceptor 模式）
     * 从 MDC 获取 traceId 写入 X-Trace-Id 请求头，
     * chat-service 侧的 TraceIdFilter 会从该 header 提取 traceId 写入 MDC，
     * 实现跨服务链路追踪。
     */
    @Bean
    public ExchangeFilterFunction traceIdExchangeFilter() {
        return (request, next) -> {
            String traceId = MDC.get("traceId");
            if (StringUtil.isNotBlank(traceId)) {
                return next.exchange(request.mutate()
                    .header("X-Trace-Id", traceId)
                    .build());
            }
            return next.exchange(request);
        };
    }
}
```

> **使用方式**：`chatWebClientBuilder.build().post().uri("lb://zmbdp-chat-service/chat/completions/stream")...`
> 
> **关键说明**：
> - 使用 `@LoadBalanced` 注解 + `lb://zmbdp-chat-service` 协议，通过 Nacos 服务发现解析实际地址，**不硬编码 IP/端口**
> - 端口 18084 由 Nacos 注册的服务实例提供，WebClient 自动从注册中心获取
> - 如需配置超时、连接池等，在 Builder 中添加相应配置

**错误处理**：
- RAG 检索失败：记录 warning 日志，使用空上下文继续对话（不中断流程）
- WebClient 调用失败：返回错误帧 `{"error": "AI服务连接失败", "code": 500023, "done": true}`
- WebClient 超时（60秒）：返回超时帧 `{"error": "AI服务响应超时", "code": 500024, "done": true}`

### 3.0.2 C端认证流程

**认证方式**：JWT Token（通过 Gateway 的 AuthFilter 校验）

**认证流程**（基于 `AuthFilter.java`、`TokenService.java`、`JwtUtil.java` 源码）：
```
1. 前端请求携带 Authorization: Bearer {token}
2. Gateway 的 AuthFilter 校验 JWT（源码：zmbdp-gateway/AuthFilter.java）：
   ├─ 检查请求路径是否在白名单中（IgnoreWhiteProperties），白名单内直接放行
   ├─ 从 Authorization Header 提取 token（去除 "Bearer " 前缀，TokenConstants.PREFIX）
   ├─ JwtUtil.parseToken(token, secret) 解析 JWT，获取 Claims（HS512算法）
   ├─ JwtUtil.getUserKey(claims) 提取 userKey（UUID）
   ├─ 检查 Redis 中是否存在 logintoken:{userKey}，不存在则登录已过期（LOGIN_STATUS_OVERTIME）
   ├─ JwtUtil.getUserFrom(claims) 提取 userFrom 字段
   ├─ **来源校验（脚手架现有逻辑）**：若 URL 包含 `sys_user`（HttpConstants.SYS_USER_PATH）且 userFrom 不是系统用户（USER_FROM_TU_B="sys"），则拒绝（TOKEN_CHECK_FAILED）
   │   ├─ 该校验保证 C端用户（userFrom="app"）无法访问 B端系统用户接口（/sys_user/**）
   │   └─ **v1.0 无需改动此逻辑**：C端接口由 portal-service 提供（路径为 /user/**、/chat/**、/history/** 等），不含 "sys_user"，天然通过校验
   └─ 将 userKey、userId、userName、userFrom 设置到请求 Header 中转发（URL编码）
3. Gateway 转发请求到 portal-service（StripPrefix=1）
4. portal-service 的 Controller 接收请求，获取用户信息（两种方式）：
   ├─ 方式A（推荐）：从 Header 获取用户信息（AuthFilter 已设置 USER_KEY、USER_ID、USERNAME、USER_FROM Header，无需重复解析JWT）
   └─ 方式B（备选）：调用 tokenService.getLoginUser(token, secret) 重新解析（内部会再次解析JWT + 查Redis，存在重复开销）
```

> **重要说明（基于源码）**：
> - **AuthFilter 来源校验逻辑**：脚手架现有实现为 `url.contains(HttpConstants.SYS_USER_PATH)`（即 `url.contains("sys_user")`），用于阻止 C端用户访问 B端系统用户接口。v1.0 **无需改动**此逻辑——C端接口由 `zmbdp-portal-service` 提供（路径 `/user/**`、`/chat/**`、`/history/**` 等），不包含 "sys_user" 字符串，天然通过校验
> - **方式A vs 方式B**：方式A直接从Header读（Gateway已校验），无重复开销；方式B调用 `tokenService.getLoginUser(token, secret)`，会再次执行"解析JWT + 查Redis"，存在重复劳动。**推荐方式A**
> - `tokenService.parseToken()` 方法**不存在**，实际方法是 `tokenService.getLoginUser(token, secret)`（需传入 jwt.token.secret 密钥，位于 share-token-${env}.yaml）
> - **Redis Key 格式**：`logintoken:{uuid}`，存储完整的 `LoginUserDTO` 对象（含 userId、userName、userFrom、token、loginTime、expireTime）

**C端用户登录方式**（由 `zmbdp-portal-service` 的 `UserController` 提供，对应 `/user/login/code`、`/user/login/wechat` 接口）：
- 手机号 + 验证码登录（`/user/send_code` 下发验证码，`/user/login/code` 登录）
- openId 登录（微信，`/user/login/wechat`）
- 邮箱 + 验证码登录（`/user/send_code` 下发验证码，`/user/login/code` 登录，`AccountManager` 自动识别邮箱格式）

> **代码依据**：`CodeLoginStrategy` 同时支持手机号和邮箱的验证码登录（通过 `AccountManager.validateAndFindOrRegisterUser(account)` 自动识别账号格式），不提供邮箱+密码登录。

> **登录服务归属说明（基于源码）**：
> - **C端登录接口**由 `zmbdp-portal-service` 的 `UserController` 处理（路径 `/user/login/code`、`/user/login/wechat`），**不是** `zmbdp-admin-service`
> - `zmbdp-admin-service` 的 `SysUserController`（路径 `/sys_user/login/password`）是 **B端登录**，仅支持账号密码登录，**不参与** C端登录流程
> - 登录成功后，由 `TokenService.createToken()` 生成 JWT Token，前端携带 Token 请求后续接口
> - `AppUserApi` 是 Feign 接口，供 portal-service 查询/注册 C端用户信息（如首次 openId 登录时自动注册），**不是**登录入口

## 3.2 功能模块详细设计

### 3.2.1 AI对话模块（核心）

**功能描述**：与AI助手进行对话，获取关于脚手架的专业回答

**输入输出示例**：

| 输入 | 输出 |
|-----|------|
| "三级缓存怎么用？入参出参是什么？" | 说明使用方法，给出代码示例和参数说明 |
| "@Idempotent注解有哪些参数？" | 列出所有参数，给出使用示例 |

**实现方式**：RAG（基于docs/*.md文档 + JavaDoc）+ Agent工具调用

**核心逻辑**：
```
1. 用户提问 → Embedding生成向量
2. Milvus检索相关文档
3. 判断是否需要工具调用（实时源码查询等）
4. 如果需要，调用工具获取实时信息
5. 拼接上下文 → LLM生成回答
6. 返回回答（包含引用来源）
```

**请求参数**（`ChatReqDTO`）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `message` | String | 是 | 用户消息 |
| `sessionId` | String | 否 | 会话ID，不传则新建会话 |
| `model` | String | 否 | 模型名称，默认使用配置的主模型 |
| `temperature` | Double | 否 | 温度系数（0-1），默认0.7 |
| `topK` | Integer | 否 | RAG检索数量（1-20），默认5 |

**图文请求参数**（`ChatWithImageReqDTO`，继承 ChatReqDTO）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `images` | List<String> | 是 | 图片Base64编码列表 |

**SSE 响应帧格式**（参考 OpenAI Assistants API、LangChain Streaming 的 tool_start/tool_end 事件对设计）：
```
# 内容帧
data: {"chunk": "回答内容片段", "done": false}

# 工具调用开始帧（AI决定调用工具时发送，让前端展示"正在调用工具"）
data: {"toolCall": {"name": "ReadFileTool", "args": {"filePath": "/path"}}, "done": false}

# 工具调用结果帧（工具执行完成后发送，让前端展示工具返回了什么）
data: {"toolResult": {"name": "ReadFileTool", "success": true, "summary": "读取文件成功，共123行", "duration": 45}, "done": false}

# 结束帧
data: {"chunk": "", "done": true, "sessionId": "xxx", "sources": [...], "model": "qwen-max"}

# 错误帧
data: {"error": "AI服务连接失败", "code": 500023, "done": true}
```

> **工具调用帧设计说明**（市面主流做法）：
> - **工具调用开始帧（toolCall）**：AI决定调用工具时立即发送，前端可展示"正在调用 ReadFileTool..."的加载状态
> - **工具调用结果帧（toolResult）**：工具执行完成后发送，让用户看到工具调用的结果摘要（不返回完整结果，避免SSE流过大；完整结果由AI整合到后续回答中）
> - 参考实现：OpenAI Assistants API（`thread.run.step` 事件）、LangChain（`on_tool_start`/`on_tool_end` 回调）、Dify（工具节点流式状态）

**响应字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `chunk` | String | AI回答内容片段（流式输出） |
| `done` | Boolean | 是否结束（true=流结束） |
| `sessionId` | String | 会话ID（结束帧返回） |
| `sources` | List | 引用来源列表（结束帧返回） |
| `sources[].title` | String | 引用文档标题 |
| `sources[].path` | String | 引用文档路径 |
| `sources[].module` | String | 所属模块 |
| `model` | String | 实际使用的模型名称 |
| `toolCall` | Object | 工具调用开始信息（可选，AI决定调用工具时发送） |
| `toolCall.name` | String | 工具名称 |
| `toolCall.args` | Object | 工具调用参数 |
| `toolResult` | Object | 工具调用结果信息（可选，工具执行完成后发送） |
| `toolResult.name` | String | 工具名称 |
| `toolResult.success` | Boolean | 是否执行成功 |
| `toolResult.summary` | String | 结果摘要（非完整结果，避免SSE流过大） |
| `toolResult.duration` | Long | 工具执行耗时（毫秒） |
| `error` | String | 错误信息（错误帧） |
| `code` | Integer | 错误码（错误帧） |

### 3.2.2 历史记录模块

**功能描述**：查看和管理个人对话历史

**输入输出示例**：

| 操作 | 输出 |
|-----|------|
| 查看历史 | 显示对话列表，按时间排序 |
| 搜索历史 | 按关键词搜索对话记录 |
| 删除历史 | 删除指定对话记录 |
| 继续对话 | 基于历史记录继续提问 |

**实现方式**：Redis 短期对话记忆 + MySQL 持久化历史记录（双存储策略）

> **双存储策略说明**（与 07-项目架构设计.md 一致）：
> - **Redis**（`IChatMemoryService`）：存储**近期对话上下文**，供 AI 模型作为 memory 使用（保持对话连贯性），TTL 7天，过期自动清理
> - **MySQL**（`IHistoryService` + `sys_ai_conversation` 表）：持久化**完整对话历史**，供用户查看历史记录、搜索历史、继续对话，不自动过期
> - 流式对话结束后，portal-service 异步调用 `HistoryApi` 将对话保存到 MySQL

**存储结构**：

**Redis（对话记忆，供 AI 使用，详见 07-项目架构设计.md 7.2.18 节）**：
```
Key: ai:chat:memory:{sessionId}    （与 AiCacheConstants.CHAT_MEMORY_REDIS_PREFIX 一致）
Type: List
Value: JSON 序列化的 ChatMessage 列表（lPush 头部插入，每会话最多 100 条消息，约 50 轮）
TTL: 7天（604800秒）
L1 Caffeine 缓存：key=CHAT_MEMORY:ai:chat:memory:{sessionId}，TTL=5分钟，最大1000条
注：Prompt 拼接时取最近 5 轮（scaffold.rag.memory-rounds 配置），与 Redis 存储 100 条上限是不同概念
```

**MySQL（持久化历史，供用户查看）**：
```
表: sys_ai_conversation（详见 07-项目架构设计.md 7.4.4节）
每轮对话一条记录（question/answer 模型，非每条消息一条记录）
字段: id, session_id, user_id, user_from, question, answer, images, model, 
      temperature, top_k, response_time, token_usage, tool_usage, sources, 
      status, is_deleted, error_msg, create_date, create_time
索引: idx_session_id, idx_user_id, idx_user_from, idx_model, 
      idx_status, idx_is_deleted, idx_create_date, idx_create_time
```

**ChatMessage 字段定义**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `role` | String | 角色（user/assistant） |
| `content` | String | 消息内容 |
| `timestamp` | Long | 时间戳（毫秒） |
| `model` | String | 模型名称（assistant 消息才有） |
| `hasImage` | Boolean | 是否包含图片 |
| `sources` | List | 引用来源列表（assistant 消息才有） |
| `tokenUsage` | Integer | Token 消耗量（assistant 消息才有） |

**历史记录 API**：

| HTTP方法 | 前端请求路径 | 说明 |
|---------|------------|------|
| GET | `/portal/history` | 获取对话历史列表（分页） |
| GET | `/portal/history/{sessionId}` | 获取会话详情 |

**历史列表响应**（`HistoryVO`）：

**ChatMessageVO 字段**：

### 3.2.4 引用查看模块

**功能描述**：查看AI回答引用的文档来源，方便深入学习

**输入输出示例**：

| 操作 | 输出 |
|-----|------|
| 查看引用 | 显示引用的文档列表，包含文档名称、路径、引用内容 |
| 点击引用 | 跳转到对应文档的详细内容 |

**实现方式**：引用来源在 SSE 结束帧的 `sources` 字段中返回，不需要独立 API。

**引用来源数据结构**（SSE 结束帧的 `sources` 字段）：

```json
{
  "sources": [
    {
      "documentId": 1,
      "title": "三级缓存架构.md",
      "path": "/path/to/docs/CACHE_ARCHITECTURE.md",
      "module": "common-cache",
      "category": "cache",
      "score": 0.95,
      "chunkIndex": 0,
      "content": "引用的分块内容片段..."
    }
  ]
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `documentId` | Long | 文档ID |
| `title` | String | 文档标题 |
| `path` | String | 文件路径 |
| `module` | String | 所属模块 |
| `category` | String | 功能类别 |
| `score` | Double | 相似度分数（0-1） |
| `chunkIndex` | Integer | 分块序号 |
| `content` | String | 引用的分块内容片段 |

**前端交互流程**：
```
1. 前端接收 SSE 结束帧，解析 sources 字段
2. 在回答下方展示引用来源列表（显示标题、模块、相似度）
3. 用户点击某个引用来源
4. 前端直接展开 sources 中对应的 content 字段，展示引用的分块内容片段
5. 如需查看完整文档，跳转到B端知识库管理页面（需B端权限）
```

> **设计说明**：C端引用查看**不调用任何后端接口**，直接使用 SSE 结束帧中 sources 数组里的 content 字段展示分块内容。这样设计的原因：
> - 避免C端用户调历史接口（语义不对，历史接口是查对话记录的，不是查文档内容的）
> - sources 中已包含 content 字段，无需额外请求
> - 如需查看完整文档，由B端知识库管理（`GET /admin/knowledge/documents/{id}`）负责，C端不提供此能力

### 3.2.5 回答反馈模块

**功能描述**：用户对AI回答进行点赞/点踩反馈，形成产品质量闭环，反哺B端统计模块的"回答满意度"指标

**业务价值**：
- 发现RAG检索效果差的问题（点踩"答非所问"→ 检索策略需优化）
- 发现知识库内容过时的问题（点踩"信息过时"→ 知识同步需更新）
- 为B端提供"回答满意度"统计指标，量化AI回答质量

**输入输出示例**：

| 操作 | 输入 | 输出 |
|-----|------|------|
| 点赞 | conversationId + feedbackType=LIKE | 反馈成功 |
| 点踩 | conversationId + feedbackType=DISLIKE + dislikeReason + comment（可选） | 反馈成功 |
| 查看反馈状态 | conversationId | 当前用户对该对话的反馈状态 |
| 撤销反馈 | conversationId | 撤销成功 |

**反馈类型与点踩原因**（字段名与 07/08 文档统一，枚举值大写，见 07 文档 FeedbackType/DislikeReason 枚举）：

| 字段 | 值 | 说明 |
|------|-----|------|
| `feedbackType` | `LIKE` | 点赞（回答有帮助） |
| `feedbackType` | `DISLIKE` | 点踩（回答有问题） |
| `dislikeReason`（仅feedbackType=DISLIKE时填写） | `OUTDATED` | 信息过时 |
| `dislikeReason` | `IRRELEVANT` | 答非所问 |
| `dislikeReason` | `CODE_ERROR` | 代码错误 |
| `dislikeReason` | `OTHER` | 其他 |
| `comment`（可选） | String | 文字评论（最多500字） |

**请求参数**（`FeedbackReqDTO`）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `conversationId` | Long | 是 | 对话记录ID（从SSE结束帧或历史记录获取） |
| `feedbackType` | String | 是 | 反馈类型（LIKE/DISLIKE，见 FeedbackType 枚举） |
| `dislikeReason` | String | feedbackType=DISLIKE时必填 | 点踩原因（OUTDATED/IRRELEVANT/CODE_ERROR/OTHER，见 DislikeReason 枚举） |
| `comment` | String | 否 | 文字评论（最多500字） |

**反馈 API**：

| HTTP方法 | 前端请求路径 | 说明 |
|---------|------------|------|
| POST | `/portal/feedback` | 提交反馈（点赞/点踩） |
| GET | `/portal/feedback/{conversationId}` | 查询当前用户对该对话的反馈状态 |
| DELETE | `/portal/feedback/{conversationId}` | 撤销反馈 |

**响应结构**（`FeedbackVO`，字段与 07 文档 7.3.28、08 文档 8.6.30 一致）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Long | 反馈记录ID |
| `conversationId` | Long | 对话记录ID |
| `userId` | Long | 用户ID |
| `feedbackType` | String | 反馈类型（LIKE/DISLIKE，null表示未反馈） |
| `dislikeReason` | String | 点踩原因（仅feedbackType=DISLIKE时有值） |
| `comment` | String | 文字评论 |
| `createDate` | Long | 反馈日期（YYYYMMDD，BIGINT） |
| `createTime` | String | 反馈时间（yyyy-MM-dd HH:mm:ss） |

**前端交互流程**：
```
1. AI回答完成后（SSE结束帧），在回答下方显示 👍 / 👎 按钮
2. 用户点击 👍 → 调用 POST /portal/feedback {"conversationId": xxx, "feedbackType": "LIKE"}
3. 用户点击 👎 → 弹出点踩原因选择框（信息过时/答非所问/代码错误/其他）
   ├─ 用户选择原因后，可选填文字评论
   └─ 调用 POST /portal/feedback {"conversationId": xxx, "feedbackType": "DISLIKE", "dislikeReason": "IRRELEVANT", "comment": "可选"}
4. 反馈成功后，按钮变为已选中状态（👍 高亮 / 👎 高亮）
5. 用户可再次点击切换反馈类型，或点击"撤销"调用 DELETE /portal/feedback/{conversationId}
6. 用户查看历史记录时，通过 GET /portal/feedback/{conversationId} 恢复反馈状态显示
```

**业务规则**：
- 每个用户对每条对话只能有一个反馈（点赞或点踩，二选一）
- 重复提交反馈：覆盖上一次的反馈（如从点赞改为点踩）
- 反馈权限：只能对自己的对话记录进行反馈（校验 conversation.userId == 当前用户ID）
- 反馈数据存储在 `sys_ai_feedback` 表（详见 07-项目架构设计.md）
