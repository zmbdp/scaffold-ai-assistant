# 📄 C端功能设计文档

> **文档说明**：本文档详细描述 Scaffold AI Assistant C端（用户端）的功能设计。
> **复用说明**：C端用户认证复用现有 `zmbdp-admin-service` 的用户体系（手机号/邮箱/openId登录），通过 `AppUserApi` Feign接口实现。
> **架构说明**：`zmbdp-portal-service` 承担业务编排职责，流式对话使用WebClient调用 `zmbdp-chat-service` 的SSE端点。

---

## 3.0 zmbdp-portal-service 业务编排职责

> **重要架构原则**：AI对话**必须使用流式输出**（WebClient + SSE），**不使用Feign**。
> 原因：Feign不支持SSE流式传输，AI响应较慢需要打字机效果提升用户体验。
> Feign仅用于同步的非对话接口（如RAG检索、历史记录查询、知识库管理）。

**流式对话流程**（核心编排逻辑）：
```
前端 → zmbdp-portal-service
         ├─ 步骤1: 解析请求（message, sessionId, model等）
         ├─ 步骤2: Feign调用 zmbdp-chat-service /chat/retrieve（RAG检索，获取上下文）
         ├─ 步骤3: 拼接完整Prompt（system prompt + 上下文 + 用户提问）
         ├─ 步骤4: WebClient调用 zmbdp-chat-service SSE端点 /chat/completions/stream
         └─ 步骤5: 透传Flux流数据给前端
```

**流式图文对话流程**：
```
前端 → zmbdp-portal-service
         ├─ 步骤1: 解析请求（message, images, sessionId, model等）
         ├─ 步骤2: Feign调用 zmbdp-chat-service /chat/retrieve（RAG检索，获取上下文）
         ├─ 步骤3: 拼接完整Prompt
         ├─ 步骤4: WebClient调用 zmbdp-chat-service SSE端点 /chat/image/completions/stream
         └─ 步骤5: 透传Flux流数据给前端
```

**关键设计**：
- RAG检索在 `zmbdp-portal-service` 层完成，便于灵活调整Prompt拼接逻辑
- 流式调用使用WebClient而非Feign，避免Feign的流式限制
- `zmbdp-chat-service` 专注于AI核心能力实现，不处理业务编排
- **不提供同步对话接口**，所有C端AI对话均采用流式输出

### 3.0.1 PortalChatService 详细设计

**类名**：`PortalChatService`（无接口，直接实现）

**路径**：`com.zmbdp.portal.service.service.PortalChatService`

**依赖注入**：
- `ChatApi` - Feign 接口（RAG 检索）
- `HistoryApi` - Feign 接口（对话历史）
- `WebClient chatWebClient` - 用于调用 chat-service 的 SSE 端点
- `TokenService` - 用于从 JWT 解析用户信息

**核心方法**：

**streamChat()**：
```java
public Flux<String> streamChat(ChatReqDTO request, String token)
```
- **参数**：`request` - 对话请求（message、sessionId、model、temperature、topK），`token` - JWT Token
- **返回值**：`Flux<String>` - SSE 流数据
- **说明**：流式文本对话的核心编排方法

**streamChatWithImage()**：
```java
public Flux<String> streamChatWithImage(ChatWithImageReqDTO request, String token)
```
- **参数**：`request` - 图文对话请求（message、images、sessionId、model），`token` - JWT Token
- **返回值**：`Flux<String>` - SSE 流数据
- **说明**：流式图文对话的核心编排方法

**streamChat 执行流程**：
```
1. 解析 JWT Token，提取 userId、username、userFrom
   └─ 调用 tokenService.parseToken(token)
2. 生成或复用 sessionId
   ├─ 若 request.sessionId 为空，生成新 sessionId（UUID）
   └─ 若 request.sessionId 非空，校验归属当前用户
3. RAG 检索（Feign 调用）：
   ├─ 构建 RetrieveReqDTO（question=message, topK=request.topK或默认5）
   ├─ 调用 chatApi.retrieveContext(retrieveReqDTO)
   └─ 获取 Result<List<DocumentVO>>，提取 List<DocumentVO>
4. 拼接完整 Prompt（详见 06-RAG流程设计.md 6.8节）：
   ├─ System Prompt（从 resources/prompt/system-prompt.txt 加载）
   ├─ 文档上下文（按 score 降序，限制 8000 字符）
   ├─ 对话历史（最近5轮，从 HistoryApi 获取）
   └─ 用户提问
5. 构建 ChatStreamReqDTO：
   ├─ message: request.message
   ├─ sessionId: sessionId
   ├─ prompt: 完整 Prompt
   ├─ model: request.model（默认 qwen-max）
   └─ temperature: request.temperature（默认 0.7）
6. WebClient 调用 chat-service SSE 端点：
   ├─ URL: http://{chat-service-ip}:18084/chat/completions/stream
   ├─ Method: POST
   ├─ Content-Type: application/json
   ├─ Accept: text/event-stream
   └─ Body: ChatStreamReqDTO
7. 接收 Flux<String>，透传给前端
8. 流结束后异步保存对话历史（Feign 调用 HistoryApi）
```

**WebClient 配置**（位于 `portal-service` 的 `WebClientConfig`）：
```java
@Configuration
public class WebClientConfig {
    @Bean
    public WebClient chatWebClient() {
        return WebClient.builder()
            .baseUrl("http://zmbdp-chat-service:18084")  // Docker 内部服务名
            .defaultHeader(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE)
            .codecs(configurer -> configurer.defaultCodecs().maxInMemorySize(16 * 1024 * 1024))  // 16MB
            .build();
    }
}
```

**错误处理**：
- RAG 检索失败：记录 warning 日志，使用空上下文继续对话（不中断流程）
- WebClient 调用失败：返回错误帧 `{"error": "AI服务连接失败", "code": 500023, "done": true}`
- WebClient 超时（60秒）：返回超时帧 `{"error": "AI服务响应超时", "code": 500024, "done": true}`

### 3.0.2 C端认证流程

**认证方式**：JWT Token（通过 Gateway 的 AuthFilter 校验）

**认证流程**：
```
1. 前端请求携带 Authorization: Bearer {token}
2. Gateway 的 AuthFilter 校验 JWT：
   ├─ 解析 token，验证有效期
   ├─ 提取 userFrom 字段（tu/sys）
   ├─ 若 userFrom=tu（C端用户），允许访问 /portal/**
   └─ 若 userFrom=sys（B端用户），禁止访问 /portal/**
3. Gateway 转发请求到 portal-service（StripPrefix=1）
4. portal-service 的 Controller 接收请求：
   ├─ 从 Request Header 提取 token
   └─ 调用 tokenService.parseToken(token) 获取用户信息
```

**C端用户登录方式**（复用现有 zmbdp-admin-service）：
- 手机号 + 验证码登录
- openId 登录（微信/支付宝）
- 邮箱 + 密码登录

> **说明**：C端用户认证复用现有 `zmbdp-admin-service` 的用户体系，通过 `AppUserApi` Feign接口实现。

## 3.2 功能模块详细设计

### 3.2.1 AI对话模块（核心）

**功能描述**：与AI助手进行对话，获取关于脚手架的专业回答

**输入输出示例**：

| 输入 | 输出 |
|-----|------|
| "三级缓存怎么用？入参出参是什么？" | 说明使用方法，给出代码示例和参数说明 |
| "@Idempotent注解有哪些参数？" | 列出所有参数，给出使用示例 |

**实现方式**：RAG（基于docs/*.md文档 + JavaDoc）+ Agent工具调用

**核心逻辑**：
```
1. 用户提问 → Embedding生成向量
2. Milvus检索相关文档
3. 判断是否需要工具调用（实时源码查询等）
4. 如果需要，调用工具获取实时信息
5. 拼接上下文 → LLM生成回答
6. 返回回答（包含引用来源）
```

**请求参数**（`ChatReqDTO`）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `message` | String | 是 | 用户消息 |
| `sessionId` | String | 否 | 会话ID，不传则新建会话 |
| `model` | String | 否 | 模型名称，默认使用配置的主模型 |
| `temperature` | Double | 否 | 温度系数（0-1），默认0.7 |
| `topK` | Integer | 否 | RAG检索数量（1-20），默认5 |

**图文请求参数**（`ChatWithImageReqDTO`，继承 ChatReqDTO）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `images` | List<String> | 是 | 图片Base64编码列表 |

**SSE 响应帧格式**：
```
# 内容帧
data: {"chunk": "回答内容片段", "done": false}

# 工具调用帧（可选）
data: {"toolCall": {"name": "ReadFileTool", "args": {"filePath": "/path"}}, "done": false}

# 结束帧
data: {"chunk": "", "done": true, "sessionId": "xxx", "sources": [...], "model": "qwen-max"}

# 错误帧
data: {"error": "AI服务连接失败", "code": 500023, "done": true}
```

**响应字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `chunk` | String | AI回答内容片段（流式输出） |
| `done` | Boolean | 是否结束（true=流结束） |
| `sessionId` | String | 会话ID（结束帧返回） |
| `sources` | List | 引用来源列表（结束帧返回） |
| `sources[].title` | String | 引用文档标题 |
| `sources[].path` | String | 引用文档路径 |
| `sources[].module` | String | 所属模块 |
| `model` | String | 实际使用的模型名称 |
| `toolCall` | Object | 工具调用信息（可选） |
| `error` | String | 错误信息（错误帧） |
| `code` | Integer | 错误码（错误帧） |

### 3.2.2 历史记录模块

**功能描述**：查看和管理个人对话历史

**输入输出示例**：

| 操作 | 输出 |
|-----|------|
| 查看历史 | 显示对话列表，按时间排序 |
| 搜索历史 | 按关键词搜索对话记录 |
| 删除历史 | 删除指定对话记录 |
| 继续对话 | 基于历史记录继续提问 |

**实现方式**：Redis存储对话历史 + 分页查询

**存储结构**：
```
Key: chat:history:{sessionId}
Type: List
Value: JSON 序列化的 ChatMessage 列表
TTL: 7天（604800秒）
```

**ChatMessage 字段定义**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `role` | String | 角色（user/assistant） |
| `content` | String | 消息内容 |
| `timestamp` | Long | 时间戳（毫秒） |
| `model` | String | 模型名称（assistant 消息才有） |
| `hasImage` | Boolean | 是否包含图片 |
| `sources` | List | 引用来源列表（assistant 消息才有） |
| `tokenUsage` | Integer | Token 消耗量（assistant 消息才有） |

**历史记录 API**：

| HTTP方法 | 前端请求路径 | 说明 |
|---------|------------|------|
| GET | `/portal/history` | 获取对话历史列表（分页） |
| GET | `/portal/history/{sessionId}` | 获取会话详情 |
| DELETE | `/portal/history/{sessionId}` | 删除会话 |

**历史列表响应**（`HistoryVO`）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `sessionId` | String | 会话ID |
| `lastMessage` | String | 最后一条消息 |
| `timestamp` | Long | 最后消息时间戳 |
| `model` | String | 使用的模型 |
| `messageCount` | Integer | 消息数量 |

**会话详情响应**（`HistoryDetailVO`）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `sessionId` | String | 会话ID |
| `messages` | List<ChatMessageVO> | 消息列表 |

**ChatMessageVO 字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `role` | String | 角色（user/assistant） |
| `content` | String | 消息内容 |
| `timestamp` | Long | 时间戳 |
| `hasImage` | Boolean | 是否包含图片 |
| `sources` | List | 引用来源（assistant 才有） |
| `model` | String | 模型名称（assistant 才有） |

### 3.2.3 问题分类模块

**功能描述**：按类别浏览常见问题，快速找到答案

**分类列表**：

| 分类 | 说明 | 预设问题示例 |
|-----|------|------------|
| 架构设计 | 项目结构、模块职责、依赖关系 | "项目结构是什么样的？" |
| 核心功能 | 三级缓存、幂等性、限流、数据权限 | "三级缓存怎么用？" |
| API参考 | 类/方法的入参出参、使用示例 | "ResultCode有哪些错误码？" |
| 部署配置 | 部署步骤、配置修改、环境差异 | "dev环境怎么部署？" |
| 问题排查 | 错误定位、解决方案 | "Token验证失败怎么办？" |

**实现方式**：问题分类不需要独立 API，通过前端预设分类标签实现。用户点击分类后，发送预设问题到 `/portal/chat/stream` 接口，走标准 AI 对话流程。

**前端交互流程**：
```
1. 前端展示 5 个分类标签（架构设计、核心功能、API参考、部署配置、问题排查）
2. 用户点击某个分类标签
3. 前端展示该分类下的预设问题列表（3-5 个问题）
4. 用户点击某个预设问题
5. 前端发送 POST /portal/chat/stream，请求体 {"message": "预设问题内容"}
6. 走标准 AI 对话流程，返回流式回答
```

**预设问题配置**：预设问题存储在前端配置文件中（不需要后端接口），可根据需要动态调整。

### 3.2.4 引用查看模块

**功能描述**：查看AI回答引用的文档来源，方便深入学习

**输入输出示例**：

| 操作 | 输出 |
|-----|------|
| 查看引用 | 显示引用的文档列表，包含文档名称、路径、引用内容 |
| 点击引用 | 跳转到对应文档的详细内容 |

**实现方式**：引用来源在 SSE 结束帧的 `sources` 字段中返回，不需要独立 API。

**引用来源数据结构**（SSE 结束帧的 `sources` 字段）：

```json
{
  "sources": [
    {
      "documentId": 1,
      "title": "三级缓存架构.md",
      "path": "/path/to/docs/CACHE_ARCHITECTURE.md",
      "module": "common-cache",
      "category": "cache",
      "score": 0.95,
      "chunkIndex": 0,
      "content": "引用的分块内容片段..."
    }
  ]
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `documentId` | Long | 文档ID |
| `title` | String | 文档标题 |
| `path` | String | 文件路径 |
| `module` | String | 所属模块 |
| `category` | String | 功能类别 |
| `score` | Double | 相似度分数（0-1） |
| `chunkIndex` | Integer | 分块序号 |
| `content` | String | 引用的分块内容片段 |

**前端交互流程**：
```
1. 前端接收 SSE 结束帧，解析 sources 字段
2. 在回答下方展示引用来源列表（显示标题、模块、相似度）
3. 用户点击某个引用来源
4. 前端发送 GET /portal/history/{sessionId} 或直接使用 sources 中的 content 展示
5. 展示引用文档的详细内容（分块内容）
```

**获取文档详情**（可选）：如需查看完整文档内容，可通过 B 端接口 `GET /admin/knowledge/documents/{id}` 获取（需 B 端权限）。C 端默认只展示 sources 中的 content 片段。

## 3.4 C端功能优先级

| 优先级 | 功能模块 | 理由 |
