# 📄 部署方案文档

> **文档说明**：本文档详细描述 Scaffold AI Assistant 的部署方案。核心原则：复用现有脚手架的基础组件（MySQL、Redis、Nacos），仅新增 Milvus 向量数据库。

---

## 9.0 基础组件复用说明

| 组件 | 是否复用 | 说明 |
|-----|---------|------|
| **MySQL** | ✅ 复用 | 使用现有脚手架的 MySQL 数据库，共享 `sys_user` 表（identity字段标识身份，权限通过Gateway的userFrom字段控制） |
| **Redis** | ✅ 复用 | 使用现有脚手架的 Redis，用于 Token 存储、对话记忆、限流、幂等性 |
| **Nacos** | ✅ 复用 | 使用现有脚手架的 Nacos，用于配置管理和服务发现 |
| **Gateway** | ✅ 复用 | 使用现有脚手架的 Gateway，仅需新增路由配置 |
| **SkyWalking** | ✅ 复用 | 使用现有脚手架的 SkyWalking，用于分布式链路追踪（详见 9.9.3 节） |
| **Prometheus** | ✅ 复用 | 使用现有脚手架的 Prometheus，用于监控 |
| **Grafana** | ✅ 复用 | 使用现有脚手架的 Grafana，用于可视化 |
| **Milvus** | ❌ 新增 | 向量数据库，现有脚手架未包含，需新增部署 |

## 9.1 端口规范

> **说明**：以下端口来自现有脚手架 `zmbdp-{service}-service-{env}.yaml` Nacos 配置文件，dev 和 prd 环境端口一致。chat-service 新增端口遵循现有命名规律（18081-18085 已被占用，chat-service 使用 18084）。

| 服务 | 端口 | 说明 | 配置来源 |
|-----|------|------|---------|
| **zmbdp-gateway-service** | 10030 | 网关端口（dev/prd 一致） | `zmbdp-gateway-service-{env}.yaml` |
| **zmbdp-admin-service** | 18081 | B端管理服务端口 | `zmbdp-admin-service-{env}.yaml` |
| **zmbdp-portal-service** | 18083 | C端用户服务端口 | `zmbdp-portal-service-{env}.yaml` |
| **zmbdp-file-service** | 18082 | 文件服务端口 | `zmbdp-file-service-{env}.yaml` |
| **zmbdp-mstemplate-service** | 18085 | 模板服务端口 | `zmbdp-mstemplate-service-{env}.yaml` |
| **zmbdp-chat-service** | 18084 | **新增**：AI 服务端口 | `zmbdp-chat-service-{env}.yaml` |
| **MySQL（dev/test 容器内）** | 3306 | 数据库端口 | docker-compose-mid.yml |
| **MySQL（prd 宿主机映射）** | 3308 | 容器内 3306 映射到宿主机 3308 | docker-compose-mid.yml: `"3308:3306"` |
| **Redis（dev/test）** | 6379 | 单节点缓存端口 | docker-compose-mid.yml |
| **Redis（prd）** | 6380-6385 | 缓存集群端口（3主3从，跨2台VM） | `deploy/prd/vm1/app/redis/init.sh` |
| **Milvus** | 19530 | Milvus主端口（gRPC） | docker-compose |
| **Milvus** | 9091 | Milvus管理端口（HTTP） | docker-compose |

## 9.2 Docker部署

> **复用说明**：脚手架现有服务使用 `docker-maven-plugin`（IDEA 的 deploy 功能）构建并部署到测试环境，prd 环境使用 docker-compose 在本地构建。AI 模块遵循相同部署方式。下面给出的是独立部署 chat-service 时可用的配置，**实际部署时建议复用现有脚手架的 deploy 目录结构**（如 `deploy/dev/app/zmbdp-chat-service/`）。

**Dockerfile**（放在 `zmbdp-chat-service/src/main/docker/` 下）：
```dockerfile
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY target/zmbdp-chat-service.jar app.jar
EXPOSE 18084
# 注意：exec 形式（JSON数组）不经过 shell，$JAVA_OPTS 不会展开
# 必须用 sh -c 包装，让 shell 展开 $JAVA_OPTS 环境变量
ENTRYPOINT ["sh", "-c", "exec java $JAVA_OPTS -jar app.jar"]
```

## 9.3 docker-compose配置

> **说明**：以下配置是独立部署 chat-service 的简化配置。实际部署时，chat-service 应作为现有 `deploy/{env}/app/docker-compose-app.yml` 的一个 service 加入，复用现有的 MySQL/Redis/Nacos 网络。

### 9.3.1 开发环境（docker-compose-dev.yml，独立部署参考）

> **重要说明**：Milvus 2.4.x standalone 模式**依赖 etcd（元数据存储）和 minio（对象存储）**，单独启动 milvus 容器会失败。以下配置包含完整的三个依赖服务。
>
> **网络复用**：实际部署时应加入脚手架的 `docker-compose-mid.yml` 所在的 `network_dev` 网络，复用 MySQL/Redis/Nacos 等基础组件。

```yaml
version: '3.8'
services:
  zmbdp-chat-service:
    build: .
    container_name: zmbdp-chat-service
    ports:
      - "18084:18084"
    environment:
      - RUN_ENV=dev
      - NACOS_ADDR=scaffold-ai-assistant-nacos:8848
      - KNOWLEDGE_BASE_PATH=/knowledge
      - JAVA_OPTS=-Xms512m -Xmx512m
    depends_on:
      milvus:
        condition: service_healthy
    volumes:
      - ./scaffold-ai-assistant:/knowledge:ro
    networks:
      - network_dev
    healthcheck:
      test: ["CMD", "wget", "-q", "--spider", "http://localhost:18084/actuator/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

  # ==================== Milvus 向量数据库（standalone 模式） ====================
  # Milvus 依赖 etcd（元数据）和 minio（对象存储），三者必须一起部署
  etcd:
    container_name: milvus-etcd
    image: quay.io/coreos/etcd:v3.5.5
    environment:
      - ETCD_AUTO_COMPACTION_MODE=revision
      - ETCD_AUTO_COMPACTION_RETENTION=1000
      - ETCD_QUOTA_BACKEND_BYTES=4294967296
      - ETCD_SNAPSHOT_COUNT=50000
    volumes:
      - ../data/scaffold-ai-assistant-data/milvus/etcd:/etcd
    command: etcd -advertise-client-urls=http://127.0.0.1:2379 -listen-client-urls http://0.0.0.0:2379 --data-dir /etcd
    networks:
      - network_dev
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

  minio:
    container_name: milvus-minio
    image: minio/minio:RELEASE.2023-03-20T20-16-18Z
    environment:
      MINIO_ACCESS_KEY: minioadmin
      MINIO_SECRET_KEY: minioadmin
    volumes:
      - ../data/scaffold-ai-assistant-data/milvus/minio:/minio_data
    command: minio server /minio_data
    networks:
      - network_dev
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:9000/minio/health/live"]
      interval: 30s
      timeout: 20s
      retries: 3
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

  milvus:
    container_name: milvus-standalone
    image: milvusdb/milvus:v2.4.0
    command: ["milvus", "run", "standalone"]
    environment:
      ETCD_ENDPOINTS: etcd:2379
      MINIO_ADDRESS: minio:9000
    ports:
      - "19530:19530"
      - "9091:9091"
    volumes:
      - ../data/scaffold-ai-assistant-data/milvus/milvus:/var/lib/milvus
    depends_on:
      - etcd
      - minio
    networks:
      - network_dev
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:9091/healthz"]
      interval: 30s
      start_period: 90s
      timeout: 20s
      retries: 3
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

networks:
  network_dev:
    external: true
```

## 9.4 Nacos配置

### 9.4.1 网关路由配置（需在 zmbdp-gateway-service-dev.yaml 中添加）

> **重要**：网关路由路径前缀必须与现有脚手架规则一致：`/portal/**` 路由到 portal-service，`/admin/**` 路由到 admin-service。**不要**使用 `/chat/**` 作为路由前缀，因为 AI 的 C 端接口属于 portal-service，应走 `/portal/chat/**`。

**chat-service 不直接暴露给前端**——所有前端请求经 Gateway 路由到 portal-service 或 admin-service，再通过 Feign/WebClient 内部调用 chat-service。因此**不需要为 chat-service 配置网关路由**。

**Feign 调用 chat-service 的方式**（遵循脚手架现有规范，无需在 Nacos yml 中新增任何 Feign 配置）：

1. chat-service 的 Feign 接口定义在 `zmbdp-chat-api` 模块中，使用 `@FeignClient` 注解，通过 `name` 属性指定服务名（靠 Nacos 服务发现自动路由），例如：

```java
@FeignClient(contextId = "chatApi", name = "zmbdp-chat-service", path = "/chat")
public interface ChatApi {
    // ...
}
```

2. portal-service 和 admin-service 在启动类上通过 `@EnableFeignClients(clients = {...})` 显式声明需要引入的 chat-service Feign 接口，例如：

```java
@EnableFeignClients(clients = {
        ChatApi.class,
        HistoryApi.class,
        // ... 其他需要调用的 chat-service Feign 接口
})
```

> **说明**：脚手架现有的 Feign 调用均采用此模式（参考 portal-service 调用 admin-service 的 `AppUserApi`），不使用 `spring.cloud.openfeign.client.config` 这种 yml 配置方式。

**Gateway 路由保持现有配置不变**（参考 `deploy/dev/res/sql/DEFAULT_GROUP/zmbdp-gateway-service-dev.yaml`）：

```yaml
spring:
  cloud:
    gateway:
      routes:
        # 现有路由配置（保持不变）
        - id: zmbdp-portal-service
          uri: lb://zmbdp-portal-service
          predicates:
            - Path=/portal/**
          filters:
            - StripPrefix=1
        - id: zmbdp-admin-service
          uri: lb://zmbdp-admin-service
          predicates:
            - Path=/admin/**
          filters:
            - StripPrefix=1
        # ... 其他现有路由
```

> **路径映射示例**：
> - 前端请求 `http://ip:10030/portal/chat/completions/stream` → Gateway 匹配 `/portal/**` + StripPrefix=1 → portal-service 收到 `/chat/completions/stream`
> - 前端请求 `http://ip:10030/admin/knowledge/sources` → Gateway 匹配 `/admin/**` + StripPrefix=1 → admin-service 收到 `/knowledge/sources`

### 9.4.3 bootstrap.yml配置（应用启动时加载）

> **关键说明**：
> 1. Nacos namespace 使用**字符串名称** `scaffold-ai-assistant-${RUN_ENV}`（如 `scaffold-ai-assistant-dev`），**不是 UUID**
> 2. 共享配置统一命名规则：`share-xxx-${env}.yaml`（**单数 share + 带环境后缀**）
> 3. `share-token-${env}.yaml` 包含 JWT 密钥 `${jwt.token.secret}`，**必须引入**，否则 AuthFilter 启动失败
> 4. Nacos 用户名密码复用脚手架现有配置

```yaml
spring:
  output:
    ansi:
      enabled: ALWAYS
  application:
    name: zmbdp-chat-service
  profiles:
    active: ${RUN_ENV}
  cloud:
    nacos:
      discovery:
        username: nacos
        password: Hf@173503494
        namespace: scaffold-ai-assistant-${RUN_ENV}
        server-addr: ${NACOS_ADDR}
      config:
        username: nacos
        password: Hf@173503494
        namespace: scaffold-ai-assistant-${RUN_ENV}
        server-addr: ${NACOS_ADDR}
        file-extension: yaml
        # 配置共享的配置（与 zmbdp-portal-service / zmbdp-admin-service 保持一致命名规则）
        shared-configs:
          # Redis 配置（必需）
          - data-id: share-redis-${spring.profiles.active}.${spring.cloud.nacos.config.file-extension}
            refresh: true
          # Caffeine 本地缓存配置（可选，如需要本地缓存）
          - data-id: share-caffeine-${spring.profiles.active}.${spring.cloud.nacos.config.file-extension}
            refresh: true
          # MySQL 配置（必需）
          - data-id: share-mysql-${spring.profiles.active}.${spring.cloud.nacos.config.file-extension}
            refresh: true
          # Token 配置（必需，含 jwt.token.secret）
          - data-id: share-token-${spring.profiles.active}.${spring.cloud.nacos.config.file-extension}
            refresh: true
          # 限流配置（必需，@RateLimit 注解依赖）
          - data-id: share-ratelimit-${spring.profiles.active}.${spring.cloud.nacos.config.file-extension}
            refresh: true
          # 日志配置（必需）
          - data-id: share-log-${spring.profiles.active}.${spring.cloud.nacos.config.file-extension}
            refresh: true
          # 监控配置（必需，Prometheus + Grafana）
          - data-id: share-monitor-${spring.profiles.active}.${spring.cloud.nacos.config.file-extension}
            refresh: true
          # xxl-job 配置（必需，KnowledgeSyncScheduler 定时任务依赖）
          - data-id: share-xxljob-${spring.profiles.active}.${spring.cloud.nacos.config.file-extension}
            refresh: true

# MyBatis Plus 配置（与现有服务保持一致）
mybatis-plus:
  configuration:
    log-impl: org.apache.ibatis.logging.stdout.StdOutImpl
```

> **环境变量说明**：
> - `RUN_ENV`：运行环境（dev/test/prd），与现有服务一致
> - `NACOS_ADDR`：Nacos 地址，开发环境为 `127.0.0.1:8848`，容器环境为 `scaffold-ai-assistant-nacos:8848`

### 9.9.2 portal-service WebClient 配置

> **详细配置**：详见 [03-C端功能设计.md](03-C端功能设计.md) 3.0.1 节 WebClientConfig。

portal-service 调用 chat-service SSE 端点需配置 `@LoadBalanced` WebClient：

```java
@Configuration
public class WebClientConfig {

    @Bean
    @LoadBalanced  // 关键：启用服务发现，通过服务名解析实际地址
    public WebClient.Builder chatWebClientBuilder() {
        return WebClient.builder()
            .defaultHeader(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE)
            .codecs(configurer -> configurer.defaultCodecs().maxInMemorySize(16 * 1024 * 1024));  // 16MB
    }
}
```

> **关键说明**：
> - 使用 `@LoadBalanced` + `lb://zmbdp-chat-service` 协议，通过 Nacos 服务发现解析实际地址，**不硬编码 IP/端口**
> - portal-service 的 `bootstrap.yml` 需引入 `share-token-${env}.yaml`（JWT 密钥）和 `share-redis-${env}.yaml`（对话记忆）
> - portal-service 的 Nacos 配置文件命名：`zmbdp-portal-service-${env}.yaml`

### 9.9.3 AI 可观测性说明（不使用 LangSmith）

> **重要说明**：本项目**不使用 LangSmith**，AI 可观测性通过以下现有方案实现：

| 可观测性需求 | 实现方案 | 说明 |
|------------|---------|------|
| **分布式链路追踪** | SkyWalking（复用脚手架） | 追踪请求从 Gateway → portal-service → chat-service 的完整链路 |
| **应用指标监控** | Prometheus + Grafana（复用脚手架） | 监控 QPS、响应时间、错误率等 |
| **AI 调用指标** | Spring AI Micrometer + `sys_ai_operation_log` 表 | Spring AI 内置 Micrometer 指标 + 自定义操作日志表 |
| **日志收集** | Logback + ELK（可选） | 应用日志通过 Logback 输出，可选 ELK 集中收集 |

**不使用 LangSmith 的原因**：
1. **技术栈不匹配**：LangSmith 是 LangChain 的 observability 平台，本项目使用 Spring AI Alibaba（非 LangChain）
2. **额外成本**：LangSmith 为 SaaS 服务，需额外订阅和外部网络调用
3. **已有方案覆盖**：SkyWalking + Prometheus + Grafana 已满足分布式追踪和监控需求
4. **AI 专用指标**：通过 `sys_ai_operation_log` 表记录 AI 调用日志（token 消耗、模型、耗时等），配合 `IStatisticsService` 实现统计

> **注意**：04-B端功能设计.md 和 07-项目架构设计.md 中原提到的 LangSmith 集成已移除，改为上述方案。

### 9.9.4 sys_argument 表 AI 配置键说明

> **说明**：AI 运行时参数通过 `sys_argument` 表管理，B端修改后即时生效，无需重启服务。Nacos 中 `spring.ai.dashscope.chat.options` 及 `scaffold.rag` 等配置仅作为启动默认值，运行时以本表为准。

| config_key | 说明 | 默认值 | 与 Nacos 配置的关系 |
|------------|------|--------|-------------------|
| `ai.temperature` | 模型采样温度，控制生成随机性 | 0.7 | 启动时取 Nacos `spring.ai.dashscope.chat.options.temperature`，运行时以本表为准 |
| `ai.max_tokens` | 单次生成最大 token 数 | 4096 | 启动时取 Nacos `spring.ai.dashscope.chat.options.max-tokens`，运行时以本表为准 |
| `ai.top_k` | RAG 检索默认返回数量 | 5 | 启动时取 Nacos `scaffold.rag.top-k`，运行时以本表为准 |
| `ai.enable_rag` | 是否启用 RAG 检索增强 | true | Nacos 无对应项，仅本表管理 |
| `ai.enable_tools` | 是否启用 Agent 工具调用 | true | Nacos 无对应项，仅本表管理 |
| `ai.tool.{toolName}.enabled` | 单个工具的启停控制（如 `ai.tool.code_search.enabled`） | true | Nacos 无对应项，仅本表管理 |
