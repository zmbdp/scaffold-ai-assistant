# 📊 用户级 AI 调用统计接口设计

> **文档说明**：本文档描述"按单个用户查看大模型调度 / 使用量"能力的接口设计，作为对现有 7 个统计接口（[08-API接口设计.md](./08-API接口设计.md) 8.6.19 ~ 8.6.36）的补充。
>
> **适用范围**：B 端管理后台（admin-service 经 Feign 调用 chat-service），用于管理员查看指定用户的 AI 调用聚合指标、调用记录明细，定位异常用户、核算 Token 成本。
>
> **不引入新组件**：基于现有 `sys_ai_operation_log` 表 + 现有 Feign / Service / Mapper 分层 + 现有 Redis 缓存机制实现，无需引入 Prometheus / Grafana / LangSmith / Langfuse 等外部观测组件。

---

## 11.1 背景与问题

### 11.1.1 现状

当前 [StatisticsApi](../../zmbdp-chat/zmbdp-chat-api/src/main/java/com/zmbdp/chat/api/statistics/feign/StatisticsApi.java) 提供 7 个统计接口，但**没有一个能回答"某个具体用户的 AI 调用 / Token 使用情况"**：

| 接口 | 数据维度 | 局限 |
| ---- | ------ | --- |
| `GET /statistics/users` | 用户聚合 | 只有 `conversationCount`（对话次数），**没有 AI 调用次数 / Token / 模型** |
| `GET /statistics/ai-metrics` | 全局聚合 | 只有全局 `totalAiCalls / totalTokenUsage`，**没有按用户拆分** |
| `GET /statistics/ai-metrics/{operationId}` | 单条详情 | 按 `operationId` 查单条，**不能按 userId 列表分页查** |
| `GET /operation-log/list` | 全局分页 | [OperationLogApi](../../zmbdp-chat/zmbdp-chat-api/src/main/java/com/zmbdp/chat/api/operationlog/feign/OperationLogApi.java) 当前**不支持 `userId` 筛选参数** |

### 11.1.2 数据基础现状

`sys_ai_operation_log` 表已建立，字段已定义完整（含 `user_id / model / total_tokens / prompt_tokens / completion_tokens / response_time / status / operation_type`），但**原有代码未在 AI 调用处埋点写入**，导致表长期为空、7 个统计接口的 AI 指标全部返回 0。

**本设计文档配套的埋点补全方案详见 11.12 节**，分三阶段渐进实施：
- 第一阶段（已实施）：基础埋点，写入用户/Prompt/Response/状态等字段，Token 与 ToolCalls 暂留 null
- 第二阶段（已实施）：改用 `stream().chatResponse()` 捕获 Usage，填充 promptTokens/completionTokens/totalTokens
- 第三阶段（规划中）：实现 ToolCallback 装饰器捕获工具调用链路

### 11.1.3 为什么不用 Prometheus / LangSmith

| 方案 | 适用场景 | 不适用原因 |
| ---- | ------- | ---------- |
| Prometheus + Grafana | 系统级时序监控（QPS / 延迟 / 错误率） | label 基数爆炸：user_id 是高基数维度，Prometheus 官方明确警告会内存爆掉；且 Prometheus 不适合存原始 Prompt/Response 文本 |
| LangSmith | LangChain（Python/JS）LLM 可观测 | 官方只提供 Python / JS SDK，**无 Java SDK**；Java 生态等价方案为 Langfuse（需自部署新服务） |
| Langfuse | 开源 LLM 可观测平台 | 需新部署一个服务，与现有 `sys_ai_operation_log` 表能力重叠 |

**结论**：现有 `sys_ai_operation_log` 表已经是一个简化版的 LangSmith（记录 Prompt / Response / 工具调用链路 / Token / 延迟 / 状态），只需补 2 个查询接口即可满足业务需求，**零新组件**。

## 11.2 设计目标

1. **按用户聚合**：查看指定用户的 AI 调用总数、成功率、Token 消耗、平均延迟、Top 模型
2. **按用户分页**：查看指定用户的 AI 调用记录列表（不含 Prompt/Response，避免响应过大）
3. **复用现有 VO**：聚合接口复用 `AiMetricsVO`，列表接口复用 `OperationLogVO` + `BasePageVO`
4. **复用现有分层**：Controller → Service → Mapper → XML，与现有 7 个统计接口保持一致
5. **复用现有缓存**：Redis TTL=300 秒，Key 带 `userId` 避免互相覆盖
6. **兼容现有接口**：增强 `OperationLogApi.getLogs()` 加 `userId` 可选参数，不破坏既有调用

## 11.4 接口详细设计

### 11.4.1 GET /admin/statistics/users/{userId}/ai-metrics（用户 AI 调用聚合）

**功能**：查看指定用户的 AI 调用聚合指标，复用 `AiMetricsVO`。用于管理员核算单用户 Token 成本、定位异常用户（调用失败率高 / Token 消耗异常）。

**请求头**：`Authorization: Bearer {token}`

**路径参数**：

| 参数 | 类型 | 必填 | 说明 |
| ---- | ---- | ---- | ---- |
| `userId` | Long | 是 | 用户ID（关联 sys_ai_operation_log.user_id） |

**响应体**：

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": {
    "totalAiCalls": 320,
    "successRate": 96.88,
    "avgTokenUsage": 1480,
    "totalTokenUsage": 473600,
    "avgLatency": 2850,
    "topModels": [
      {"model": "deepseek-v4-flash", "count": 280},
      {"model": "text-embedding-v1", "count": 40}
    ]
  }
}
```

**字段说明**：复用 [AiMetricsVO](../../zmbdp-chat/zmbdp-chat-api/src/main/java/com/zmbdp/chat/api/statistics/domain/vo/AiMetricsVO.java)，字段含义与 8.6.20 节一致，差别仅在于数据范围限定为 `WHERE user_id = #{userId}`。

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| `totalAiCalls` | Long | 该用户 AI 总调用次数 |
| `successRate` | Double | 成功率（%，保留两位小数） |
| `avgTokenUsage` | Long | 平均 Token 消耗（仅统计 status=SUCCESS） |
| `totalTokenUsage` | Long | 总 Token 消耗（仅统计 status=SUCCESS） |
| `avgLatency` | Long | 平均延迟（毫秒，仅统计 status=SUCCESS） |
| `topModels` | List | Top 模型调用统计（按调用次数降序，默认 Top 10） |
| `topModels[].model` | String | 模型名称 |
| `topModels[].count` | Long | 该用户调用该模型的次数 |

**调用链路**：

```
admin-service StatisticsController.getUserAiMetrics()
  → Feign StatisticsApi.getUserAiMetrics(userId)
  → chat-service StatisticsController.getUserAiMetrics(userId)
  → IStatisticsService.getUserAiCallMetrics(userId)
  → SysAiOperationLogMapper.countTotalAiCallsByUser(userId)
                         .countSuccessAiCallsByUser(userId)
                         .avgTokenUsageByUser(userId)
                         .sumTotalTokensByUser(userId)
                         .avgLatencyByUser(userId)
                         .selectTopModelsByUser(userId, limit)
  → 表 sys_ai_operation_log (WHERE user_id = #{userId})
```

**缓存策略**：Redis 缓存 Key `stats:user_ai_metrics:{userId}`，TTL=300 秒（与现有统计接口一致）。

### 11.4.2 GET /admin/statistics/users/{userId}/operations（用户 AI 调用记录分页）

**功能**：查看指定用户的 AI 调用记录分页列表，列表页仅返回摘要字段（不含 Prompt/Response/ToolCalls，避免响应过大），完整链路详情通过 8.6.35 节接口 `GET /statistics/ai-metrics/{operationId}` 查看。

**请求头**：`Authorization: Bearer {token}`

**路径参数**：

| 参数 | 类型 | 必填 | 说明 |
| ---- | ---- | ---- | ---- |
| `userId` | Long | 是 | 用户ID |

**查询参数**：

| 参数 | 类型 | 必填 | 说明 |
| ---- | ---- | ---- | ---- |
| `pageNo` | Integer | 否 | 页码，默认 1 |
| `pageSize` | Integer | 否 | 每页条数，默认 20 |
| `operationType` | String | 否 | AI 操作类型筛选（CHAT/RETRIEVE/EMBEDDING/RERANK） |
| `model` | String | 否 | 模型名称筛选（精确匹配） |
| `status` | String | 否 | 调用状态筛选（SUCCESS/FAILED/TIMEOUT） |
| `startDate` | Long | 否 | 起始日期（格式：20260712） |
| `endDate` | Long | 否 | 结束日期（格式：20260712） |

**响应体**：

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": {
    "totals": 320,
    "totalPages": 16,
    "list": [
      {
        "id": 100001,
        "userId": 1001,
        "userFrom": "app",
        "conversationId": 1234567890,
        "operationType": "CHAT",
        "model": "deepseek-v4-flash",
        "promptTokens": 1200,
        "completionTokens": 350,
        "totalTokens": 1550,
        "responseTime": 2300,
        "status": "SUCCESS",
        "errorMsg": null,
        "createDate": 20260714,
        "createTime": "2026-07-14 15:30:00"
      }
    ]
  }
}
```

**字段说明**：复用 [OperationLogVO](../../zmbdp-chat/zmbdp-chat-api/src/main/java/com/zmbdp/chat/api/operationlog/domain/vo/OperationLogVO.java) 和 [BasePageVO](../../zmbdp-common/zmbdp-common-domain/src/main/java/com/zmbdp/common/domain/domain/vo/BasePageVO.java)。**列表页省略 `prompt / response / toolCalls` 三个大字段**（Mapper 查询时 SELECT 显式列，不查这三列），前端如需查看完整链路，再调用 8.6.35 节接口。

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| `totals` | Integer | 总记录数 |
| `totalPages` | Integer | 总页数 |
| `list` | List | 调用记录列表 |
| `list[].id` | Long | 日志ID |
| `list[].userId` | Long | 用户ID |
| `list[].userFrom` | String | 用户来源（sys/app） |
| `list[].conversationId` | Long | 关联对话ID |
| `list[].operationType` | String | AI 操作类型（CHAT/RETRIEVE/EMBEDDING/RERANK） |
| `list[].model` | String | 模型名称 |
| `list[].promptTokens` | Integer | Prompt Token 数 |
| `list[].completionTokens` | Integer | 响应 Token 数 |
| `list[].totalTokens` | Integer | 总 Token 数 |
| `list[].responseTime` | Integer | 响应耗时（毫秒） |
| `list[].status` | String | 调用状态（SUCCESS/FAILED/TIMEOUT） |
| `list[].errorMsg` | String | 失败原因 |
| `list[].createDate` | Long | 创建日期（20260712 格式） |
| `list[].createTime` | LocalDateTime | 创建时间 |

> **注意**：列表项**不含** `prompt / response / toolCalls` 字段，这三列由 8.6.35 节单次调用详情接口返回。

**调用链路**：

```
admin-service StatisticsController.getUserOperations()
  → Feign StatisticsApi.getUserOperations(userId, pageNo, pageSize, operationType, model, status, startDate, endDate)
  → chat-service StatisticsController.getUserOperations(...)
  → IStatisticsService.getUserOperations(userId, pageNo, pageSize, operationType, model, status, startDate, endDate)
  → SysAiOperationLogMapper.selectUserOperationPage(...)
  → 表 sys_ai_operation_log (WHERE user_id = #{userId} [AND 其他条件])
```

**缓存策略**：**不缓存**（与现有 8.6.35 节单次调用详情接口一致，分页列表实时查询，避免管理员看到过期数据）。

### 11.6.2 Service 实现要点

```java
// StatisticsServiceImpl.java 新增方法

private static final String CACHE_KEY_USER_AI_METRICS_PREFIX = "stats:user_ai_metrics:";

@Override
public AiMetricsVO getUserAiCallMetrics(Long userId) {
    // 1. 尝试读取缓存（Key 带 userId 避免互相覆盖）
    String cacheKey = CACHE_KEY_USER_AI_METRICS_PREFIX + userId;
    AiMetricsVO cached = redisService.getCacheObject(cacheKey,
            new TypeReference<AiMetricsVO>() {});
    if (cached != null) {
        return cached;
    }
    // 2. 数据库聚合查询（全部 WHERE user_id = #{userId}）
    AiMetricsVO vo = new AiMetricsVO();
    vo.setTotalAiCalls(sysAiOperationLogMapper.countTotalAiCallsByUser(userId));
    Long successCalls = sysAiOperationLogMapper.countSuccessAiCallsByUser(userId);
    vo.setAvgTokenUsage(sysAiOperationLogMapper.avgTokenUsageByUser(userId));
    vo.setTotalTokenUsage(sysAiOperationLogMapper.sumTotalTokensByUser(userId));
    vo.setAvgLatency(sysAiOperationLogMapper.avgLatencyByUser(userId));
    // 成功率
    if (vo.getTotalAiCalls() != null && vo.getTotalAiCalls() > 0) {
        double rate = successCalls * 100.0 / vo.getTotalAiCalls();
        vo.setSuccessRate(Math.round(rate * 100.0) / 100.0);
    } else {
        vo.setSuccessRate(0.0);
    }
    // Top 模型
    List<AiMetricsVO.TopModel> topModels = sysAiOperationLogMapper.selectTopModelsByUser(userId, TOP_MODELS_LIMIT);
    vo.setTopModels(topModels != null ? topModels : new ArrayList<>());
    // 3. 写入缓存
    redisService.setCacheObject(cacheKey, vo, STATS_CACHE_TTL, TimeUnit.SECONDS);
    return vo;
}

@Override
public BasePageVO<OperationLogVO> getUserOperations(Long userId, Integer pageNo, Integer pageSize,
                                                     String operationType, String model,
                                                     String status, Long startDate, Long endDate) {
    // 参数兜底
    if (pageNo == null || pageNo < 1) pageNo = 1;
    if (pageSize == null || pageSize < 1) pageSize = 20;
    // 1. 查询总数
    Long total = sysAiOperationLogMapper.countUserOperations(userId, operationType, model, status, startDate, endDate);
    BasePageVO<OperationLogVO> pageVO = new BasePageVO<>();
    pageVO.setTotals(total != null ? total.intValue() : 0);
    pageVO.setTotalPages(pageVO.getTotals() % pageSize == 0
            ? pageVO.getTotals() / pageSize
            : pageVO.getTotals() / pageSize + 1);
    if (total == null || total == 0) {
        pageVO.setList(new ArrayList<>());
        return pageVO;
    }
    // 2. 查询分页数据（列表页不查 prompt/response/tool_calls 三个大字段）
    int offset = (pageNo - 1) * pageSize;
    List<SysAiOperationLog> records = sysAiOperationLogMapper.selectUserOperationPage(
            userId, operationType, model, status, startDate, endDate, offset, pageSize);
    // 3. 转换 VO（列表页不填充 prompt/response/toolCalls）
    List<OperationLogVO> list = BeanCopyUtil.copyListProperties(records, OperationLogVO.class);
    // 显式清空三个大字段（防止 BaseMapper 全字段查询时被填充）
    list.forEach(vo -> {
        vo.setPrompt(null);
        vo.setResponse(null);
        vo.setToolCalls(null);
    });
    pageVO.setList(list);
    return pageVO;
}
```

### 11.6.3 Mapper 接口新增方法

```java
// SysAiOperationLogMapper.java 新增方法

 * 统计指定用户的 AI 总调用次数

/**
 * 统计指定用户的平均 Token 消耗（仅 status=SUCCESS）
```

## 11.7 缓存策略

| 接口 | 是否缓存 | 缓存 Key | TTL | 说明 |
| ---- | -------- | -------- | --- | ---- |
| `GET /statistics/users/{userId}/ai-metrics` | ✅ 缓存 | `stats:user_ai_metrics:{userId}` | 300 秒 | 与现有统计接口一致，Key 带 userId 避免互相覆盖 |
| `GET /statistics/users/{userId}/operations` | ❌ 不缓存 | - | - | 分页列表实时查询，避免管理员看到过期数据 |

**Key 设计说明**：用户级聚合 Key 必须带 `userId` 后缀，否则不同用户的聚合结果会互相覆盖（这是一个常见的缓存设计陷阱）。

**缓存失效**：TTL=300 秒自然过期，不做主动失效（统计数据更新频率低，无需主动失效；如需立即失效，可手动 `redisService.deleteObject(cacheKey)`）。

## 11.8 性能与索引建议

### 11.8.1 索引建议

`sys_ai_operation_log` 表建议补充以下索引（如尚未建立）：

```sql
-- 用户维度的查询索引（覆盖用户聚合 + 用户分页 + 用户筛选）
ALTER TABLE sys_ai_operation_log
    ADD INDEX idx_user_id (user_id);

-- 用户 + 创建日期的复合索引（覆盖用户分页 + 日期筛选）
ALTER TABLE sys_ai_operation_log
    ADD INDEX idx_user_id_create_date (user_id, create_date);
```

> **注意**：如 `user_id` 字段已有单列索引，则不必重复创建。复合索引 `idx_user_id_create_date` 适用于"查某个用户近 N 天调用记录"这种高频查询场景。

### 11.8.2 分页查询性能

- **小页码**（pageNo ≤ 100）：`LIMIT offset, pageSize` 性能良好
- **大页码**（pageNo > 100）：`LIMIT offset, pageSize` 会有深翻页性能问题（需扫描 offset + pageSize 行）
- **当前设计**：管理后台翻页场景，默认 pageSize=20，深翻页概率低，暂不优化
- **后续优化**：如出现深翻页需求，可改为基于 `id < #{lastId}` 的游标分页（要求按 id 倒序，且前端不显示总页数）

### 11.8.3 数据量预估

- 单次 AI 调用产生 1 条 `sys_ai_operation_log` 记录
- 假设 1000 活跃用户，人均每日 20 次调用 → 单日 2 万条，单年 730 万条
- **建议**：表数据保留 90 天，超过 90 天的记录归档到冷库（`sys_ai_operation_log_archive`）或直接清理，避免单表数据量过大影响查询性能

### 11.12.4 第二阶段：Token 精确捕获（已实施）

#### 11.12.4.1 方案选型（实事求是纠偏）

本节最初规划用「Spring AI Observation + ObservationHandler」方案，调研后改用更简洁的方案，对比如下：

| 方案 | 改动量 | 新依赖 | 新配置 | 风险 | 采用 |
| ---- | ------ | ------ | ------ | ---- | ---- |
| **A. 改用 `stream().chatResponse()`** | 约 20 行 | 无 | 无 | 低 | ✅ 采用 |
| **B. Spring AI Observation + ObservationHandler** | 约 50 行 + 1 个新类 | `spring-boot-starter-actuator` | `spring.ai.chat.observations.enabled=true` | 中（流式场景 Observation 跨线程传递 Usage 有坑） | ❌ 弃用 |

**弃用方案 B 的理由**：方案 B 本质上也是从 `ChatResponse.metadata.usage` 取 Token，只是多了一层 Observation 包装。Observation 在流式场景下存在跨线程传递问题（doOnComplete 在 reactor 线程执行，与 Observation.stop() 触发的 handler 不在同一线程，需要 Reactor Context 桥接）。方案 A 直接在 Flux 链路里捕获 Usage 到 `AtomicReference`，绕开线程切换问题。

#### 11.12.4.2 技术原理

**Spring AI `ChatClient.stream()` 提供两种返回类型**：

| API | 返回类型 | 是否含 Usage |
| --- | -------- | ----------- |
| `.stream().content()` | `Flux<String>` | ❌ 只返回字符串内容，丢弃 metadata |
| `.stream().chatResponse()` | `Flux<ChatResponse>` | ✅ 每个 ChatResponse 都带 metadata，**最后一帧含完整 Usage** |

**DashScope 流式响应行为**（基于 OpenAI 兼容规范）：
- 流式响应由多个 SSE chunk 组成
- 每个 chunk 是一个 JSON，含 `choices[].delta.content` 字段（增量内容）
- **最后一个 chunk 额外含 `usage` 字段**，包含 `prompt_tokens / completion_tokens / total_tokens`（累计值，非增量）
- Spring AI Alibaba 的 `DashScopeChatModel` 会把 usage 解析到 `ChatResponse.metadata.usage` 中

#### 11.12.4.3 streamChat 改造方案（文本对话）

**改动文件**：[ChatServiceImpl.java](../../zmbdp-chat/zmbdp-chat-service/src/main/java/com/zmbdp/chat/service/service/impl/ChatServiceImpl.java) 的 `streamChat()` 方法

**改造点**：

1. **改用 `.chatResponse()` 替代 `.content()`**：拿到 `Flux<ChatResponse>` 而非 `Flux<String>`
2. **新增 `AtomicReference<Usage>` 持有最后一个非 null 的 Usage**：每个 ChatResponse 都检查 `metadata.usage`，非 null 就覆盖（最后一帧的 Usage 是累计值）
3. **在 `doOnNext` 中既累积内容又捕获 Usage**：从 `chatResponse.getResult().getOutput().getText()` 取内容片段
4. **`doOnComplete` 时把 `AtomicReference.get()` 传给 `saveConversation`**

**为什么用 `AtomicReference` 而非 `volatile` 或 `final`**：
- `final` 变量在 lambda 中不可修改
- `volatile` 不能在 lambda 中赋值（lambda 闭包要求 effectively final）
- `AtomicReference` 是线程安全的，可在 reactor 多线程切换中安全使用

#### 11.12.4.4 streamChatWithImage 改造方案（图文对话）

**改动文件**：[ChatServiceImpl.java](../../zmbdp-chat/zmbdp-chat-service/src/main/java/com/zmbdp/chat/service/service/impl/ChatServiceImpl.java) 的 `streamChatWithImage()` 方法

**特殊性**：图文对话**不走 Spring AI**（绕过 DashScopeChatModel 多模态路由 bug），直接用 WebClient 调 DashScope OpenAI 兼容端点，所以拿不到 `ChatResponse`，需要自己从原始 SSE JSON 解析 `usage` 字段。

**改造点**：

1. **新增私有方法 `extractUsage(String event)`**：从 SSE 事件 JSON 中提取 `usage.prompt_tokens / completion_tokens / total_tokens`
2. **新增 `AtomicReference<long[]>` 持有 usage**：长度 3 的数组分别对应 prompt/completion/total
3. **在 `.doOnEach(signal -> ...)` 的 `isOnNext` 分支调用 `extractUsage(event)`**：每帧都尝试解析，覆盖前一次的值（最后一帧才真正含 usage）
4. **`doOnComplete` 时把数组传给 `saveImageConversation`**

**DashScope OpenAI 兼容模式流式响应最后一帧格式**：
```json
{
  "id":"...",
  "choices":[{"index":0,"delta":{"content":"","role":"assistant"},"finish_reason":"stop"}],
  "usage":{"prompt_tokens":50,"completion_tokens":100,"total_tokens":150}
}
```

**关键参数：`stream_options.include_usage=true`**

DashScope OpenAI 兼容协议**默认不在流式响应中返回 usage**，必须显式在请求体中设置 `stream_options: {"include_usage": true}`，否则最后一帧不含 `usage` 字段，`extractUsage` 永远解析不到 Token。

请求体示例：
```json
{
  "model": "qwen-vl-max",
  "stream": true,
  "stream_options": {"include_usage": true},
  "messages": [...]
}
```

**为什么文本对话不需要这个参数？** 文本对话走 Spring AI 的 `ChatClient.stream().chatResponse()`，Spring AI Alibaba 的 `DashScopeChatModel` 内部用的是 DashScope 原生 API（非 OpenAI 兼容模式），原生 API 默认返回 usage，不需要额外参数。

#### 11.12.4.5 recordOperationLog 改造方案

**改动点**：方法签名增加 3 个 Token 参数

```java
private void recordOperationLog(SysAiConversation conversation, String prompt, String response,
                                 String modelName, long responseTime, String status,
                                 String errorMsg, String operationType,
                                 Integer promptTokens, Integer completionTokens, Integer totalTokens)
```

**填入字段**：
- `opLog.setPromptTokens(promptTokens)`
- `opLog.setCompletionTokens(completionTokens)`
- `opLog.setTotalTokens(totalTokens)`

**类型说明**：实体类 `SysAiOperationLog` 三个 Token 字段均为 `Integer`（Token 数远未超过 `Integer.MAX_VALUE ≈ 21 亿`），而 Spring AI `Usage.getPromptTokens()` 返回 `long`，需要 `Math.toIntExact()` 显式转换；WebClient 自解析的 JSON 字段也是 long，同样转换。

**null 兼容**：参数传 null 时，MyBatis-Plus `BaseMapper.insert()` 默认 NOT_NULL 策略会跳过该字段，数据库字段保持 NULL，与第一阶段行为一致。

#### 11.12.4.6 兼容性与降级策略

| 场景 | 行为 |
| ---- | ---- |
| DashScope 正常返回 usage | 三个 Token 字段精确填充 |
| DashScope 异常未返回 usage（如 SSE 提前断开） | 三个 Token 字段为 null，不影响其他字段 |
| 调用失败进入 `onErrorResume` | Token 字段为 null（失败时本来就没有 Usage） |
| 流式响应中 usage 在中间帧（非最后一帧） | `AtomicReference` 会持续覆盖，最终保留最后一帧的值 |

### 11.12.5 第三阶段：ToolCalls 拦截（规划中）

**难点**：Spring AI 工具调用通过 `@Tool` 注解 + `ToolCallbackProvider` 自动扫描，没有统一拦截点。
