# C 端前端接口文档 - 总说明

> 本文档为 C 端（用户端）前端开发的公共约定，所有 C 端功能模块的分文档均遵循此约定。
>
> **适用对象**：C 端用户（JWT 中 `userFrom=app`）使用的全部接口，包括用户登录注册、对话、历史、反馈、文件上传。
>
> **本文档所有路径、字段名、类型、校验规则、枚举值、错误码均来自后端源码确认，可直接作为前端开发对接依据。**

---

## 一、网关地址

```
http://localhost:10030
```

所有接口都经过网关（zmbdp-gateway-service）统一入口，前端只需请求「网关地址 + 接口路径」。

---

## 二、路由前缀与路径映射

C 端接口统一前缀 `/portal`。网关配置 `Path=/portal/**` + `StripPrefix=1`，即网关剥离第一段 `/portal` 后转发到 portal-service（AI 相关接口由 portal-service 内部再通过 WebClient/Feign 调用 chat-service）。

| 前端请求 | 网关转发目标 | 说明 |
| --- | --- | --- |
| `/portal/user/**` | portal-service `/user/**` | 用户登录注册、信息、退出 |
| `/portal/chat/**` | portal-service `/chat/**` → WebClient → chat-service | 流式对话 |
| `/portal/history/**` | portal-service `/history/**` → Feign → chat-service | 对话历史 |
| `/portal/feedback/**` | portal-service `/feedback/**` → Feign → chat-service | 回答反馈 |
| `/file/**` | file-service `/file/**` | 通用文件上传（**不剥离前缀**，无 `/portal` 前缀） |

> **注意**：
> - 前端始终请求 `/portal/xxx`（文件接口除外，请求 `/file/xxx`），无需关心后端 Feign/WebClient 转发细节。
> - 文件接口 `/file/**` 路由到 file-service 时**不做 StripPrefix**，前端直接请求 `/file/upload`、`/file/sign`。

## 三、JWT 认证机制

### 3.1 请求头格式（所有需认证的接口必填）

```
Authorization: Bearer {accessToken}
Content-Type: application/json
```

> **注意**：`Bearer` 后必须有一个空格，再接 token 值。token 来自登录接口返回的 `accessToken` 字段。

### 3.2 C 端用户身份

C 端用户的 JWT 中 `userFrom=app`。网关 `AuthFilter` 会阻止 C 端 token（`userFrom=app`）访问 `/admin/**` 路径（B 端管理接口），返回 `401004 令牌验证失败！`。

**测试 C 端接口时务必使用 C 端登录（`/portal/user/login/**`）获取的 token，切勿复用 B 端 `/admin/sys_user/**` 产出的 token。**

### 3.3 网关白名单（以下路径不需要 token）

来自网关配置 `security.ignore.whites`：

- `/**/register/**`
- `/admin/codeLogin`
- `/**/login/**`
- `/**/send_code/**`
- `/**/nologin/**`
- `/**/test/**`

C 端接口的匹配情况：

| 接口 | 匹配规则 | 是否需要 token |
| --- | --- | --- |
| `/portal/user/login/wechat` | `/**/login/**` | 否 |
| `/portal/user/send_code` | `/**/send_code/**` | 否 |
| `/portal/user/login/code` | `/**/login/**` | 否 |
| `/portal/user/edit` | 无 | 是 |
| `/portal/user/login_info/get` | 无 | 是 |
| `/portal/user/logout` | 无 | 是 |
| `/portal/chat/**` | 无 | 是 |
| `/portal/history/**` | 无 | 是 |
| `/portal/feedback/**` | 无 | 是 |
| `/file/**` | 无 | 是 |

### 3.4 网关校验流程

不在白名单的请求，网关 `AuthFilter` 依次校验：

1. token 是否存在（缺失返回 `401000 令牌不能为空`，HTTP 401）
2. token 是否有效（无效返回 `401001 令牌已过期或验证不正确！`，HTTP 401）
3. Redis 登录态是否存在（过期返回 `401003 登录状态已过期！`，HTTP 401）
4. `/admin/**` 路径校验 `userFrom=sys`（不匹配返回 `401004 令牌验证失败！`，HTTP 401）

> **重要**：网关鉴权失败时，HTTP 状态码为 401/403/500，响应体为 `Result` JSON 结构（含 `code`/`errMsg`）。前端应同时处理 HTTP 状态码和业务 code。

## 四、统一响应结构 `Result<T>`

除 SSE 流式对话接口外，所有接口返回统一 JSON 结构：

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": {}
}
```

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `code` | int | 响应码，成功固定 `200000` |
| `errMsg` | String | 消息描述，成功为 `"操作成功"` |
| `data` | T | 业务数据，无数据时为 `null` |

> **前端判断成功**：`code === 200000`。不要仅用 HTTP 状态码判断（业务错误统一返回 HTTP 200 + 非 200000 的 code；网关鉴权错误返回 HTTP 401/403 + Result JSON）。

## 五、分页响应结构 `BasePageVO<T>`

分页接口的 `data` 字段结构：

```json
{
  "totals": 100,
  "totalPages": 5,
  "list": []
}
```

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `totals` | Integer | 总记录数 |
| `totalPages` | Integer | 总页数 |
| `list` | List&lt;T&gt; | 数据列表 |

### 分页请求参数约定

C 端分页接口统一使用 `pageNo` / `pageSize` 两个查询参数：

| 参数 | 类型 | 默认值 | 含义 |
| --- | --- | --- | --- |
| `pageNo` | Integer | `1` | 页码，从 1 开始 |
| `pageSize` | Integer | 接口而定 | 每页大小 |

## 七、常用错误码

错误码定义来自 `com.zmbdp.common.domain.domain.ResultCode`。

### 7.1 通用错误码

| code | errMsg | 含义 | 前端处理建议 |
| --- | --- | --- | --- |
| `200000` | 操作成功 | 成功 | — |
| `400000` | 无效的参数 | 参数校验失败 / 账号格式错误 / token 在 Service 层解析失败 | 检查入参 |
| `400001` | 无效的验证码 | 验证码不存在或已过期（缓存未命中） | 提示重新发送验证码 |
| `400002` | 错误的验证码 | 验证码输入错误 | 提示重新输入 |
| `400020` | 请求过于频繁，请稍后重试 | 触发限流（`@RateLimit`） | 提示稍后重试 |
| `401000` | 令牌不能为空 | 缺 token（HTTP 401） | 跳转登录 |
| `401001` | 令牌已过期或验证不正确！ | token 无效（HTTP 401） | 跳转登录 |
| `401003` | 登录状态已过期！ | Redis 登录态过期（HTTP 401） | 跳转登录 |
| `401004` | 令牌验证失败！ | 越权访问 / token 校验失败（HTTP 401） | 跳转登录 |
| `404000` | 服务未找到 | 目标服务不可用（HTTP 404） | 提示服务异常 |
| `404001` | url 未找到 | 接口路径不存在（HTTP 404） | 检查请求路径 |
| `500000` | 服务繁忙请稍后重试 | 服务异常（HTTP 500） | 提示服务异常 |
| `500001` | 操作失败 | 业务操作失败 | 提示失败原因 |

### 7.2 C 端业务相关错误码

| code | errMsg | 含义 | 常见场景 |
| --- | --- | --- | --- |
| `500002` | 短信发送失败 | 发送验证码失败 / 单账号每日发送次数超限 | `send_code` 每日上限（默认 50 次/账号） |
| `500003` | 获取直传地址失败 | 生成 OSS 直传签名失败 | 文件签名接口 |
| `500004` | 上传 oss 异常，请稍后重试 | 文件上传 OSS 失败 | 文件上传接口 |
| `500023` | AI 服务连接失败 | portal-service 调用 chat-service SSE 端点连接失败 | 对话 SSE 错误帧 |
| `500024` | AI 服务响应超时 | portal-service 调用 chat-service 超时（默认 60 秒） | 对话 SSE 错误帧 |
| `500031` | 点踩原因不能为空 | `feedbackType=DISLIKE` 时未填 `dislikeReason` | 反馈提交 |
| `500032` | 会话不属于当前用户 | `sessionId` 归属校验失败 | 对话传非己 sessionId |

> **注意**：`ResultCode` 中 `400004 超过每日发送次数限制` 枚举已定义，但 `send_code` 实际超限时抛出的是 `500002 短信发送失败`（见 `CaptchaService.sendCode`）。前端以实际返回为准。

## 九、限流与幂等说明

### 9.1 限流（`@RateLimit`）

基于 AOP + Redis 实现。C 端涉及限流的接口：

| 接口 | 限流配置 | 说明 |
| --- | --- | --- |
| `POST /portal/chat/completions/stream` | `@RateLimit`（无显式参数） | 使用 Nacos 全局配置 `ratelimit.default-limit`（默认 60）/ `ratelimit.default-window-sec`（默认 60 秒），维度默认 `IP` |
| `POST /portal/chat/image/completions/stream` | `@RateLimit`（无显式参数） | 同上 |
| `GET /portal/user/send_code` | `@RateLimit(limit=1, windowSec=60, dimensions=BOTH, message="操作过于频繁，请稍后重试")` | 每 IP + 每账号 60 秒内仅 1 次；未登录时退化为 IP 限流 |

- 触发限流返回 `400020 请求过于频繁，请稍后重试`（`send_code` 的 message 为「操作过于频繁，请稍后重试」）。
- 对话接口限流触发时，会同步返回 `Result` JSON（**非 SSE 帧**），前端需同时处理 SSE 流和 JSON 错误响应。

### 9.2 幂等（`@Idempotent`）

基于 Redis 实现。C 端涉及幂等的接口：

| 接口 | 幂等配置 | 说明 |
| --- | --- | --- |
| `POST /portal/feedback` | `@Idempotent(message = "请勿重复提交反馈")` | 请求头 `Idempotent-Token`，过期时间默认 300 秒（5 分钟） |

- 同一 `Idempotent-Token` 在 300 秒内只允许一次请求，重复提交返回「请勿重复提交反馈」。
- 切换反馈类型（如点赞 → 点踩）时**必须生成新的 `Idempotent-Token`**，否则会被拦截。

功能三：对话历史管理。提供会话列表分页查询、会话消息详情查询、删除会话三个接口，用于 C 端历史会话列表页、会话详情回看、清空某条会话等场景。

功能五：通用文件上传。提供服务端代理上传（multipart）与 OSS 直传签名获取两个接口，B 端 / C 端共用，用于 C 端图文对话前上传图片获取 URL、头像上传等场景。

## 十二、前端集成注意事项

### 12.1 Token 使用

- 登录返回 `accessToken` 后，所有需鉴权接口在请求头携带 `Authorization: Bearer {accessToken}`
- `Bearer` 后必须有一个空格
- token 过期或登录态失效时（`401xxx` 错误码）跳转登录页
- 退出登录后旧 token 即便未过期也会失效（网关校验 Redis 登录态是否存在）

### 12.2 路径前缀

- 所有 C 端业务接口统一加 `/portal` 前缀
- 文件接口 `/file/**` 无 `/portal` 前缀（通用接口，B/C 端共用）

### 12.3 SSE 对话处理

- 两个对话接口返回 `text/event-stream`，前端需用 `EventSource` 或 `fetch` + `ReadableStream` 处理
- 每帧是 JSON 字符串，需 `JSON.parse` 后按 `done` 字段判断帧类型（详见对话模块文档）
- **首次对话**：不传 `sessionId`，从结束帧获取后端生成的 `sessionId` 并保存，后续多轮对话回传同一 `sessionId`
- 限流 / token 失效时返回的是 `Result` JSON（非 SSE 帧），前端需根据响应 Content-Type 区分处理

### 12.4 反馈幂等

- 提交反馈必须每次生成新的 UUID 作为 `Idempotent-Token` 请求头
- 切换点赞 ↔ 点踩时尤其要换新 token，否则会被「请勿重复提交反馈」拦截
- 反馈是覆盖语义，同一用户对同一对话重复提交不会报「已反馈」错误（后端先删后插）

### 12.5 Long 类型精度

- `userId`、`conversationId`、`feedback.id` 等字段为 Java `Long` 类型（雪花算法 ID，可达 19 位），JSON 序列化为数字。JavaScript Number 精度上限为 2^53，超出会丢精度。前端如需精确处理这些 ID，建议后端已按数字返回时，前端以字符串方式存储与比较，避免做数值运算。

### 12.6 限流

- 对话接口限流阈值由 Nacos 全局配置控制（默认 60 次/60 秒，IP 维度），超限返回 `400020`
- `send_code` 每 IP + 每账号 60 秒内仅 1 次，超限返回 `400020`（message「操作过于频繁，请稍后重试」）
- `send_code` 单账号每日发送上限默认 50 次，超限返回 `500002 短信发送失败`
