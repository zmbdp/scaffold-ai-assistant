# 用户统计

## 功能介绍

用户统计接口用于在 B 端管理后台展示用户活跃情况，包括活跃用户数、总用户数以及 Top 活跃用户列表（按对话次数降序）。

数据来源：`sys_ai_conversation` 表的用户维度聚合。**`days` 过滤参数已生效**，用于控制活跃用户数统计的时间窗口。

接口走 Redis 缓存（TTL=300 秒），**缓存 key 包含 `days` 参数**（`stats:users:{days}`），不同 `days` 值的统计结果互相隔离，不会覆盖。

---

## 接口详情

### 获取用户统计

| 项 | 值 |
| --- | --- |
| 路径 | `GET /admin/statistics/users` |
| 认证 | 必须携带 B 端 JWT（`userFrom=sys`），见总说明第三章 |
| Content-Type | `application/json` |

#### 调用链路

前端请求 `/admin/statistics/users` → 网关 StripPrefix=1 → admin-service `StatisticsController#getUserStatistics(days)` → Feign 调用 chat-service `StatisticsApi#getUserStatistics(days)` → `StatisticsServiceImpl#getUserStats(days)`。

---

### 请求参数

| 参数名 | 类型 | 必传 | 默认值 | 校验规则 | 含义 |
| --- | --- | --- | --- | --- | --- |
| `days` | Integer | 否 | `7` | Service 层兜底：为 `null` 或 `<= 0` 时取默认值 `7` | 活跃用户统计窗口（近 N 天，含当天） |

> **过滤参数生效说明**：`days` 仅影响 `activeUsers` 字段的统计窗口，不影响 `totalUsers`（始终统计全量）和 `topUsers`（始终按对话次数降序取前 10）。
>
> **窗口语义**：`days=7` 表示统计近 7 天（含当天）有对话记录的去重用户数。SQL 实现为 `create_date >= DATE_FORMAT(DATE_SUB(NOW(), INTERVAL #{days} - 1 DAY), '%Y%m%d')`，由数据库计算日期阈值。

---

### 请求示例

**示例 1：使用默认 7 天窗口（不传参）**

```
GET /admin/statistics/users
Authorization: Bearer {accessToken}
```

**示例 2：指定 30 天窗口**

```
GET /admin/statistics/users?days=30
Authorization: Bearer {accessToken}
```

---

### 响应结构

返回 `Result<UserStatisticsVO>`，`data` 为用户统计聚合 VO。

**UserStatisticsVO 字段**：

| 字段 | 类型 | 含义 | 数据来源 / 计算口径 |
| --- | --- | --- | --- |
| `activeUsers` | Long | 活跃用户数（近 `days` 天，按 `user_id` 去重） | `COUNT(DISTINCT user_id)` WHERE `is_deleted=0 AND status='SUCCESS' AND create_date >= 近N天起始日` |
| `totalUsers` | Long | 总用户数（全量，按 `user_id` 去重） | `COUNT(DISTINCT user_id)` WHERE `is_deleted=0`（不限 status） |
| `topUsers` | List&lt;TopUser&gt; | Top 活跃用户列表（按对话次数降序，固定取前 10） | 联表 `sys_user`/`app_user` 补全 `username` |

**TopUser 字段**：

| 字段 | 类型 | 含义 | 数据来源 |
| --- | --- | --- | --- |
| `userId` | Long | 用户ID | `sys_ai_conversation.user_id` |
| `username` | String | 用户名 | `COALESCE(sys_user.nick_name, app_user.nick_name)`，LEFT JOIN 用户表 |
| `conversationCount` | Integer | 对话次数 | `COUNT(*)` 分组统计 |
| `lastActiveTime` | Long | 最后活跃时间（毫秒时间戳） | `UNIX_TIMESTAMP(MAX(create_time)) * 1000` |

> **username 联表逻辑**：`sys_ai_conversation.user_from='sys'` 时 JOIN `sys_user` 表取 `nick_name`；`user_from='app'` 时 JOIN `app_user` 表取 `nick_name`。使用 `COALESCE` 取非空值，使用 LEFT JOIN 避免用户记录缺失导致行丢失。
>
> **topUsers 固定 10 条**：`topUsers` 固定返回前 10 名（Service 常量 `TOP_USERS_LIMIT=10`），不受 `days` 参数影响，统计范围为全量有效对话。

> **空数据场景**：无任何对话记录时，`activeUsers=0`、`totalUsers=0`、`topUsers` 返回空数组 `[]`。

### 错误码说明

| code | errMsg | 含义 | 场景 |
| --- | --- | --- | --- |
| `200000` | 操作成功 | 成功 | — |
| `401000` | 令牌不能为空 | 缺 token（HTTP 401） | 未携带 Authorization |
| `401003` | 登录状态已过期！ | Redis 登录态过期（HTTP 401） | token 过期 |
| `401004` | 令牌验证失败！ | 越权访问（HTTP 401） | C 端 token 访问 `/admin/**` |
| `404000` | 服务未找到 | chat-service 不可用（HTTP 404） | chat-service 未注册到 Nacos |
| `500000` | 服务繁忙请稍后重试 | 服务异常（HTTP 500） | Feign 调用异常或数据库异常 |

> 当 chat-service Feign 调用返回非 `200000` 时，admin-service 抛出 `ServiceException("获取用户统计失败")`，HTTP 200 + `code=500001`。

## 前端注意事项

1. **days 过滤参数已生效**：`days` 参数控制 `activeUsers` 的统计窗口。传 `30` 表示统计近 30 天活跃用户；不传或传非正数默认近 7 天。

2. **缓存 key 含 days 参数**：缓存 key 为 `stats:users:{days}`（如 `stats:users:7`、`stats:users:30`），不同 `days` 值的统计结果独立缓存、互不覆盖，切换窗口可立即获取对应窗口的缓存或重新统计。

3. **统计口径差异**：
   - `activeUsers`：仅统计 `status='SUCCESS'` 且 `is_deleted=0` 的对话对应的去重用户。
   - `totalUsers`：统计 `is_deleted=0`（不限 `status`）的去重用户，含失败对话的发起者。
   - `topUsers`：仅统计 `status='SUCCESS'` 且 `is_deleted=0` 的对话，按对话次数降序取前 10，统计范围为全量（不受 `days` 影响）。

4. **topUsers 不受 days 影响**：`topUsers` 固定基于全量有效对话统计，不随 `days` 参数变化。如需按时间窗口筛选 Top 用户，当前接口不支持。

5. **username 联表**：`username` 取自 `sys_user.nick_name`（B 端用户）或 `app_user.nick_name`（C 端用户），由 `user_from` 字段决定联表对象。用户记录被删除时 `username` 为 `null`。

6. **lastActiveTime 单位**：毫秒级时间戳（Long），由 SQL `UNIX_TIMESTAMP(MAX(create_time)) * 1000` 转换。

7. **无审计/限流注解**：本接口为只读查询，未标注 `@LogAction` 和 `@RateLimit`。
