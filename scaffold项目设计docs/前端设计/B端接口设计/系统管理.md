# 系统管理

## 功能介绍

系统管理接口用于在 B 端管理后台检测系统核心组件的连通性，包括 MySQL、Redis、Nacos、Milvus、LLM（DashScope API）共 5 个组件。

**降级策略**：单个组件检测失败不影响其他组件，每个组件独立计时和异常捕获。`overallStatus` 根据正常组件数量返回 `UP`（全部正常）/ `PARTIAL`（部分异常）/ `DOWN`（全部异常）。

**无缓存**：本接口为实时检测，每次请求都会真实探测各组件连通性，不走 Redis 缓存。建议前端控制调用频率（如每 30 秒刷新一次），避免频繁探测增加系统负担。

---

## 接口详情

### 获取系统健康状态

| 项 | 值 |
| --- | --- |
| 路径 | `GET /admin/system/health` |
| 认证 | 必须携带 B 端 JWT（`userFrom=sys`），见总说明第三章 |
| Content-Type | `application/json` |
| 请求参数 | 无 |

#### 调用链路

前端请求 `/admin/system/health` → 网关 StripPrefix=1 → admin-service `SystemController#health()` → Feign 调用 chat-service `SystemApi#health()` → `SystemHealthServiceImpl#checkHealth()`。

---

### 检测组件与探测方式

| 组件名称 | 探测方式 | 成功 details 内容 |
| --- | --- | --- |
| `MySQL` | 通过 `DataSource.getConnection()` 获取连接并执行 `SELECT 1` | `连接正常` |
| `Redis` | 通过 `StringRedisTemplate.getConnectionFactory().getConnection().ping()` 执行 PING | `连接正常` |
| `Nacos` | 通过 `DiscoveryClient.getServices()` 获取服务列表 | `连接正常，已注册服务数：{N}` |
| `Milvus` | 通过 `MilvusServiceClient.hasCollection()` 检查集合是否存在 | `连接正常，集合 {collectionName} 已存在` 或 `连接正常，集合 {collectionName} 不存在` |
| `LLM` | 通过 `ChatClient` 发送极简 prompt（`"1"`）验证 DashScope API | `DashScope API 可达`（响应长度 ≤ 20 时追加响应内容） |

> **异常 details 内容**：组件检测失败时，`details` 为 `连接失败：{异常消息}`（如 `连接失败：Connection refused`）。
>
> **LLM 组件特殊性**：LLM 检测会真实调用一次 DashScope API，会消耗少量 Token。检测耗时通常较长（数百毫秒到数秒），`latency` 字段会反映这一延迟。

---

### 响应结构

返回 `Result<SystemHealthVO>`，`data` 为系统健康状态 VO。

**SystemHealthVO 字段**：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `overallStatus` | String | 整体状态（`UP`/`PARTIAL`/`DOWN`），见下方枚举 |
| `components` | List&lt;Component&gt; | 各组件状态列表（固定 5 个，顺序为 MySQL→Redis→Nacos→Milvus→LLM） |
| `checkTime` | String | 检测时间（格式 `yyyy-MM-dd HH:mm:ss`） |

**Component 字段**：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `name` | String | 组件名称（`MySQL`/`Redis`/`Nacos`/`Milvus`/`LLM`） |
| `status` | String | 组件状态（`UP`/`DOWN`），见下方枚举 |
| `latency` | Long | 检测延迟（毫秒） |
| `details` | String | 详细信息（异常时含错误原因） |

> **组件顺序**：`components` 列表固定顺序为 MySQL → Redis → Nacos → Milvus → LLM（按 Service 层检测顺序依次添加）。

### 枚举值

#### overallStatus（整体状态）

| 值 | 含义 | 判定规则 |
| --- | --- | --- |
| `UP` | 全部正常 | 5 个组件全部 `status=UP` |
| `PARTIAL` | 部分异常 | 1~4 个组件 `status=UP`（即至少 1 个 UP 且至少 1 个 DOWN） |
| `DOWN` | 全部异常 | 5 个组件全部 `status=DOWN`（0 个 UP） |

#### status（组件状态）

| 值 | 含义 |
| --- | --- |
| `UP` | 组件连通正常 |
| `DOWN` | 组件连通异常 |

> **注意**：组件状态只有 `UP`/`DOWN` 两种，没有 `PARTIAL`。`PARTIAL` 仅用于 `overallStatus` 整体状态。

---

### 错误码说明

| code | errMsg | 含义 | 场景 |
| --- | --- | --- | --- |
| `200000` | 操作成功 | 成功 | 即使组件异常也返回成功（组件异常体现在 `status=DOWN`） |
| `401000` | 令牌不能为空 | 缺 token（HTTP 401） | 未携带 Authorization |
| `401003` | 登录状态已过期！ | Redis 登录态过期（HTTP 401） | token 过期 |
| `401004` | 令牌验证失败！ | 越权访问（HTTP 401） | C 端 token 访问 `/admin/**` |
| `404000` | 服务未找到 | chat-service 不可用（HTTP 404） | chat-service 未注册到 Nacos |
| `500000` | 服务繁忙请稍后重试 | 服务异常（HTTP 500） | Feign 调用异常 |

> **重要**：本接口在组件异常时**不会返回错误码**。即使所有组件都 DOWN，接口仍返回 `code=200000`，组件异常信息体现在 `components[].status=DOWN` 和 `components[].details` 字段中。前端判断系统健康状态应以 `overallStatus` 和 `components[].status` 为准，而非 `code`。
>
> 当 chat-service Feign 调用返回非 `200000` 时，admin-service 抛出 `ServiceException("获取系统健康状态失败")`，HTTP 200 + `code=500001`。这通常意味着 chat-service 自身不可用（如服务未启动），此时无法检测任何组件。

---

## 前端注意事项

1. **无缓存，实时检测**：本接口不走 Redis 缓存，每次请求都会真实探测 5 个组件。建议前端控制调用频率（如每 30 秒或手动刷新），避免频繁探测增加系统负担。LLM 组件检测会真实调用一次 DashScope API，消耗少量 Token。

2. **组件异常不返回错误码**：组件检测失败不影响接口本身的成功返回。即使所有组件都 DOWN，`code` 仍为 `200000`。前端应以 `overallStatus` 和 `components[].status` 判断系统状态。

3. **降级策略**：单个组件检测失败（如 Milvus 超时）不影响其他组件的检测结果。每个组件独立 try-catch，异常仅记录为 `status=DOWN` + `details=连接失败：{异常消息}`，不抛出。

4. **LLM 组件延迟较高**：LLM 检测需真实调用 DashScope API，`latency` 通常在数百毫秒到数秒之间，远高于其他组件（MySQL/Redis 通常 < 50ms）。前端展示时可考虑对 LLM 组件单独标注「检测耗时较长」。

5. **组件顺序固定**：`components` 列表顺序固定为 MySQL → Redis → Nacos → Milvus → LLM，前端可按此顺序渲染。

6. **Milvus 集合检测**：Milvus 检测通过 `hasCollection` 验证连通性，`details` 会返回集合是否存在。集合不存在不代表 Milvus 异常（`status=UP`），仅说明尚未创建知识库向量集合（需先触发知识同步）。

7. **checkTime 格式**：`checkTime` 为字符串格式 `yyyy-MM-dd HH:mm:ss`，由 `LocalDateTime.now().format(...)` 生成，为 chat-service 服务器时间。

8. **latency 单位**：毫秒（ms），由 `System.currentTimeMillis()` 差值计算，包含网络往返和组件响应时间。

9. **无审计/限流注解**：本接口为只读检测，未标注 `@LogAction` 和 `@RateLimit`。
