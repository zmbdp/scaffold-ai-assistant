# 字典数据管理

> 模块：字典管理 / 字典数据（`sys_dictionary_data` 表）
>
> 适用对象：B 端管理系统（`userFrom=sys` 的系统用户）
>
> 公共约定（网关地址、JWT 认证、统一响应结构 `Result<T>`、分页结构 `BasePageVO<T>`、错误码、枚举值等）见 `总说明.md`。

---

## 功能介绍

字典数据是归属某个字典类型下的具体枚举项。例如字典类型「用户身份」（`typeKey=user_identity`）下包含 `super_admin`（超级管理员）、`platform_admin`（平台管理员）两条字典数据。

本模块提供字典数据的**新增、分页列表查询、编辑**三个接口。

> **重要**：
> - 当前版本**没有字典数据删除接口**，也没有字典数据启用/停用（status 切换）接口。`status` 字段不在新增/编辑入参 DTO 中，无法通过接口修改（仅列表查询返回）。
> - 字典数据的新增/编辑/查询均与所属字典类型 `typeKey` 关联。新增前需确保对应 `typeKey` 的字典类型已存在。
>
> 本模块接口均不涉及操作审计（源码未标注 `@LogAction`）。

## 一、新增字典数据

### 1.1 接口信息

| 项目 | 内容 |
| --- | --- |
| 路径 | `POST /admin/dictionary/addData` |
| 认证 | 需要 token（`Authorization: Bearer {accessToken}`），且 `userFrom=sys` |
| 权限 | 所有已登录 B 端用户（未做 `identity` 区分） |
| Content-Type | `application/json` |
| 操作审计 | 无 |

### 1.2 请求参数

请求体为 `DictionaryDataAddReqDTO`：

| 字段 | 类型 | 必传 | 校验规则 | 默认值 | 含义 |
| --- | --- | --- | --- | --- | --- |
| `typeKey` | String | 是 | `@NotBlank(message = "字典类型业务主键不能为空")`；服务端校验对应字典类型存在 | — | 所属字典类型业务主键 |
| `dataKey` | String | 是 | `@NotBlank(message = "字典数据业务主键不能为空")`；服务端校验**全局唯一** | — | 字典数据业务主键（如 `super_admin`），**新增后不可修改** |
| `value` | String | 是 | `@NotBlank(message = "字典数据名称不能为空")`；服务端校验**全局唯一** | — | 字典数据名称（如「超级管理员」） |
| `sort` | Integer | 否 | — | `null` | 排序值（列表按 `sort` 升序、`id` 升序排列） |
| `remark` | String | 否 | — | — | 备注 |

> **重要（前端必读）**：
> - 服务端唯一性校验：`dataKey` 或 `value` 任一已存在即拒绝。该校验是**全局范围**（跨所有 `typeKey`），并非仅限当前 `typeKey` 下。因此不同字典类型之间也不能有重复的 `dataKey` 或 `value`。
> - 新增前服务端会先校验 `typeKey` 对应的字典类型存在，不存在则抛「字典类型不存在」。
> - `status` 字段不在入参中，新增时不显式设置（依赖数据库默认值）。

### 1.3 请求示例

```http
POST /admin/dictionary/addData
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.....
Content-Type: application/json

{
  "typeKey": "user_identity",
  "dataKey": "super_admin",
  "value": "超级管理员",
  "sort": 1,
  "remark": "可管理用户账号"
}
```

### 1.4 响应结构

`Result<Long>`，`data` 为新增记录的数据库自增 ID。

### 1.5 响应示例

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": 11
}
```

### 1.6 错误码说明

| code | errMsg | 触发场景 |
| --- | --- | --- |
| `400000`/`500001` | 操作失败 | 字典类型不存在（`typeKey` 查不到对应字典类型）；字典数据键或值已存在（`dataKey` 或 `value` 全局重复）；`@NotBlank` 校验失败 |
| `401000`/`401001`/`401003`/`401004` | token 相关 | 未携带/无效/过期/越权 token（HTTP 401） |

> 注：服务端业务异常由 `ServiceException` 抛出，前端应以 `code !== 200000` 判断失败并展示 `errMsg`。

## 二、查询字典数据列表

### 2.1 接口信息

| 项目 | 内容 |
| --- | --- |
| 路径 | `GET /admin/dictionary/listData` |
| 认证 | 需要 token（`Authorization: Bearer {accessToken}`），且 `userFrom=sys` |
| 权限 | 所有已登录 B 端用户 |
| Content-Type | 无请求体（参数通过 query string 传递） |
| 操作审计 | 无 |

### 2.2 请求参数

query 参数为 `DictionaryDataListReqDTO`（继承 `BasePageReqDTO`，含分页字段）：

| 字段 | 类型 | 必传 | 校验规则 | 默认值 | 含义 | 查询匹配方式 |
| --- | --- | --- | --- | --- | --- | --- |
| `pageNo` | Integer | 否 | — | `1` | 页码，从 1 开始 | — |
| `pageSize` | Integer | 否 | — | `10` | 每页大小 | — |
| `typeKey` | String | 是 | `@NotBlank(message = "字典类型业务主键不能为空")` | — | 字典类型业务主键 | 精确匹配（`type_key =`） |
| `value` | String | 否 | — | `null` | 字典数据名称 | 模糊匹配（`value like concat('%', ?, '%')`） |

> **重要**：
> - `typeKey` 为**必传**，必须指定要查询的字典类型。不传会触发 `@NotBlank` 校验失败。
> - 查询结果按 `sort` 升序、`id` 升序排列（排序值小的在前，相同排序值按 id 升序）。
> - `value` 为模糊匹配，适合在指定字典类型下搜索具体枚举项。

### 2.3 请求示例

查询「用户身份」字典类型下的所有数据：

```http
GET /admin/dictionary/listData?pageNo=1&pageSize=20&typeKey=user_identity
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.....
```

按名称模糊查询：

```http
GET /admin/dictionary/listData?pageNo=1&pageSize=10&typeKey=user_identity&value=管理
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.....
```

### 2.4 响应结构

`Result<BasePageVO<DictionaryDataVO>>`，外层 `BasePageVO` 字段：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `totals` | Integer | 总记录数 |
| `totalPages` | Integer | 总页数 |
| `list` | List&lt;DictionaryDataVO&gt; | 数据列表 |

列表元素 `DictionaryDataVO` 字段：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `id` | Long | 字典数据自增主键 |
| `typeKey` | String | 所属字典类型业务主键 |
| `dataKey` | String | 字典数据业务主键 |
| `value` | String | 字典数据名称 |
| `remark` | String | 备注 |
| `sort` | Integer | 排序值 |
| `status` | Integer | 状态：`1` 正常 / `0` 停用 |

### 2.5 响应示例

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": {
    "totals": 2,
    "totalPages": 1,
    "list": [
      {
        "id": 11,
        "typeKey": "user_identity",
        "dataKey": "super_admin",
        "value": "超级管理员",
        "remark": "可管理用户账号",
        "sort": 1,
        "status": 1
      },
      {
        "id": 12,
        "typeKey": "user_identity",
        "dataKey": "platform_admin",
        "value": "平台管理员",
        "remark": "不可管理用户账号",
        "sort": 2,
        "status": 1
      }
    ]
  }
}
```

无匹配数据时：

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": {
    "totals": 0,
    "totalPages": 0,
    "list": []
  }
}
```

### 2.6 错误码说明

| code | errMsg | 触发场景 |
| --- | --- | --- |
| `400000` | 无效的参数 | `typeKey` 为空（`@NotBlank` 校验失败） |
| `400006` | 参数类型不匹配 | `pageNo`/`pageSize`/`sort` 传非数字 |
| `401000`/`401001`/`401003`/`401004` | token 相关 | 未携带/无效/过期/越权 token（HTTP 401） |

> 该接口为纯查询，不主动抛业务错误。无匹配数据时 `totals=0`、`list=[]`。

## 三、编辑字典数据

### 3.1 接口信息

| 项目 | 内容 |
| --- | --- |
| 路径 | `POST /admin/dictionary/editData` |
| 认证 | 需要 token（`Authorization: Bearer {accessToken}`），且 `userFrom=sys` |
| 权限 | 所有已登录 B 端用户 |
| Content-Type | `application/json` |
| 操作审计 | 无 |

### 3.2 请求参数

请求体为 `DictionaryDataEditReqDTO`：

| 字段 | 类型 | 必传 | 校验规则 | 默认值 | 含义 |
| --- | --- | --- | --- | --- | --- |
| `dataKey` | String | 是 | `@NotBlank(message = "字典数据业务主键不能为空")`；作为定位主键 | — | 字典数据业务主键（**不可修改**，仅用于定位） |
| `value` | String | 是 | `@NotBlank(message = "字典数据名称不能为空")`；服务端校验除自身外全局唯一 | — | 字典数据名称（可修改） |
| `sort` | Integer | 否 | — | `null` | 排序值（仅当传入非 `null` 时才更新） |
| `remark` | String | 否 | — | — | 备注（仅当传入非空值时才更新） |

> 服务端逻辑：
> - 根据 `dataKey` 查询记录，不存在则抛「字典数据不存在」。
> - 校验 `value` 唯一性（排除当前 `dataKey` 自身，全局范围），若其他记录已使用该 `value` 则抛「字典数据值已存在」。
> - `dataKey` 不会被修改（编辑仅更新 `value`/`sort`/`remark`）。
> - `typeKey` 不在编辑 DTO 中，**无法通过编辑接口修改字典数据所属的字典类型**。
> - `sort` 仅当传入非 `null` 时才覆盖原值。
> - `remark` 仅当传入非空（`StringUtil.isNotBlank`）时才覆盖原值。
> - `status` 不在入参中，编辑不会修改状态。

### 3.3 请求示例

```http
POST /admin/dictionary/editData
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.....
Content-Type: application/json

{
  "dataKey": "super_admin",
  "value": "超级管理员",
  "sort": 1,
  "remark": "更新备注：拥有全部权限"
}
```

### 3.4 响应结构

`Result<Long>`，`data` 为编辑记录的数据库 ID。

### 3.5 响应示例

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": 11
}
```

### 3.6 错误码说明

| code | errMsg | 触发场景 |
| --- | --- | --- |
| `400000`/`500001` | 操作失败 | 字典数据不存在（`dataKey` 查不到）；字典数据值已存在（`value` 与其他记录重复）；`@NotBlank` 校验失败 |
| `401000`/`401001`/`401003`/`401004` | token 相关 | 未携带/无效/过期/越权 token（HTTP 401） |

> 注：「字典数据不存在」「字典数据值已存在」由 `ServiceException` 抛出，前端应以 `code !== 200000` 判断失败并展示 `errMsg`。

## 四、枚举值

### 4.1 字典数据状态（status）

| 值 | 含义 |
| --- | --- |
| `1` | 正常 |
| `0` | 停用 |

> **重要**：字典 `status` 为 **Integer 类型**（`1`/`0`），与 B 端用户状态（String `enable`/`disable`）不同，请勿混淆。当前接口不提供 `status` 修改能力，状态仅可通过列表查询读取。

### 4.2 排序（sort）

`sort` 为 Integer 类型，数值小的排在前面。列表查询固定按 `sort ASC, id ASC` 排序。前端维护字典数据顺序时应通过编辑接口调整 `sort` 值。

---

## 五、前端注意事项

1. **无删除接口**：字典数据当前无删除接口，新增后只能编辑 `value`/`sort`/`remark`，无法删除。前端「字典数据」列表不应提供删除按钮。
2. **dataKey 不可修改**：编辑时 `dataKey` 仅用于定位记录，服务端不会更新它。前端编辑表单中 `dataKey` 字段应设为只读/禁用。
3. **typeKey 不可修改**：编辑 DTO 不含 `typeKey`，无法通过编辑接口变更字典数据所属的字典类型。如需移动到其他类型，当前不支持。
4. **全局唯一性**：`dataKey` 和 `value` 的唯一性校验是**全局范围**（跨所有字典类型），不仅是同一 `typeKey` 下。前端新增/编辑时若提示「键或值已存在」，需检查是否与其他字典类型下的数据冲突。
5. **listData 必传 typeKey**：`GET /admin/dictionary/listData` 的 `typeKey` 为必传，必须先选定字典类型再查询其下数据。前端通常在字典类型列表中点击某类型后，再用该 `typeKey` 查询其下字典数据。
6. **排序控制**：列表按 `sort` 升序、`id` 升序排列。前端如需调整顺序，通过编辑接口修改 `sort` 值。
7. **sort/remark 空值处理**：
   - 编辑时 `sort` 仅当传入非 `null` 才更新；不传则保留原值。
   - `remark` 仅当传入非空字符串才更新；传空串不会清空原备注。如需清空需后端调整。
8. **status 无法修改**：`status` 不在新增/编辑入参中，无法通过接口切换字典数据启用/停用。前端不应提供状态切换控件。
9. **列表为 GET 请求**：`GET /admin/dictionary/listData` 参数走 query string，注意 `value` 模糊查询时的 URL 编码（中文需 `encodeURIComponent`）。
10. **下拉选项应用**：前端各类下拉选项（如 B 端用户身份、状态）应通过 `listData` 接口按 `typeKey` 获取，以保证枚举值与后端字典数据一致。
11. **错误码判断**：服务端业务异常可能以 `500001` 返回，前端应统一以 `code !== 200000` 判断失败并展示 `errMsg`。
