# 字典类型管理

> 模块：字典管理 / 字典类型（`sys_dictionary_type` 表）
>
> 适用对象：B 端管理系统（`userFrom=sys` 的系统用户）
>
> 公共约定（网关地址、JWT 认证、统一响应结构 `Result<T>`、分页结构 `BasePageVO<T>`、错误码、枚举值等）见 `总说明.md`。

---

## 功能介绍

字典采用「字典类型 + 字典数据」两级结构：

- **字典类型**（`sys_dictionary_type`）：定义一组同类字典的容器，由 `typeKey`（业务主键）和 `value`（类型名称）标识，如「用户身份」「用户状态」等。
- **字典数据**（`sys_dictionary_data`）：归属某个字典类型下的具体枚举项，详见 `字典数据管理.md`。

本模块提供字典类型的**新增、分页列表查询、编辑**三个接口。

> **重要**：当前版本**没有字典类型删除接口**，也没有字典类型启用/停用（status 切换）接口。字典类型的 `status` 字段在新增/编辑接口的入参 DTO 中不存在，无法通过接口修改（仅列表查询返回）。
>
> 本模块接口均不涉及操作审计（源码未标注 `@LogAction`）。

## 一、新增字典类型

### 1.1 接口信息

| 项目 | 内容 |
| --- | --- |
| 路径 | `POST /admin/dictionary/addType` |
| 认证 | 需要 token（`Authorization: Bearer {accessToken}`），且 `userFrom=sys` |
| 权限 | 所有已登录 B 端用户（未做 `identity` 区分，`super_admin` / `platform_admin` 均可） |
| Content-Type | `application/json` |
| 操作审计 | 无 |

### 1.2 请求参数

请求体为 `DictionaryTypeWriteReqDTO`：

| 字段 | 类型 | 必传 | 校验规则 | 默认值 | 含义 |
| --- | --- | --- | --- | --- | --- |
| `typeKey` | String | 是 | `@NotBlank(message = "字典类型键不能为空")`；服务端校验全局唯一 | — | 字典类型业务主键（如 `user_identity`），**新增后不可修改** |
| `value` | String | 是 | `@NotBlank(message = "字典类型值不能为空")`；服务端校验全局唯一 | — | 字典类型名称（如「用户身份」） |
| `remark` | String | 否 | — | — | 备注 |

> 服务端唯一性校验：`typeKey` 或 `value` 任一已存在即拒绝（`typeKey` OR `value` 重复）。`status` 字段不在入参中，新增时不显式设置（依赖数据库默认值）。

### 1.3 请求示例

```http
POST /admin/dictionary/addType
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.....
Content-Type: application/json

{
  "typeKey": "user_identity",
  "value": "用户身份",
  "remark": "B端用户身份枚举"
}
```

### 1.4 响应结构

`Result<Long>`，`data` 为新增记录的数据库自增 ID。

### 1.5 响应示例

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": 10
}
```

### 1.6 错误码说明

| code | errMsg | 触发场景 |
| --- | --- | --- |
| `400000` | 无效的参数 | `typeKey` 或 `value` 已存在（消息为「字典类型键或者值已存在」）；`@NotBlank` 校验失败 |
| `401000`/`401001`/`401003`/`401004` | token 相关 | 未携带/无效/过期/越权 token（HTTP 401） |

> 注：服务端抛出「字典类型键或者值已存在」时使用的是 `ServiceException` 默认错误码（`500001 操作失败`），实际返回 code 以全局异常处理为准。前端应同时以 `code !== 200000` 判断失败并展示 `errMsg`。

## 二、查询字典类型列表

### 2.1 接口信息

| 项目 | 内容 |
| --- | --- |
| 路径 | `GET /admin/dictionary/listType` |
| 认证 | 需要 token（`Authorization: Bearer {accessToken}`），且 `userFrom=sys` |
| 权限 | 所有已登录 B 端用户 |
| Content-Type | 无请求体（参数通过 query string 传递） |
| 操作审计 | 无 |

### 2.2 请求参数

query 参数为 `DictionaryTypeListReqDTO`（继承 `BasePageReqDTO`，含分页字段）：

| 字段 | 类型 | 必传 | 校验规则 | 默认值 | 含义 | 查询匹配方式 |
| --- | --- | --- | --- | --- | --- | --- |
| `pageNo` | Integer | 否 | — | `1` | 页码，从 1 开始 | — |
| `pageSize` | Integer | 否 | — | `10` | 每页大小 | — |
| `typeKey` | String | 否 | — | `null` | 字典类型业务主键 | 精确匹配（`type_key =`） |
| `value` | String | 否 | — | `null` | 字典类型名称 | 模糊匹配（`value like concat('%', ?, '%')`） |

> 该接口使用 `@Validated` 但 DTO 字段无校验注解，参数全部可空。全部不传则分页返回全部字典类型。

### 2.3 请求示例

```http
GET /admin/dictionary/listType?pageNo=1&pageSize=10&value=用户
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.....
```

查询全部（仅分页）：

```http
GET /admin/dictionary/listType?pageNo=1&pageSize=20
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.....
```

### 2.4 响应结构

`Result<BasePageVO<DictionaryTypeVO>>`，外层 `BasePageVO` 字段：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `totals` | Integer | 总记录数 |
| `totalPages` | Integer | 总页数 |
| `list` | List&lt;DictionaryTypeVO&gt; | 数据列表 |

列表元素 `DictionaryTypeVO` 字段：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `id` | Long | 字典类型自增主键 |
| `typeKey` | String | 字典类型业务主键 |
| `value` | String | 字典类型名称 |
| `remark` | String | 备注 |
| `status` | Integer | 状态：`1` 正常 / `0` 停用 |

### 2.5 响应示例

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": {
    "totals": 5,
    "totalPages": 1,
    "list": [
      {
        "id": 1,
        "typeKey": "user_identity",
        "value": "用户身份",
        "remark": "B端用户身份枚举",
        "status": 1
      },
      {
        "id": 2,
        "typeKey": "user_status",
        "value": "用户状态",
        "remark": "B端用户状态枚举",
        "status": 1
      }
    ]
  }
}
```

### 2.6 错误码说明

| code | errMsg | 触发场景 |
| --- | --- | --- |
| `400006` | 参数类型不匹配 | `pageNo`/`pageSize` 传非数字 |
| `401000`/`401001`/`401003`/`401004` | token 相关 | 未携带/无效/过期/越权 token（HTTP 401） |

> 该接口为纯查询，不主动抛业务错误。无匹配数据时 `totals=0`、`list=[]`。

## 三、编辑字典类型

### 3.1 接口信息

| 项目 | 内容 |
| --- | --- |
| 路径 | `POST /admin/dictionary/editType` |
| 认证 | 需要 token（`Authorization: Bearer {accessToken}`），且 `userFrom=sys` |
| 权限 | 所有已登录 B 端用户 |
| Content-Type | `application/json` |
| 操作审计 | 无 |

### 3.2 请求参数

请求体为 `DictionaryTypeWriteReqDTO`（与新增共用 DTO）：

| 字段 | 类型 | 必传 | 校验规则 | 默认值 | 含义 |
| --- | --- | --- | --- | --- | --- |
| `typeKey` | String | 是 | `@NotBlank`；作为定位主键，服务端据此查待编辑记录 | — | 字典类型业务主键（**不可修改**，仅用于定位） |
| `value` | String | 是 | `@NotBlank`；服务端校验除自身外全局唯一 | — | 字典类型名称（可修改） |
| `remark` | String | 否 | — | — | 备注（仅当传入非空值时才更新） |

> 服务端逻辑：
> - 根据 `typeKey` 查询记录，不存在则抛「字典类型不存在」。
> - 校验 `value` 唯一性（排除当前 `typeKey` 自身），若其他记录已使用该 `value` 则抛「字典类型的值已存在」。
> - `typeKey` 不会被修改（编辑仅更新 `value` 与 `remark`）。
> - `remark` 仅当传入非空（`StringUtil.isNotBlank`）时才覆盖原值，否则保留原备注。
> - `status` 不在入参中，编辑不会修改状态。

### 3.3 请求示例

```http
POST /admin/dictionary/editType
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9.....
Content-Type: application/json

{
  "typeKey": "user_identity",
  "value": "B端用户身份",
  "remark": "更新后的备注"
}
```

### 3.4 响应结构

`Result<Long>`，`data` 为编辑记录的数据库 ID。

### 3.5 响应示例

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": 1
}
```

### 3.6 错误码说明

| code | errMsg | 触发场景 |
| --- | --- | --- |
| `400000`/`500001` | 操作失败 | 字典类型不存在（`typeKey` 查不到）；字典类型的值已存在（`value` 与其他记录重复）；`@NotBlank` 校验失败 |
| `401000`/`401001`/`401003`/`401004` | token 相关 | 未携带/无效/过期/越权 token（HTTP 401） |

> 注：「字典类型不存在」「字典类型的值已存在」由 `ServiceException` 抛出，前端应以 `code !== 200000` 判断失败并展示 `errMsg`。

## 四、枚举值

### 4.1 字典类型状态（status）

| 值 | 含义 |
| --- | --- |
| `1` | 正常 |
| `0` | 停用 |

> **重要**：字典 `status` 为 **Integer 类型**（`1`/`0`），与 B 端用户状态（String `enable`/`disable`）不同，请勿混淆。当前接口不提供 `status` 修改能力，状态仅可通过列表查询读取。

---

## 五、前端注意事项

1. **无删除接口**：字典类型当前无删除接口，新增后只能编辑 `value`/`remark`，无法删除。前端「字典类型」列表不应提供删除按钮（若提供需后端先补接口）。
2. **typeKey 不可修改**：编辑时 `typeKey` 仅用于定位记录，服务端不会更新它。前端编辑表单中 `typeKey` 字段应设为只读/禁用。
3. **value 唯一性**：`value`（类型名称）全局唯一，新增和编辑都会校验。编辑时排除自身后判断重复。
4. **remark 空值处理**：编辑时 `remark` 仅当传入非空字符串才更新；若想清空备注，当前逻辑无法实现（传空串不会覆盖）。如需清空需后端调整。
5. **status 无法修改**：`status` 不在新增/编辑入参中，无法通过接口切换字典类型启用/停用。前端不应提供状态切换控件。
6. **列表为 GET 请求**：`GET /admin/dictionary/listType` 参数走 query string，注意 `value` 模糊查询时的 URL 编码（中文需 `encodeURIComponent`）。
7. **分页默认值**：`pageNo` 默认 1，`pageSize` 默认 10。若不传分页参数，使用默认值。
8. **与字典数据的层级关系**：字典类型是字典数据的容器，删除/停用字典类型不会级联处理其下字典数据（当前也无删除/停用接口）。新增字典数据时需先确保对应 `typeKey` 的字典类型已存在。
9. **错误码判断**：服务端业务异常（如「字典类型键或者值已存在」「字典类型不存在」）可能以 `500001` 返回，前端应统一以 `code !== 200000` 判断失败并展示 `errMsg`，不要硬编码特定 code。
