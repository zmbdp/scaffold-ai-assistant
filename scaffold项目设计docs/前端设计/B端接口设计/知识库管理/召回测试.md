# 召回测试

## 功能介绍

召回测试（Retrieve Test）用于验证知识库的 RAG 检索质量。B 端「知识库管理 → 召回测试」页面让管理员输入一个测试问题，系统执行完整的 RAG 检索链路（问题 → Embedding 向量 → Milvus 向量检索 → Reranking 重排序），返回命中的文档分块及其相似度分数，**不调用 LLM 生成回答**。

用途：

- 验证某篇文档同步后是否真的能被检索到；
- 对比不同 `topK`、不同过滤条件下的召回效果；
- 排查「用户提问却召不回相关知识」的问题（是文档没同步、还是分块质量差、还是问题表述差异）；
- 评估知识库整体质量（高分块是否相关、低分块是否噪音）。

> 公共约定（网关地址、JWT 认证、`Result<T>` 结构、统一错误码、操作审计说明）见 `B端接口设计/总说明.md`，本文不再重复，仅说明本模块特有内容。

## 1. 知识源召回测试

### 1.1 接口信息

| 项 | 值 |
| --- | --- |
| 方法 | `POST` |
| 路径 | `/admin/knowledge/retrieve-test` |
| 认证 | 需要（`userFrom=sys`） |
| Content-Type | `application/json` |
| 操作审计 | 否（读操作，无 `@LogAction`） |
| 是否调用 LLM | 否（仅做向量检索 + 重排序，不生成回答） |

### 1.2 请求参数（Body：RetrieveReqDTO）

| 字段 | 类型 | 必传 | 默认值 | 校验规则（注解原文） | 含义 |
| --- | --- | --- | --- | --- | --- |
| `question` | String | 是 | - | `@NotBlank(message = "问题不能为空")` | 测试问题，用于生成 Embedding 向量检索 |
| `topK` | Integer | 否 | `5` | `@Min(value = 1, message = "topK最小为1")` `@Max(value = 20, message = "topK最大为20")` | RAG 检索返回的分块数量 |
| `sourceType` | String | 否 | 不过滤 | 无 | 文档类型过滤：`doc` / `javadoc` / `config` / `code` |
| `module` | String | 否 | 不过滤 | 无 | 模块过滤（如 `common-cache`） |

> **topK 默认值来源**：`topK` 为 null 时，chat-service `KnowledgeController.doRetrieve` 使用 Nacos 配置项 `scaffold.rag.top-k` 的值，配置缺省时默认 `5`（`@Value("${scaffold.rag.top-k:5}")`）。

### 1.3 检索逻辑（chat-service `doRetrieve`）

- 若 `sourceType` 或 `module` 任一非空，调用 `vectorStoreService.searchWithFilter(question, topK, filters)`，`filters` 包含 `source_type`（当 sourceType 非空）和 `module`（当 module 非空）两个 Milvus 过滤条件。
- 若两者均为空，调用 `vectorStoreService.search(question, topK)` 默认检索（全库）。
- 返回结果为空时返回空数组 `[]`（非 null）。

> **过滤条件语义**：`sourceType` 过滤的是文档类型（与知识源 `type` 一致），`module` 过滤的是从知识源 path 提取的模块名（如 `common-cache`，无法提取时为 `unknown`）。两者同时传时为 AND 关系。

### 1.4 请求示例

```http
POST /admin/knowledge/retrieve-test
Authorization: Bearer {accessToken}
Content-Type: application/json

{
  "question": "如何实现 Caffeine + Redis 二级缓存？",
  "topK": 5,
  "sourceType": "doc",
  "module": "common-cache"
}
```

**不带过滤条件（全库检索）**：

```json
{
  "question": "脚手架如何接入 Nacos 配置中心？",
  "topK": 10
}
```

### 1.5 响应结构

`Result<List<DocumentVO>>`，`data` 为命中的文档分块列表（按相似度降序，已 Reranking）。无命中时为空数组 `[]`。

**DocumentVO 字段**：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `content` | String | 分块内容（向量化前的文本，纯文本，HTML 已清理为纯文本） |
| `documentId` | Long | 文档ID（关联 `sys_ai_document` 表主键） |
| `title` | String | 文档标题（取文件名） |
| `module` | String | 所属模块（如 `common-cache`） |
| `sourcePath` | String | 源文件绝对路径 |
| `score` | Double | 相似度分数（0-1，越高越相关，由 Reranking 模型给出） |
| `chunkIndex` | Integer | 分块索引（在同一文档中的序号，**从 0 开始**） |

## 错误码说明

> **Feign 透传机制**：本接口经 admin-service 通过 Feign 调用 chat-service `KnowledgeApi.retrieveTest`。chat-service 业务错误以 `Result{code, errMsg}` 返回；admin 的 `unwrap` 检测到 `code != 200000` 时抛 `new ServiceException(errMsg)`，默认码 `500000`（`ResultCode.ERROR`）。因此**前端实际收到的业务错误码统一为 `500000`，`errMsg` 为 chat-service 透传的具体原因**。
>
> **DTO 参数校验**（`@NotBlank` / `@Min` / `@Max`）在 admin Controller 层（`@Validated`）触发，由 admin 全局异常处理器返回 `400000`（`INVALID_PARA`），不经过 Feign。

| 场景 | chat-service 原始码 | 前端实际收到 | errMsg 示例 |
| --- | --- | --- | --- |
| `question` 为空 | — | `400000` | `问题不能为空` |
| `topK` 越界（<1 或 >20） | — | `400000` | `topK最小为1` / `topK最大为20` |
| Embedding 服务连接失败 / 超时 | `500023` / `500024` | `500000` | `AI 服务连接失败` / `AI 服务响应超时`（chat-service 侧原始码，经 Feign 透传后前端观测 500000） |
| AI 配置未设置（无 Embedding API Key） | `500022` | `500000` | `AI 配置未设置` |
| Milvus 检索异常 | — | `500000` | chat-service 透传的具体异常信息 |
| Feign 调用失败 / 网络异常 | — | `500000` | `召回测试失败`（admin 侧描述） |
| token 缺失/无效/越权 | — | `401000` / `401001` / `401003` / `401004`（HTTP 401） | 见总说明 |

> **检索无结果不是错误**：当知识库中无相关分块时，返回 `code=200000` + `data=[]`（空数组），不是错误码。前端应区分「请求失败」（非 200000）与「无召回」（200000 + 空数组）。
>
> **关于 `500022` / `500023` / `500024`**：这些是 Embedding/LLM 服务异常码，在 chat-service 侧抛出，经 Feign 透传后前端观测码为 `500000`，需依据 `errMsg` 文案区分具体原因。

## 枚举值

### `sourceType`（文档类型过滤）

| 值 | 含义 |
| --- | --- |
| `doc` | Markdown 文档 |
| `javadoc` | JavaDoc HTML 文档 |
| `config` | 配置文件 |
| `code` | Java 源码 |

> 不传 `sourceType` 表示不按类型过滤（全库检索）。

### `module`（模块过滤）

字符串，取值为同步时从知识源 path 提取的模块名（如 `common-cache`、`common-core`、`common-web`），无法提取时为 `unknown`。不传表示不按模块过滤。

> 模块名提取规则：从 path 中匹配 `zmbdp-xxx-yyy` 模式，去掉 `zmbdp-` 前缀取首段（如 `zmbdp-common-core` → `common-core`）。前端若要按模块过滤，可先从文档列表观察实际 `module` 取值。

## 前端注意事项

1. **`score` 含义**：相似度分数，范围 0-1，由 Reranking 模型给出（非原始向量余弦相似度）。越高越相关。前端可按分数排序展示，并用颜色/进度条直观区分相关度（如 ≥0.8 高相关、0.5-0.8 中相关、<0.5 低相关）。注意：不同 Reranking 模型的分数分布可能不同，不应硬编码绝对阈值判断「是否相关」，建议作为相对排序参考。
2. **`topK` 默认 5**：不传 `topK` 时默认 5。验证召回率时建议调大（如 10、20）观察完整召回情况；验证精度时保持默认。范围 1-20，越界返回 `400000`。
3. **`chunkIndex` 从 0 开始**：同一文档的分块序号，从 0 递增。注意：分块编号是过滤掉无意义短块（trim 长度 < 20 字符）后重新编号的，因此可能不连续于原始切分顺序，但同一文档内递增唯一。
4. **不调用 LLM**：本接口仅返回检索到的分块，不会生成自然语言回答。若需测试完整 RAG 问答效果，应使用 C 端对话接口。
5. **无召回的排查思路**：若 `data` 为空数组，依次排查：
   - 文档是否已同步（`GET /admin/knowledge/documents` 看 `chunkCount > 0` 且 `status=ACTIVE`）；
   - `sourceType` / `module` 过滤是否过严（先去掉过滤条件全库检索验证）；
   - 问题表述是否与文档语义差异过大（换关键词重试）；
   - 知识源是否 `enabled=true`（同步只处理 enabled=1 的知识源）。
6. **过滤条件是 AND**：`sourceType` 与 `module` 同时传时为 AND 关系，即只召回「同时满足类型和模块」的分块。前端组合过滤时注意可能因条件过严导致无召回。
7. **请求耗时**：检索涉及 Embedding 调用 + Milvus 检索 + Reranking，通常 1-3 秒，受向量服务负载影响。前端设置合理超时（建议 ≥ 30s）并展示 loading。
8. **`content` 是纯文本**：JavaDoc 类型分块的 `content` 已剥离 HTML 标签、反转义实体、折叠空白，并前置章节标题为 `## 标题`，前端直接展示纯文本即可，无需再做 HTML 渲染。
9. **无操作审计**：本接口为读操作，不记录到 `operation_log` 表。频繁测试不会产生审计日志。
