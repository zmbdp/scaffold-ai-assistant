# 知识同步

## 功能介绍

知识同步是把知识源路径下的文件「扫描 → 哈希比对 → 分块 → 向量化 → 写入 Milvus」的过程，是知识库从「配置」变成「可检索」的关键动作。B 端「知识库管理 → 知识同步」页面用于：

- 手动触发一次同步（可选择只同步某类知识源，或强制全量重建）；
- 同步后通过「文档列表」（见 `文档管理.md`）查看结果（新增/更新/删除的文档、分块数量）。

本接口为**异步**实现：admin-service 收到请求后仅向 RabbitMQ 投递一条同步消息并立即返回「已提交」提示，真正的同步流程由 chat-service 的 `KnowledgeSyncConsumer` 异步消费执行。前端无法从本接口直接拿到同步统计结果，需轮询文档列表观察。

> 公共约定（网关地址、JWT 认证、`Result<T>` 结构、统一错误码、操作审计说明）见 `B端接口设计/总说明.md`，本文不再重复，仅说明本模块特有内容。

## 1. 触发知识同步（异步）

### 1.1 接口信息

| 项 | 值 |
| --- | --- |
| 方法 | `POST` |
| 路径 | `/admin/knowledge/sync` |
| 认证 | 需要（`userFrom=sys`） |
| Content-Type | `application/json` |
| 操作审计 | 是：`@LogAction(value = "知识同步", module = "knowledge", recordParams = true)`（记录入参到 `operation_log` 表） |
| 同步方式 | **异步**（RabbitMQ），接口立即返回 |

### 1.2 请求参数（Body：SyncReqDTO）

| 字段 | 类型 | 必传 | 默认值 | 校验规则 | 含义 |
| --- | --- | --- | --- | --- | --- |
| `sourceType` | String | 否 | `all`（null 也视为全部） | 无校验注解 | 同步类型过滤：`all` / `doc` / `javadoc` / `config` / `code` |
| `force` | Boolean | 否 | `false` | 无校验注解 | 是否强制全量重新同步。`false`=增量（仅同步哈希变更的文件）；`true`=全量（跳过哈希检查，所有文件视为更新） |

> **`force` 判定**：源码中 `boolean force = dto != null && Boolean.TRUE.equals(dto.getForce())`，即仅当显式传 `true` 才全量，传 `false` / `null` / 不传均为增量。

### 1.3 请求示例

**全量增量同步（最常用）**：

```http
POST /admin/knowledge/sync
Authorization: Bearer {accessToken}
Content-Type: application/json

{
  "sourceType": "all",
  "force": false
}
```

**强制全量重新同步**（修改了 chunkSize/chunkOverlap 或怀疑向量丢失时使用）：

```json
{
  "sourceType": "all",
  "force": true
}
```

**仅同步代码类知识源**：

```json
{
  "sourceType": "code",
  "force": false
}
```

### 1.4 响应结构

`Result<String>`，`data` 为提示信息字符串（**不是同步统计结果**）。

> **关键**：`data` 是固定文案 `"知识同步任务已提交，请稍后通过文档列表查看同步结果"`，仅表示 MQ 消息已成功投递，不代表同步已完成或成功。同步流程在 chat-service 后台异步执行，统计结果（`SyncResultVO`）只记录在 chat-service 日志中，不返回给前端。

### 1.5 响应示例

**成功（MQ 消息已投递）**：

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": "知识同步任务已提交，请稍后通过文档列表查看同步结果"
}
```

### 1.6 同步内部流程（chat-service 异步执行）

admin-service `AiAdminServiceImpl.syncKnowledge` 将 `sourceType` / `force` 封装为 `KnowledgeSyncMessage`，投递到 Direct 交换机 `knowledge.sync.exchange`（路由键 `knowledge.sync`）。chat-service `KnowledgeSyncConsumer` 消费后调用 `KnowledgeLoaderServiceImpl.syncKnowledge(sourceType, force)`，执行 12 步同步流程：

1. 查询所有 `enabled=1` 的知识源（`sourceType` 非空且非 `all` 时按类型过滤）。
2. 遍历每个知识源，按知识源粒度加 Redisson 分布式锁（`knowledge:sync:{knowledgeSourceId}`，租约 30 分钟）。
3. 遍历知识源路径，收集当前所有文件（path 可指向单文件或目录，目录时递归扫描匹配扩展名的文件）。
4. 从 `sys_ai_document` 查询该知识源已有文档（不含 DELETED）。
5. 对比文件路径，识别「新增 / 更新 / 删除」三种状态。
6. **新增文件**：读内容 → 算 SHA-256 哈希 → 分块 → 插 MySQL → 写 Milvus。
7. **更新文件**：读内容 → 算哈希 → MySQL `version+1` → 删旧 Milvus 分块 → 重新分块 → 写新 Milvus 分块。
8. **删除文件**：删 Milvus 分块 → MySQL `status=DELETED`（软删除）。
9. **`force=true` 时**：跳过哈希检查，所有已存在文件均视为「更新」重新分块写入。
10. **增量跳过条件**（`force=false` 时，三者同时满足才跳过）：哈希相同 + `chunk_count > 0` + 哈希非空字符串。任一不满足都重新同步（覆盖上次失败回滚的场景）。
11. 更新知识源 `lastSyncDate` 与 `updateDate`。
12. 释放分布式锁。

> **失败处理**：单文件处理失败计入 `failedDocuments`，不影响其他文件；同步流程内部已按知识源加分布式锁，多实例并发触发也只有一个实例同步同一知识源。MQ 消费失败会自动重试（默认 3 次）。

### 1.7 同步状态查询（间接）

本接口不返回同步结果，前端可通过以下方式间接判断同步进度：

| 观察对象 | 接口 | 判断依据 |
| --- | --- | --- |
| 知识源 `lastSyncDate` | `GET /admin/knowledge/sources` | `lastSyncDate` 更新为当天日期，表示该知识源本轮同步流程已走完 |
| 文档列表变化 | `GET /admin/knowledge/documents?sourceId=xxx` | 新增文档出现、`chunkCount > 0`、`updateDate` 变更为当天，表示该文件已同步成功 |
| 失败文档 | `GET /admin/knowledge/documents?status=ACTIVE` | 若文件反复同步失败，DB 中可能无该文档记录（新增失败回滚）或 `chunkCount=0`（更新失败回滚） |

> 前端建议：触发同步后轮询文档列表（如每 5-10 秒一次），直到 `lastSyncDate` 更新或预期文档均出现 `chunkCount > 0`，再提示用户「同步完成」。

## 错误码说明

> **注意**：本接口的业务逻辑在 admin-service 侧仅为「投递 MQ 消息」，不调用 Feign。MQ 投递失败时 admin 抛 `new ServiceException("触发知识同步失败：" + e.getMessage())`，默认码 `500000`。chat-service 异步同步过程中的失败**不会返回给前端**（仅记录在 chat-service 日志）。

| 场景 | 前端实际收到 | errMsg 示例 |
| --- | --- | --- |
| MQ 消息投递成功 | `200000` | `操作成功`（`data` 为提示文案） |
| MQ 投递失败（RabbitMQ 异常） | `500000` | `触发知识同步失败：xxx` |
| token 缺失/无效/越权 | `401000` / `401001` / `401003` / `401004`（HTTP 401） | 见总说明 |

> **异步同步内部失败不回传**：若 chat-service 消费端同步执行中部分文件失败（如 Embedding 超时、Milvus 写入异常），前端**不会**收到错误码，只能通过文档列表观察到「某些文件未出现 / `chunkCount=0`」。需排查 chat-service 日志定位。

## 枚举值

### 同步类型 `sourceType`

| 值 | 含义 |
| --- | --- |
| `all` | 全部知识源（传 null 也等同 all） |
| `doc` | 仅 Markdown 文档知识源 |
| `javadoc` | 仅 JavaDoc HTML 知识源 |
| `config` | 仅配置文件知识源 |
| `code` | 仅 Java 源码知识源 |

> `sourceType` 实际是按 `sys_ai_knowledge_source.type` 字段过滤知识源（`sourceType` 非空且不等于 `all` 时 `WHERE type = ?`）。传任意其他字符串会因无匹配类型的知识源而空跑（同步 0 个知识源，返回成功）。

### `force` 参数语义

| 值 | 含义 | 适用场景 |
| --- | --- | --- |
| `false`（默认） | 增量同步：哈希未变且 `chunk_count>0` 且哈希非空的文件跳过 | 日常同步，文件未改不动 |
| `true` | 全量同步：跳过哈希检查，所有已存在文件重新分块写入向量 | 修改了 chunkSize/chunkOverlap、怀疑向量丢失、Embedding 模型变更后重建向量 |

## SyncResultVO（同步结果统计，仅供了解）

> **重要**：`SyncResultVO` 是 chat-service 同步流程内部产生的统计对象，**B 端接口不返回它**（本接口返回 `Result<String>`）。它仅在 chat-service 日志和 XXL-JOB 调度日志中输出。此处列出字段仅供前端了解同步流程会统计哪些指标，**前端无法通过接口获取**。

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `totalDocuments` | Integer | 扫描到的文档总数 |
| `updatedDocuments` | Integer | 新增 + 更新的文档数 |
| `deletedDocuments` | Integer | 删除的文档数 |
| `skippedDocuments` | Integer | 跳过的文档数（哈希未变，仅增量模式有值） |
| `failedDocuments` | Integer | 处理失败的文档数 |
| `duration` | Long | 同步耗时（毫秒） |

## 前端注意事项

1. **接口是异步的，返回不等于完成**：`code=200000` 只表示 MQ 消息已投递，同步在后台异步执行（可能数分钟）。前端收到成功响应后应提示「同步任务已提交，请稍后查看结果」，并引导用户通过文档列表观察。
2. **不要在接口返回后立即提示「同步完成」**：应通过轮询文档列表（1.7 节）确认 `lastSyncDate` 更新或目标文档 `chunkCount > 0` 后再提示完成。
3. **`force=true` 全量同步耗时较长**：会重新 Embedding 所有文件并重建向量，文件多时可能数分钟到数十分钟。前端应明确告知用户「全量重建耗时较长」，并建议在低峰期执行。
4. **`force=true` 会短暂导致向量缺失**：更新流程中「先删旧向量再写新向量」，若此期间发生 RAG 检索可能召回不全。非必要不用 force。
5. **修改分块参数后必须 force 同步**：知识源的 `chunkSize` / `chunkOverlap` 变更后，旧向量是按旧参数切的，增量同步（`force=false`）因哈希未变会跳过，导致向量与配置不一致。此时必须用 `force=true` 重建。
6. **无同步任务去重**：接口不校验是否有同步任务正在执行（MQ 可能堆积多条消息）。但 chat-service 同步流程内部有 Redisson 按知识源粒度锁，同一知识源并发同步时后到的会因获取锁失败而跳过（日志告警），不会产生数据不一致。前端应避免高频重复点击触发，建议加按钮防抖。
7. **定时同步任务**：除手动触发外，XXL-JOB 调度中心默认每日凌晨 2:00 执行一次增量同步（`knowledgeSyncJob`，参数 `false`）。前端无需关心，了解即可。
8. **`sourceType` 传非法值**：传 `all` / `doc` / `javadoc` / `config` / `code` 之外的值（如 `abc`）不会报错，但会因无匹配类型的知识源而空跑（同步 0 个），返回成功。前端应限制选项为合法枚举。
