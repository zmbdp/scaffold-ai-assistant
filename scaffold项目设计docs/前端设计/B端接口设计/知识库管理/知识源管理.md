# 知识源管理

## 功能介绍

知识源（Knowledge Source）是知识库的顶层组织单元，描述「一类知识的来源位置与分块规则」。B 端「知识库管理 → 知识源」页面用于维护知识源元数据：新增一个知识源（指明文件系统路径、类型、分块参数），更新其配置，或删除（级联清理其下所有文档与向量）。

知识源本身不存文本内容，只存「去哪里取、按什么规则切分」。真正的文档扫描、分块、向量化由「知识同步」触发（见 `知识同步.md`）。因此新增/更新知识源后，需手动触发一次同步才会产生可检索的向量。

> 公共约定（网关地址、JWT 认证、`Result<T>` / `BasePageVO<T>` 结构、统一错误码、日期格式、操作审计说明）见 `B端接口设计/总说明.md`，本文不再重复，仅说明本模块特有内容。

## 接口清单

| 序号 | 方法 | 路径 | 说明 | 操作审计 |
| --- | --- | --- | --- | --- |
| 1 | GET | `/admin/knowledge/sources` | 获取知识源列表（分页） | 否（读操作） |
| 2 | POST | `/admin/knowledge/sources` | 新增知识源 | 是 |
| 3 | PUT | `/admin/knowledge/sources/{id}` | 更新知识源 | 是 |
| 4 | DELETE | `/admin/knowledge/sources/{id}` | 删除知识源（级联） | 是 |

> **说明**：当前 B 端未暴露「知识源详情 GET `/admin/knowledge/sources/{id}`」接口（`KnowledgeApi` Feign 接口与 admin `KnowledgeController` 均未定义该方法）。如需查看单条详情，使用列表接口获取（列表返回的 `KnowledgeSourceVO` 已含全部字段）。

---

## 通用说明

- **认证要求**：所有接口需在请求头携带 `Authorization: Bearer {accessToken}`，且 token 中 `userFrom=sys`（B 端系统用户）。
- **Content-Type**：除特殊说明外统一为 `application/json`。
- **路径映射**：前端请求 `/admin/knowledge/sources` → 网关 `StripPrefix=1` → admin-service `/knowledge/sources` → Feign → chat-service `KnowledgeApi`。
- **操作审计**：标注「是」的接口通过 `@LogAction` 注解自动记录到 `operation_log` 表，`module = "knowledge"`。

## 1. 获取知识源列表（分页）

### 1.1 接口信息

| 项 | 值 |
| --- | --- |
| 方法 | `GET` |
| 路径 | `/admin/knowledge/sources` |
| 认证 | 需要（`userFrom=sys`） |
| Content-Type | 无请求体（Query 参数） |
| 操作审计 | 否 |

### 1.2 请求参数（Query）

| 参数名 | 类型 | 必传 | 默认值 | 校验规则 | 含义 |
| --- | --- | --- | --- | --- | --- |
| `pageNo` | Integer | 否 | `1` | Controller `defaultValue = "1"`；Service 兜底：null 或 <1 时取 1 | 页码，从 1 开始 |
| `pageSize` | Integer | 否 | `10` | Controller `defaultValue = "10"`；Service 兜底：null 或 <1 时取 **20** | 每页数量 |
| `type` | String | 否 | 查全部 | 无 | 类型过滤，枚举：`doc` / `javadoc` / `config` / `code` |

> **pageSize 兜底差异**：Controller 注解默认值为 `10`，但 chat-service `KnowledgeServiceImpl.listSources` 在 `pageSize` 为 null 或 <1 时兜底为 `20`。正常传参时以 Controller 默认 `10` 为准。

### 1.3 请求示例

```http
GET /admin/knowledge/sources?pageNo=1&pageSize=10&type=doc
Authorization: Bearer {accessToken}
```

### 1.4 响应结构

`Result<BasePageVO<KnowledgeSourceVO>>`，`data` 为分页结构：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `totals` | Integer | 总记录数 |
| `totalPages` | Integer | 总页数 |
| `list` | List&lt;KnowledgeSourceVO&gt; | 知识源列表（按 `createDate` 降序） |

**KnowledgeSourceVO 字段**：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `id` | Long | 知识源ID（雪花算法） |
| `name` | String | 知识源名称 |
| `path` | String | 知识源路径（文件系统绝对路径或相对 `knowledge.base-path` 的相对路径） |
| `type` | String | 类型：`doc` / `javadoc` / `config` / `code` |
| `enabled` | Boolean | 是否启用（实体存 Integer 0/1，VO 转 Boolean） |
| `chunkSize` | Integer | 分块大小 |
| `chunkOverlap` | Integer | 分块重叠大小 |
| `lastSyncDate` | Long | 最近同步日期，格式 `YYYYMMDD`（如 `20260723`），从未同步为 null |
| `createDate` | Long | 创建日期，格式 `YYYYMMDD` |
| `updateDate` | Long | 更新日期，格式 `YYYYMMDD` |

## 2. 新增知识源

### 2.1 接口信息

| 项 | 值 |
| --- | --- |
| 方法 | `POST` |
| 路径 | `/admin/knowledge/sources` |
| 认证 | 需要（`userFrom=sys`） |
| Content-Type | `application/json` |
| 操作审计 | 是：`@LogAction(value = "新增知识源", module = "knowledge", recordParams = true)`（记录入参到 `operation_log` 表） |

### 2.2 请求参数（Body：KnowledgeSourceReqDTO）

| 字段 | 类型 | 必传 | 默认值 | 校验规则（注解原文） | 含义 |
| --- | --- | --- | --- | --- | --- |
| `name` | String | 是 | - | `@NotBlank(message = "知识源名称不能为空")`；Service 校验名称不能重复 | 知识源名称 |
| `path` | String | 是 | - | `@NotBlank(message = "知识源路径不能为空")`；Service 校验路径必须真实存在（`FileUtil.exist`） | 知识源路径，文件系统绝对路径或相对 `knowledge.base-path` 的相对路径 |
| `type` | String | 是 | - | `@NotBlank(message = "类型不能为空")` | 类型：`doc` / `javadoc` / `config` / `code` |
| `enabled` | Boolean | 否 | `true` | 无 | 是否启用（null 视为 true） |
| `chunkSize` | Integer | 否 | `500` | `@Min(value = 100, message = "分块大小最小为100")` `@Max(value = 2000, message = "分块大小最大为2000")` | 分块大小（字符数） |
| `chunkOverlap` | Integer | 否 | `50` | `@Min(value = 0, message = "分块重叠大小最小为0")` `@Max(value = 200, message = "分块重叠大小最大为200")` | 分块重叠大小（需小于 chunkSize） |

### 2.3 请求示例

```http
POST /admin/knowledge/sources
Authorization: Bearer {accessToken}
Content-Type: application/json

{
  "name": "脚手架后端设计文档",
  "path": "d:/GitHub/scaffold-ai-assistant/scaffold项目设计docs/后端设计",
  "type": "doc",
  "enabled": true,
  "chunkSize": 500,
  "chunkOverlap": 50
}
```

> **path 路径需与部署环境一致**：本地 IDEA 运行（dev）用宿主机绝对路径（如 `d:/GitHub/...`）；Docker 部署（test/prd）用容器内路径（如 `/knowledge/...`，由 docker-compose `./scaffold-ai-assistant:/knowledge:ro` 挂载）。

### 2.4 响应结构

`Result<KnowledgeSourceVO>`，`data` 为新建后的知识源（含生成的 `id`）。字段同 1.4 节 KnowledgeSourceVO。

### 2.5 响应示例

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": {
    "id": 1894736281934594,
    "name": "脚手架后端设计文档",
    "path": "d:/GitHub/scaffold-ai-assistant/scaffold项目设计docs/后端设计",
    "type": "doc",
    "enabled": true,
```

## 4. 删除知识源（级联）

### 4.1 接口信息

| 项 | 值 |
| --- | --- |
| 方法 | `DELETE` |
| 路径 | `/admin/knowledge/sources/{id}` |
| 认证 | 需要（`userFrom=sys`） |
| Content-Type | 无请求体 |
| 操作审计 | 是：`@LogAction(value = "删除知识源", module = "knowledge")`（未记录入参） |

### 4.2 删除行为（级联）

源码 `KnowledgeServiceImpl.deleteSource`（事务）：

1. 校验知识源存在，不存在抛 `404002 知识源不存在`。
2. 查询该知识源下所有 `status != DELETED` 的文档。
3. 遍历文档：Milvus 物理删除对应向量分块（`vectorStoreService.deleteByDocumentId`） + MySQL 文档软删除（`status = DELETED`）。
4. 知识源置为 `enabled = 0`（**保留记录便于审计**，不物理删除知识源记录）。

> 操作不可逆：向量被物理删除，文档被软删除。知识源记录保留但停用。

### 4.3 请求参数

**路径参数**：

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `id` | Long | 是 | 知识源ID |

### 4.4 请求示例

```http
DELETE /admin/knowledge/sources/1894736281934594
Authorization: Bearer {accessToken}
```

### 4.5 响应结构

`Result<Void>`，`data` 为 `null`。

### 4.6 响应示例

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": null

```

## 错误码说明

> **Feign 透传机制（重要）**：B 端知识库接口经 admin-service 通过 Feign 调用 chat-service。chat-service 的业务错误（如知识源不存在）以 `Result{code, errMsg}` 返回；admin-service 的 `unwrap` 方法检测到 `code != 200000` 时，抛 `new ServiceException(errMsg)`，其默认错误码为 `500000`（`ResultCode.ERROR`）。因此**前端实际收到的业务错误码统一为 `500000`，`errMsg` 为 chat-service 透传的具体原因**（如「知识源不存在」「知识源名称已存在：xxx」）。
>
> **DTO 参数校验**（`@NotBlank` / `@Min` / `@Max`）在 admin Controller 层（`@Validated`）触发，由 admin 全局异常处理器返回 `400000`（`INVALID_PARA`），不经过 Feign。

| 场景 | chat-service 原始码 | 前端实际收到 | errMsg 示例 |
| --- | --- | --- | --- |
| DTO 校验失败（name/path/type 为空、chunkSize 越界等） | — | `400000` | `知识源名称不能为空` / `分块大小最小为100` |
| 路径变量 `id` 类型不匹配（非数字） | — | `400006` | `参数类型不匹配` |
| 知识源不存在（更新/删除时） | `404002` | `500000` | `知识源不存在` |
| 知识源路径不存在（新增/更新时） | `400000` | `500000` | `知识源路径不存在：xxx` |
| 知识源名称已存在（新增/更新时） | `400000` | `500000` | `知识源名称已存在：xxx` |
| Feign 调用失败 / 网络异常 | — | `500000` | `获取知识源列表失败` 等 admin 侧描述 |
| token 缺失/无效/越权 | — | `401000` / `401001` / `401003` / `401004`（HTTP 401） | 见总说明 |

> **关于 `404002`**：`ResultCode.AI_KNOWLEDGE_SOURCE_NOT_FOUND(404002, "知识源不存在")` 在 chat-service 抛出，但经 admin Feign 透传后前端观测码为 `500000`。前端若需精确区分「不存在」场景，应依据 `errMsg` 文案判断，而非依赖 `code`。

## 枚举值

### 知识源类型 `type`

| 值 | 含义 | 同步时扫描的文件扩展名 |
| --- | --- | --- |
| `doc` | Markdown 文档 | `.md` / `.markdown` |
| `javadoc` | JavaDoc HTML 文档 | `.html` / `.htm` |
| `config` | 配置文件 | `.yaml` / `.yml` / `.properties` |
| `code` | Java 源码 | `.java` |

> 扩展名映射源自 `KnowledgeLoaderServiceImpl.getExtensionsByType`。`path` 可指向单个文件或目录（目录时递归扫描匹配扩展名的文件）。

## 前端注意事项

1. **无详情接口**：当前未提供 GET 单条知识源详情接口，列表 VO 已含全字段，前端编辑时从列表数据回填表单即可。
2. **新增后需手动同步**：新增/更新知识源仅写元数据，不会自动扫描文件。需跳转「知识同步」手动触发一次同步（见 `知识同步.md`），`lastSyncDate` 才会被写入，文档才会产生。
3. **修改 path / chunkSize / chunkOverlap 后需重建向量**：这些字段影响分块结果，更新后建议以 `force=true` 触发一次全量同步，否则旧向量与新分块规则不一致。
4. **删除是软删知识源 + 物理删向量**：知识源记录保留（`enabled=0`），列表默认会查出 `enabled=0` 的记录（列表查询未过滤 enabled），前端可按 `enabled` 字段标识已停用项；如需彻底清理需后端介入。
5. **`enabled` 字段类型**：VO 为 `Boolean`，后端实体为 `Integer`（0/1），前端按布尔处理即可。
6. **日期字段**：`lastSyncDate` / `createDate` / `updateDate` 均为 `Long` 类型 `YYYYMMDD` 格式，前端展示时自行格式化。
