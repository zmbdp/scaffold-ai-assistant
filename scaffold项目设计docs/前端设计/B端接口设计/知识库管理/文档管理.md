# 文档管理

## 功能介绍

文档（Document）是知识源下被同步进库的具体文件单元，对应 `sys_ai_document` 表一条记录。B 端「知识库管理 → 文档」页面用于：

- 查看已同步进库的文档列表（支持按知识源、状态过滤），监控同步结果；
- 查看单篇文档完整内容（含分块数量、版本号）；
- 删除文档（级联清理其向量分块）；
- 上传新文档——上传后立即分块、向量化、写入 Milvus，**无需等待定时同步任务**。

文档记录由「知识同步」或「文档上传」产生，前端不应直接新增文档，只能通过上传或同步间接生成。

> 公共约定（网关地址、JWT 认证、`Result<T>` / `BasePageVO<T>` 结构、统一错误码、日期格式、操作审计说明）见 `B端接口设计/总说明.md`，本文不再重复，仅说明本模块特有内容。

## 通用说明

- **认证要求**：所有接口需 `Authorization: Bearer {accessToken}`，`userFrom=sys`。
- **操作审计**：标注「是」的接口通过 `@LogAction` 注解记录到 `operation_log` 表，`module = "knowledge"`。
- **列表 vs 详情差异**：列表接口不返回 `content` 字段（设为 null 以减小响应体），详情接口返回完整 `content`；`fileSize` 字段在列表/详情均会返回（实体有值即带出）。

## 1. 获取文档列表（分页）

### 1.2 请求参数（Query）

| 参数名 | 类型 | 必传 | 默认值 | 校验规则 | 含义 |
| --- | --- | --- | --- | --- | --- |

## 3. 删除文档（级联删向量）

### 3.1 接口信息

| 项 | 值 |
| --- | --- |
| 方法 | `DELETE` |
| 路径 | `/admin/knowledge/documents/{id}` |
| 认证 | 需要（`userFrom=sys`） |
| Content-Type | 无请求体 |
| 操作审计 | 是：`@LogAction(value = "删除文档", module = "knowledge")`（未记录入参） |

### 3.2 删除行为（事务）

源码 `KnowledgeServiceImpl.deleteDocument`（`@Transactional`）：

1. 校验文档存在，不存在抛 `404003 文档不存在`。
2. Milvus 物理删除该 `document_id` 的所有向量分块（`vectorStoreService.deleteByDocumentId`）。
3. MySQL 软删除：`status = DELETED`，更新 `updateDate`。

> 文档记录保留（软删除），向量被物理删除。已删除文档默认不可被 RAG 检索召回。

### 3.3 请求参数

**路径参数**：

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `id` | Long | 是 | 文档ID |

### 3.4 请求示例

```http
DELETE /admin/knowledge/documents/1894736281934595
Authorization: Bearer {accessToken}
```

### 3.5 响应结构

`Result<Void>`，`data` 为 `null`。

### 3.6 响应示例

```json
{
  "code": 200000,
  "errMsg": "操作成功",
  "data": null
```

## 4. 上传文档到指定知识源

### 4.1 接口信息

| 项 | 值 |
| --- | --- |
| 方法 | `POST` |
| 路径 | `/admin/knowledge/documents/upload` |
| 认证 | 需要（`userFrom=sys`） |
| Content-Type | `multipart/form-data`（Controller `consumes = MediaType.MULTIPART_FORM_DATA_VALUE`） |
| 操作审计 | 是：`@LogAction(value = "上传文档", module = "knowledge", recordParams = true)` |

### 4.2 上传后处理流程

上传经 Feign 转发到 chat-service，由 `KnowledgeServiceImpl.uploadDocument` + `KnowledgeLoaderServiceImpl.uploadDocument` 执行：

1. 参数校验：`knowledgeSourceId` 非空、文件非空、文件名非空、文件大小 ≤ 50MB、扩展名合法。
2. 校验知识源存在且 `enabled=1`。
3. 拼接完整路径 `resolveSourcePath(source.path) + File.separator + 安全文件名`，校验文件不存在（避免覆盖）。
4. 文件内容以 UTF-8 读取，保存到磁盘（自动创建父目录）。
5. 立即执行分块 → 调 Embedding → 写入 Milvus（复用同步流程的 `processAddedFile` 逻辑，**不等定时同步任务**）。
6. 返回新插入的文档 VO（`version=1`、`status=ACTIVE`、含生成的 `id` 和 `chunkCount`）。

> **失败回滚**：若 Embedding 或 Milvus 写入失败，`sys_ai_document` 记录会被回滚删除，但磁盘文件保留（便于下次同步重试）。前端会收到错误响应。

### 4.3 请求参数

**Query 参数**：

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `knowledgeSourceId` | Long | 是 | 知识源ID（`@RequestParam`） |

**Body（form-data）**：

| 字段名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `file` | File（二进制） | 是 | 上传的文件（`@RequestPart("file") MultipartFile`） |

### 4.4 文件限制

| 限制项 | 规则 | 违反时响应 |
| --- | --- | --- |
| 文件非空 | `file != null && !file.isEmpty()` | `400000`「上传文件不能为空」 |
| 文件名非空 | `originalFilename` 非空白 | `400000`「文件名不能为空」 |
| 文件大小 | `≤ 50MB`（`50L * 1024 * 1024` 字节） | `400000`「文件大小超过 50MB 限制：当前 XMB」 |
| 扩展名 | ∈ `{md, txt, html, java, py, xml, json}`（小写） | `400000`「不支持的文件类型：.xxx，仅支持 [...]」 |

> **重要（源码实际行为）**：`ResultCode` 中定义了 `AI_FILE_TYPE_NOT_SUPPORTED(500027)` 与 `AI_FILE_TOO_LARGE(500028)`，但**当前上传校验代码并未使用它们**，而是统一抛 `ServiceException(msg, ResultCode.INVALID_PARA.getCode())`，即 chat-service 侧原始码为 `400000`。再经 admin Feign 透传（见下方错误码说明），前端实际收到 `500000` + 具体文案。前端判断上传失败原因应依据 `errMsg` 文案，而非依赖特定 code。

### 4.5 请求示例

```http
POST /admin/knowledge/documents/upload?knowledgeSourceId=1894736281934592
Authorization: Bearer {accessToken}
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary

------WebKitFormBoundary
Content-Disposition: form-data; name="file"; filename="ADD_NEW_MODULE.md"
Content-Type: text/markdown

（文件二进制内容）
------WebKitFormBoundary--
```

curl 示例：

```bash
curl -X POST 'http://localhost:10030/admin/knowledge/documents/upload?knowledgeSourceId=1894736281934592' \
  -H 'Authorization: Bearer {accessToken}' \
  -F 'file=@/path/to/ADD_NEW_MODULE.md'
```

### 4.6 响应结构

`Result<KnowledgeDocumentVO>`，`data` 为新插入的文档（含完整 `content`、生成的 `id`、`chunkCount`）。字段同 1.4 节。

### 4.7 响应示例

## 错误码说明

> **Feign 透传机制（重要）**：B 端文档接口经 admin-service 通过 Feign 调用 chat-service。chat-service 业务错误以 `Result{code, errMsg}` 返回；admin 的 `unwrap` 检测到 `code != 200000` 时抛 `new ServiceException(errMsg)`，默认码 `500000`（`ResultCode.ERROR`）。因此**前端实际收到的业务错误码统一为 `500000`，`errMsg` 为 chat-service 透传的具体原因**。

| 场景 | chat-service 原始码 | 前端实际收到 | errMsg 示例 |
| --- | --- | --- | --- |
| 路径变量 `id` 类型不匹配（非数字） | — | `400006` | `参数类型不匹配` |
| 文档不存在（详情/删除时） | `404003` | `500000` | `文档不存在` |
| 上传：`knowledgeSourceId` 为空 | `400000` | `500000` | `知识源ID不能为空` |
| 上传：文件为空 | `400000` | `500000` | `上传文件不能为空` |
| 上传：文件名为空 | `400000` | `500000` | `文件名不能为空` |
| 上传：文件超过 50MB | `400000` | `500000` | `文件大小超过 50MB 限制：当前 60MB` |
| 上传：扩展名不支持 | `400000` | `500000` | `不支持的文件类型：.pdf，仅支持 [md, txt, html, java, py, xml, json]` |
| 上传：知识源不存在 | `404002` | `500000` | `知识源不存在` |
| 上传：知识源未启用 | `400000` | `500000` | `知识源未启用，无法上传文档：sourceId = xxx` |
| 上传：文件已存在 | `400000` | `500000` | `文件已存在，请删除后重新上传或修改文件名：xxx` |
| 上传：向量化失败（Embedding/Milvus 异常） | `400000` | `500000` | `文档向量化失败：xxx` |
| 上传：读取文件内容失败 | `400000` | `500000` | `读取文件内容失败：xxx` |
| Feign 调用失败 / 网络异常 | — | `500000` | `获取文档列表失败` / `上传文档失败` 等 admin 侧描述 |

> **关于 `500027` / `500028`**：`ResultCode.AI_FILE_TYPE_NOT_SUPPORTED(500027)` 与 `AI_FILE_TOO_LARGE(500028)` 已定义，但当前上传实现未使用，统一用 `400000`（chat-service 侧）→ `500000`（前端侧）。前端不应依赖 500027/500028 做分支判断。
>
> **关于 `404003`**：`ResultCode.AI_DOCUMENT_NOT_FOUND(404003, "文档不存在")` 在 chat-service 抛出，经 Feign 透传后前端观测码为 `500000`，需依据 `errMsg` 文案区分。

## 枚举值

### 文档类型 `type`

同知识源类型：`doc` / `javadoc` / `config` / `code`（详见 `知识源管理.md`）。

### 文档状态 `status`

| 值 | 含义 | 是否可被检索 |
| --- | --- | --- |
| `ACTIVE` | 活跃，已成功写入向量 | 是 |
| `DELETED` | 已删除（软删除） | 否 |

> **关于 `SYNCING`**：`KnowledgeDocumentVO` 与 Controller 注释中提及 `ACTIVE/DELETED/SYNCING`，但源码中同步流程不会写入 `SYNCING` 状态（新增/更新直接置 `ACTIVE`，删除置 `DELETED`），当前实际只会出现 `ACTIVE` / `DELETED`。前端列表 `status` 过滤传 `ACTIVE` 或 `DELETED` 即可。

### 上传支持的文件扩展名

| 扩展名 | 说明 |
| --- | --- |
| `.md` | Markdown |
| `.txt` | 纯文本 |
| `.html` | HTML（JavaDoc） |
| `.java` | Java 源码 |
| `.py` | Python 源码 |
| `.xml` | XML |
| `.json` | JSON |

> 注意：上传扩展名白名单（7 种）与知识源类型同步扫描的扩展名（按 type 区分）是两套独立配置。上传接口不校验文件扩展名与目标知识源 `type` 是否匹配，仅校验是否在上述 7 种白名单内。

## 前端注意事项

1. **上传是同步处理**：上传接口会同步完成「保存文件 → 分块 → Embedding → 写 Milvus」全流程，大文件或向量服务慢时接口耗时较长（可能数十秒），前端需设置合理的请求超时（建议 ≥ 60s）并展示 loading。
2. **文件大小限制 50MB**：前端应在选择文件时就做客户端预校验，超过 50MB 直接拦截，避免发起请求被服务端拒绝。
3. **扩展名白名单**：前端文件选择器应限制 `accept=".md,.txt,.html,.java,.py,.xml,.json"`。
4. **同名文件不可上传**：上传会校验目标路径下文件不存在，若已存在同名文件返回错误。前端收到「文件已存在」提示时，应引导用户先删除旧文档或改名重传。
5. **`content` 字段体积**：仅详情接口返回 `content`，列表接口强制置 null。前端列表页不要请求详情批量拉取 content，避免响应过大。
6. **`module` 字段**：从知识源 path 中提取 `zmbdp-xxx-yyy` 模式得到（如 `common-cache`），无法提取时为 `"unknown"`，前端展示需容忍该值。
7. **删除是软删**：删除后文档 `status=DELETED`，记录仍保留。列表默认查出全部状态，前端可用 `status=ACTIVE` 过滤只看有效文档；如需查看已删除项传 `status=DELETED`。
8. **上传成功后**：返回的 VO 中 `chunkCount` 反映实际写入 Milvus 的分块数（已过滤掉长度 < 20 字符的无意义短块），可用于前端提示「已切分为 N 个分块」。
