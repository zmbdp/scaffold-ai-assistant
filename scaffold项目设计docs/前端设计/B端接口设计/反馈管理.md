# 反馈管理

## 功能介绍

反馈管理接口用于 B 端管理员查看用户对 AI 回答的反馈明细，单条记录同时返回反馈信息 + 对话问答摘要（用户问了什么、AI 答了什么、为什么点踩），便于管理员定位回答质量问题。

**数据来源**：`sys_ai_feedback` 表 LEFT JOIN `sys_ai_conversation` 表（`ON f.conversation_id = c.id`），一对一关系。

**answer 截断说明**：`sys_ai_conversation.answer` 字段为 LONGTEXT，列表页 SQL 层用 `LEFT(answer, 200)` 截断 200 字避免响应过大；完整回答通过 C 端会话详情接口（`GET /portal/history/{sessionId}`）查看。

**职责边界**：
- 本接口为 B 端管理员查看用户反馈**明细**（单条反馈记录）。
- 反馈**聚合统计**（点赞率 / 点踩率 / 原因分布）由「满意度统计」接口（`GET /admin/statistics/feedback`）负责。
- C 端用户反馈提交 / 查询 / 撤销由 portal-service 的 `FeedbackPortalController` 负责，不在本接口范围。

---

## 接口详情

### B 端反馈明细分页查询

| 项 | 值 |
| --- | --- |
| 路径 | `GET /admin/feedback/list` |
| 认证 | 必须携带 B 端 JWT（`userFrom=sys`），见总说明第三章 |
| Content-Type | `application/json` |

#### 调用链路

前端请求 `/admin/feedback/list` → 网关 StripPrefix=1 → admin-service `FeedbackController#listFeedbacks(...)` → Feign 调用 chat-service `FeedbackApi#listFeedbacks(...)` → `FeedbackServiceImpl#listFeedbacks(...)` → `SysAiFeedbackMapper#selectFeedbackListPage(...)`。

---

### 请求参数

| 参数名 | 类型 | 必传 | 默认值 | 校验规则 | 含义 |
| --- | --- | --- | --- | --- | --- |
| `pageNo` | Integer | 否 | `1` | Service 层兜底：为 `null` 或 `< 1` 时取 `1` | 页码，从 1 开始 |
| `pageSize` | Integer | 否 | `20` | Service 层兜底：为 `null` 或 `< 1` 时取 `20` | 每页大小 |
| `feedbackType` | String | 否 | 不传则不过滤 | 非空字符串才生效 | 反馈类型过滤，见下方枚举 |
| `dislikeReason` | String | 否 | 不传则不过滤 | 非空字符串才生效 | 点踩原因过滤，见下方枚举（仅 `feedbackType=DISLIKE` 时有意义） |
| `userId` | Long | 否 | 不传则不过滤 | `null` 判空 | 用户ID过滤 |
| `startDate` | Long | 否 | 不传则不过滤 | BIGINT `YYYYMMDD` | 起始日期，过滤 `f.create_date >= startDate` |
| `endDate` | Long | 否 | 不传则不过滤 | BIGINT `YYYYMMDD` | 结束日期，过滤 `f.create_date <= endDate` |

> **过滤参数生效说明**：所有过滤参数均为可选条件，各自独立生效（SQL `<if>` 动态拼接）。多个条件同时传时为 AND 关系（交集查询）。均不传时返回全部反馈明细（按反馈时间降序）。
>
> **dislikeReason 与 feedbackType 的关系**：`dislikeReason` 过滤逻辑独立于 `feedbackType`，可单独传 `dislikeReason=CODE_ERROR`（不传 `feedbackType`），SQL 会过滤 `f.dislike_reason = 'CODE_ERROR'`，等价于只查该原因的点踩记录。但语义上 `dislikeReason` 仅对点踩记录有意义，建议前端同时传 `feedbackType=DISLIKE` 以保证语义清晰。

---

### 请求示例

**示例 1：查询全部反馈（仅分页，不过滤）**

```
GET /admin/feedback/list?pageNo=1&pageSize=20
Authorization: Bearer {accessToken}
```

**示例 2：查询所有点踩记录**

```
GET /admin/feedback/list?pageNo=1&pageSize=20&feedbackType=DISLIKE
Authorization: Bearer {accessToken}
```

**示例 3：查询指定点踩原因的反馈（组合过滤）**

```
GET /admin/feedback/list?pageNo=1&pageSize=20&feedbackType=DISLIKE&dislikeReason=CODE_ERROR
Authorization: Bearer {accessToken}
```

**示例 4：按用户ID + 日期范围过滤**

```
GET /admin/feedback/list?pageNo=1&pageSize=20&userId=10000020&startDate=20260701&endDate=20260723
Authorization: Bearer {accessToken}
```

### 响应结构

返回 `Result<BasePageVO<FeedbackAdminVO>>`，`data` 为分页结果。

**BasePageVO 字段**：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `totals` | Integer | 总记录数 |
| `totalPages` | Integer | 总页数 |
| `list` | List&lt;FeedbackAdminVO&gt; | 反馈明细列表 |

**FeedbackAdminVO 字段**：

| 字段 | 类型 | 含义 | 数据来源 |
| --- | --- | --- | --- |
| `id` | Long | 反馈ID（雪花算法） | `sys_ai_feedback.id` |
| `conversationId` | Long | 对话记录ID | `sys_ai_feedback.conversation_id`，关联 `sys_ai_conversation.id` |
| `userId` | Long | 反馈用户ID | `sys_ai_feedback.user_id` |
| `userFrom` | String | 用户来源（`sys`/`app`） | `sys_ai_feedback.user_from` |
| `feedbackType` | String | 反馈类型（`LIKE`/`DISLIKE`） | `sys_ai_feedback.feedback_type` |
| `dislikeReason` | String | 点踩原因（仅 `DISLIKE` 时有值） | `sys_ai_feedback.dislike_reason` |
| `comment` | String | 文字评论 | `sys_ai_feedback.comment` |
| `createTime` | LocalDateTime | 反馈时间（精确到秒，序列化为 `yyyy-MM-dd HH:mm:ss`） | `sys_ai_feedback.create_time` |
| `question` | String | 用户原始提问（SQL `LEFT(c.question, 100)` 截断 100 字） | `sys_ai_conversation.question` |
| `answerSummary` | String | AI 回答摘要（SQL `LEFT(c.answer, 200)` 截断 200 字） | `sys_ai_conversation.answer` |
| `model` | String | 使用的模型名称 | `sys_ai_conversation.model` |
| `sources` | List&lt;String&gt; | RAG 引用来源列表（由 `sourcesJson` JSON 字符串反序列化得到） | `sys_ai_conversation.sources` |

> **sourcesJson 字段不返回前端**：`FeedbackAdminVO` 中还有一个 `sourcesJson` 字段（`sys_ai_conversation.sources` 的 JSON 字符串原始值），标注了 `@JsonIgnore`，Service 层反序列化为 `sources`（`List<String>`）后清空该字段，不会出现在前端响应中。

> **截断说明**：
> - `question`：SQL `LEFT(c.question, 100)` 截断 100 字。
> - `answerSummary`：SQL `LEFT(c.answer, 200)` 截断 200 字。
> - 完整 AI 回答通过 C 端会话详情接口（`GET /portal/history/{sessionId}`）查看。

> **LEFT JOIN 说明**：使用 LEFT JOIN 兼容「对话记录被软删除但反馈记录仍存在」的场景。当对话记录被软删除（`is_deleted=1`）或物理删除时，`question`/`answerSummary`/`model`/`sources` 字段为 `null`，但反馈信息（`id`/`feedbackType`/`comment` 等）仍正常返回。

### 枚举值

#### feedbackType（反馈类型）

| 值 | 含义 |
| --- | --- |
| `LIKE` | 点赞 |
| `DISLIKE` | 点踩 |

#### dislikeReason（点踩原因，仅 `feedbackType=DISLIKE` 时有值）

| 值 | 含义 |
| --- | --- |
| `OUTDATED` | 信息过时 |
| `IRRELEVANT` | 答非所问 |
| `CODE_ERROR` | 代码错误 |
| `OTHER` | 其他 |

---

### 错误码说明

| code | errMsg | 含义 | 场景 |
| --- | --- | --- | --- |
| `200000` | 操作成功 | 成功 | — |
| `401000` | 令牌不能为空 | 缺 token（HTTP 401） | 未携带 Authorization |
| `401003` | 登录状态已过期！ | Redis 登录态过期（HTTP 401） | token 过期 |
| `401004` | 令牌验证失败！ | 越权访问（HTTP 401） | C 端 token 访问 `/admin/**` |
| `404000` | 服务未找到 | chat-service 不可用（HTTP 404） | chat-service 未注册到 Nacos |
| `500000` | 服务繁忙请稍后重试 | 服务异常（HTTP 500） | Feign 调用异常或数据库异常 |

> 当 chat-service Feign 调用返回非 `200000` 时，admin-service 抛出 `ServiceException("获取反馈明细列表失败")`，HTTP 200 + `code=500001`。

## 前端注意事项

1. **单条记录同时含反馈信息 + 对话问答摘要**：管理员可一眼看到「用户问了什么（`question`）、AI 答了什么（`answerSummary`）、为什么点踩（`dislikeReason`/`comment`）」，无需二次查询对话记录。

2. **answer 截断 200 字**：`answerSummary` 由 SQL `LEFT(c.answer, 200)` 截断，可能截断到一半字符（按字节截断，非按词）。完整 AI 回答需通过 C 端会话详情接口（`GET /portal/history/{sessionId}`）查看。前端展示时可在 `answerSummary` 末尾添加「…」并支持「查看完整回答」跳转。

3. **question 截断 100 字**：`question` 由 SQL `LEFT(c.question, 100)` 截断。

4. **sources 反序列化**：`sources` 字段由 Service 层从 `sourcesJson`（JSON 字符串，如 `["文档1","文档2"]`）反序列化为 `List<String>`。解析失败或空数组时返回 `null`。2026-07-21 之前的旧对话 `sources` 为 `null`。

5. **LEFT JOIN 兼容软删除**：使用 LEFT JOIN，当对话记录被软删除时，`question`/`answerSummary`/`model`/`sources` 为 `null`，但反馈信息正常返回。前端需对这些字段做 `null` 判断。

6. **排序规则**：按 `f.create_time DESC`（反馈时间降序），最新反馈在前。

7. **分页参数兜底**：`pageNo` 为 `null` 或 `< 1` 时取 `1`；`pageSize` 为 `null` 或 `< 1` 时取 `20`。

8. **过滤参数均为可选**：`feedbackType`/`dislikeReason`/`userId`/`startDate`/`endDate` 均可选，SQL 通过 `<if>` 动态拼接。建议前端组合使用：如「查询某时间段内所有 CODE_ERROR 点踩记录」可传 `feedbackType=DISLIKE&dislikeReason=CODE_ERROR&startDate=20260701&endDate=20260723`。

9. **userId 为 Long 类型**：用户ID为雪花算法生成的 Long，前端传参时注意精度问题（JavaScript Number 最大安全整数 `2^53-1`），建议以字符串形式传输。

10. **与满意度统计接口配合使用**：本接口查看反馈明细，满意度统计接口（`GET /admin/statistics/feedback`）查看聚合指标。建议前端先用统计接口看整体趋势，再用明细接口下钻查看具体反馈。

11. **无审计/限流注解**：本接口为只读查询，未标注 `@LogAction` 和 `@RateLimit`。
