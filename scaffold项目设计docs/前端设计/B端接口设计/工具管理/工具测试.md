# 工具测试

> 模块：测试 Agent 工具调用
>
> 公共约定（网关地址、JWT 认证、统一响应结构 `Result<T>`、错误码、操作审计）见 `总说明.md`，本文档不再重复。

## 功能介绍

本模块提供一个接口，用于在 B 端直接调用某个 Agent 工具，验证工具是否正常工作、参数是否合法。测试调用**不影响线上对话流**，仅返回单次执行的结果。

**核心要点（前端必读）**：

1. **每个工具的测试入参不同**。`{name}` 路径参数决定调用哪个工具，请求体 `params` 字段是一个 JSON 对象，其键名必须与该工具 `@Tool` 方法的**参数名完全一致**（依赖 Java `-parameters` 编译选项保留的真实参数名）。本文档第三部分逐工具列出了合法参数名、类型、含义。
2. **接口始终返回 HTTP 200 + `code=200000`**，工具执行的成功/失败由响应体 `data.success` 字段表示，**不会**以业务错误码（如 `500025`、`500026`）形式返回。
3. **工具内部错误（如路径不在白名单、文件不存在）不会导致 `success=false`**——工具方法内部捕获异常并返回错误 JSON 字符串，此时 `success=true`、错误信息在 `data.result` 字段中。前端必须解析 `result` 才能识别这类错误（详见下文「结果判定规则」）。

**调用链路**：前端 `POST /admin/tools/{name}/test` → 网关 StripPrefix=1 → admin-service `ToolController` → Feign `ToolsApi` → chat-service `ToolsController` → `AdminServiceImpl.testTool` → 反射调用对应工具 Bean 的 `@Tool` 方法。

## 一、接口信息

| 项目 | 内容 |
| --- | --- |
| 路径 | `POST /admin/tools/{name}/test` |
| 认证 | 需要 B 端 token（`userFrom=sys`） |
| 操作审计 | 无（`@LogAction` 未标注，测试操作不记录审计） |
| Content-Type | `application/json` |
| 权限 | B 端登录用户均可访问 |
| 超时控制 | 工具执行超时时间默认 **30 秒**（由 Nacos `scaffold.tool.timeout` 配置，单位秒） |

### 路径参数

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `name` | String | 是 | 工具名称（`@Tool` 方法名，小驼峰）。合法值见下文「二、合法工具名清单」 |

### 请求体（`ToolTestReqDTO`）

| 字段名 | 类型 | 必传 | 校验规则 | 默认值 | 含义 |
| --- | --- | --- | --- | --- | --- |
| `params` | Map&lt;String, Object&gt; | 否 | 无 | `{}`（空 Map） | 测试参数，键为工具方法参数名，值为参数值。不同工具参数不同，详见第三部分。不传或传 `null` 时按空参数处理 |

> **参数名匹配规则**：后端通过反射获取 `@Tool` 方法的参数名（依赖 Spring Boot 3.x 默认开启的 `-parameters` 编译选项），按参数名从 `params` 中取值。若 `params` 中缺少某参数，对应值按 `null` 处理（基本类型会转为默认值 `0`/`false`，详见下文类型转换规则）。

### 请求示例（通用结构）

```http
POST /admin/tools/{name}/test HTTP/1.1
Authorization: Bearer {accessToken}
Content-Type: application/json

{
  "params": {
    "参数名1": "参数值1",
    "参数名2": "参数值2"
  }
}
```

## 二、合法工具名清单

`{name}` 路径参数仅接受以下 7 个值（来自 `ToolRegistryService.ALL_TOOL_NAMES`，与 `@Tool` 方法名一致）：

| name | 功能 | 参数数量 |
| --- | --- | --- |
| `readFile` | 读取指定文件内容 | 1 |
| `searchCode` | 在代码库中搜索类或方法 | 2 |
| `listDir` | 列出指定目录结构 | 1 |
| `searchInFile` | 在文件中搜索关键字 | 2 |
| `nacosConfig` | 查询 Nacos 配置项内容 | 3 |
| `compareConfig` | 对比不同环境配置差异 | 3 |
| `preDeployCheck` | 部署前静态配置检查 | 1 |

> 传入不在上述清单中的 `name`，不会报 HTTP 错误，而是返回 `success=false`、`errorMsg="工具不存在或未注册：name=xxx"`（详见「结果判定规则」）。

## 三、各工具测试入参详解

> 以下参数定义全部来自各工具类 `@Tool` 注解方法的入参签名与方法注释。**参数名必须完全匹配**（区分大小写）。

### 3.1 `readFile` — 读取文件内容

**工具描述**：读取指定文件的内容，参数 filePath 为文件绝对路径。用于查看某个类的完整源码实现。

**参数**：

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `filePath` | String | 是 | 文件绝对路径，必须在路径白名单（Nacos `knowledge.allowed-paths`）范围内 |

**请求示例**：

```json
{
  "params": {
    "filePath": "d:/GitHub/scaffold-ai-assistant/README.md"
  }
}
```

**执行逻辑**：校验路径白名单 → 校验文件存在 → 文件超过 `knowledge.max-file-size`（默认 10MB）则截断返回前 N 字节并附提示 → 返回文件内容字符串。

**常见错误结果（success=true，错误在 result 中）**：

- 路径不在白名单：`result` = `{"error":"路径不在允许范围内: /xxx"}`
- 文件不存在：`result` = `{"error":"文件不存在: /xxx"}`
- 文件过大：`result` = 前 N 字节内容 + `\n\n...[文件过大，已截断，仅显示前 10485760 字节内容，完整内容请使用 SearchInFileTool 搜索关键字]`

---

### 3.2 `searchCode` — 搜索代码

**工具描述**：在代码库中搜索类或方法，参数 keyword 为搜索关键词（类名、方法名等），limit 为返回结果数量。用于查找某个功能在哪里实现。

**参数**：

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `keyword` | String | 是 | 搜索关键词（类名、方法名等），不能为空 |
| `limit` | Integer | 否 | 返回结果数量。不传或 ≤0 时默认 10；上限 50（超过按 50 截断） |

**请求示例**：

```json
{
  "params": {
    "keyword": "Caffeine",
    "limit": 5
  }
}
```

**执行逻辑**：校验 keyword 非空 → 遍历 `knowledge.allowed-paths` 下所有 `.java` 文件 → 逐行匹配 keyword 并打分（类名匹配 3 分 > 方法名匹配 2 分 > 全文匹配 1 分）→ 按分数降序排序 → 截取 limit 条 → 返回 JSON。

**成功返回 `result` 结构**：

```json
{
  "results": [
    {
      "filePath": "相对路径（相对于 knowledge.base-path）",
      "className": "类名",
      "matchedLine": "匹配行内容（已 trim）",
      "lineNumber": 123,
      "context": "匹配行内容"
    }
  ],
  "total": 8
}
```

**常见错误结果（success=true）**：

- keyword 为空：`{"error":"搜索关键词不能为空"}`
- 白名单为空：`{"error":"搜索路径不存在，请检查 knowledge.allowed-paths 配置"}`
- 无匹配：`{"results":[],"message":"未找到匹配的代码"}`

---

### 3.3 `listDir` — 列出目录结构

**工具描述**：列出指定目录的结构，参数 dirPath 为目录绝对路径。用于查看项目或模块的目录组织。

**参数**：

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `dirPath` | String | 是 | 目录绝对路径，必须在路径白名单范围内 |

**请求示例**：

```json
{
  "params": {
    "dirPath": "d:/GitHub/scaffold-ai-assistant/zmbdp-chat"
  }
}
```

**执行逻辑**：校验路径白名单 → 校验目录存在且为目录 → 递归生成 ASCII 目录树（最多 3 层深度，超出显示 `...`）→ 跳过黑名单目录和隐藏文件 → 返回目录树字符串。

**成功返回 `result` 示例**（纯文本，非 JSON）：

```
[D] zmbdp-chat/
  [D] zmbdp-chat-api/
    ...
  [D] zmbdp-chat-service/
    ...
```

> 目录前缀 `[D]`，文件前缀 `[F]`，目录名后带 `/`。

**常见错误结果（success=true）**：

- 路径不在白名单：`{"error":"路径不在允许范围内: /xxx"}`
- 目录不存在：`{"error":"目录不存在: /xxx"}`
- 路径不是目录：`{"error":"路径不是目录: /xxx"}`

---

### 3.4 `searchInFile` — 文件内搜索

**工具描述**：在指定文件中搜索关键字，参数 filePath 为文件绝对路径，keyword 为搜索关键字。用于查找某个方法或变量在文件中的具体位置。

**参数**：

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `filePath` | String | 是 | 文件绝对路径，必须在路径白名单范围内 |
| `keyword` | String | 是 | 搜索关键字，不能为空 |

**请求示例**：

```json
{
  "params": {
    "filePath": "d:/GitHub/scaffold-ai-assistant/zmbdp-chat/zmbdp-chat-service/src/main/java/com/zmbdp/chat/service/tool/ReadFileTool.java",
    "keyword": "maxFileSize"
  }
}
```

**执行逻辑**：校验路径白名单 → 校验 keyword 非空 → 校验文件存在 → 按行读取 → 查找包含 keyword 的行 → 收集匹配行及前后各 2 行上下文 → 最多返回 20 个匹配 → 返回 JSON。

**成功返回 `result` 结构**：

```json
{
  "matches": [
    {
      "lineNumber": 51,
      "lineContent": "匹配行原文",
      "contextBefore": "前 2 行上下文",
      "contextAfter": "后 2 行上下文"
    }
  ],
  "total": 3
}
```

**常见错误结果（success=true）**：

- 路径不在白名单：`{"error":"路径不在允许范围内: /xxx"}`
- keyword 为空：`{"error":"搜索关键字不能为空"}`
- 文件不存在：`{"error":"文件不存在: /xxx"}`
- 文件内容为空：`{"matches":[],"message":"文件内容为空"}`
- 无匹配：`{"matches":[],"message":"未找到匹配内容"}`

---

### 3.5 `nacosConfig` — 查询 Nacos 配置

**工具描述**：查询 Nacos 配置项的内容，参数 dataId 为配置 ID（不带环境后缀，如 share-redis），group 为分组（默认 DEFAULT_GROUP），env 为环境（dev/test/prd）。用于查看某个配置项的具体内容。

**参数**：

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `dataId` | String | 是 | 配置 ID（**不带环境后缀**，如 `share-redis`、`zmbdp-chat-service`）。工具内部自动拼接 `-{env}.yaml` 生成完整 dataId |
| `group` | String | 否 | 分组。不传时默认 `DEFAULT_GROUP` |
| `env` | String | 是 | 环境：`dev` / `test` / `prd`。用于拼接完整 dataId 和命名空间 |

**请求示例**：

```json
{
  "params": {
    "dataId": "share-redis",
    "group": "DEFAULT_GROUP",
    "env": "dev"
  }
}
```

**执行逻辑**：校验 dataId、env 非空 → 构建 `fullDataId = dataId + "-" + env + ".yaml"`（如 `share-redis-dev.yaml`）→ 构建 `namespace = scaffold-ai-assistant-{env}` → 调用 `INacosConfigService.getConfig` 查询 → 返回配置内容字符串。

**成功返回 `result`**：配置文件原始内容（YAML 文本字符串，非 JSON）。

**常见错误结果（success=true）**：

- dataId 为空：`{"error":"dataId不能为空"}`
- env 为空：`{"error":"环境参数env不能为空"}`
- 配置不存在：`{"error":"配置不存在: share-redis-dev.yaml"}`
- Nacos 连接失败：`{"error":"Nacos配置中心连接失败: ..."}`

---

### 3.6 `compareConfig` — 对比配置差异

**工具描述**：对比不同环境的配置差异，参数 dataId 为配置 ID（不带环境后缀，如 share-redis），env1 和 env2 为要对比的两个环境（dev/test/prd），工具内部自动拼接 `-{env}.yaml`。用于排查环境差异问题。

**参数**：

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `dataId` | String | 是 | 配置 ID（**不带环境后缀**，如 `share-redis`） |
| `env1` | String | 是 | 环境 1：`dev` / `test` / `prd` |
| `env2` | String | 是 | 环境 2：`dev` / `test` / `prd`。必须与 `env1` 不同 |

**请求示例**：

```json
{
  "params": {
    "dataId": "share-redis",
    "env1": "dev",
    "env2": "prd"
  }
}
```

**执行逻辑**：校验 dataId、env1、env2 非空且 env1≠env2 → 分别读取两个环境配置 → 解析 YAML 并扁平化为点分 key-value Map → 对比所有 key 的值差异 → 返回差异列表 JSON。

**成功返回 `result` 结构**：

```json
{
  "differences": [
    {
      "key": "spring.data.redis.host",
      "env1Value": "127.0.0.1",
      "env2Value": "10.0.0.1",
      "env1": "dev",
      "env2": "prd"
    },
    {
      "key": "spring.data.redis.port",
      "env1Value": "6379",
      "env2Value": "（不存在）",
      "env1": "dev",
      "env2": "prd"
    }
  ],
  "total": 2
}
```

> 某环境缺失某 key 时，对应值为字符串 `"（不存在）"`。

**常见错误结果（success=true）**：

- dataId 为空：`{"error":"dataId不能为空"}`
- env1/env2 为空：`{"error":"环境参数env1和env2不能为空"}`
- 两环境相同：`{"error":"对比的两个环境不能相同"}`
- 某环境配置不存在：`{"error":"配置不存在: share-redis（dev）"}`
- YAML 格式错误：`{"error":"配置对比失败: 配置文件格式错误: ..."}`

### 3.7 `preDeployCheck` — 部署前检查

**工具描述**：部署前静态配置检查，参数 env 为目标环境（dev/test/prd）。检查 Nacos 配置项完整性、Dockerfile 和 docker-compose 配置是否合理。不做运行时连接测试。

**参数**：

| 参数名 | 类型 | 必传 | 含义 |
| --- | --- | --- | --- |
| `env` | String | 是 | 目标环境：`dev` / `test` / `prd` |

**请求示例**：

```json
{
  "params": {
    "env": "dev"
  }
}
```

**执行逻辑**：校验 env 在 `[dev, test, prd]` 内 → 执行 5 项静态检查 → 汇总结果返回。

**检查项**：

| 检查项 | 检查内容 |
| --- | --- |
| Nacos 共享配置完整性 | 必需配置（`share-mysql`/`share-redis`/`share-token`/`share-rabbitmq`）必须存在；可选配置（`share-caffeine`/`share-map`/`share-filter`/`share-log`/`share-monitor`/`share-xxljob`）缺失则降级 |
| chat-service 专属配置 | `zmbdp-chat-service-{env}.yaml` 必须存在，且包含 `server:` / `spring:` / `ai:` / `milvus:` 配置段 |
| Dockerfile 配置 | `zmbdp-chat-service/Dockerfile` 必须存在，且包含 `FROM` / `EXPOSE` / `ENTRYPOINT` 指令 |
| docker-compose 配置 | docker-compose 文件必须存在（dev/test 用 `docker-compose-mid.yml`，prd 用 `docker-compose-app.yml`），且包含 `services:` 段和 `zmbdp-chat-service` 服务定义 |
| 环境变量完整性 | docker-compose 文件必须包含 `RUN_ENV` / `NACOS_ADDR` 环境变量 |

**成功返回 `result` 结构**：

```json
{
  "checkItems": [
    {
      "item": "Nacos共享配置完整性",
      "status": "PASS",
      "details": "必需配置: 4项；可选配置: 6项",
      "suggestion": ""

    {
      "item": "chat-service专属配置",
      "status": "PASS",
      "details": "配置文件存在，大小: 1024 字节",
      "suggestion": ""
    },
    {
      "item": "Dockerfile配置",
      "status": "FAIL",
      "details": "Dockerfile 存在，12 行，缺失指令: EXPOSE",
      "suggestion": "请补充缺失的 Dockerfile 指令"
    },
    {
      "item": "docker-compose配置",
      "status": "PASS",
      "details": "文件存在，大小: 512 字节",
      "suggestion": ""
    },
    {
      "item": "环境变量完整性",
      "status": "PASS",
      "details": "必需环境变量: RUN_ENV, NACOS_ADDR",
      "suggestion": ""
    }
  ],
  "summary": "4项通过，1项失败",
  "warnings": ["Dockerfile 缺失指令: EXPOSE"],

```

**常见错误结果（success=true）**：

- env 为空：`{"error":"目标环境不能为空"}`
- env 非法：`{"error":"无效的环境: xxx，支持 dev/test/prd"}`
- 检查异常：`{"error":"部署前检查失败: ..."}`

## 四、响应结构

`Result<ToolTestResultVO>`，`data` 为 `ToolTestResultVO`：

| 字段 | 类型 | 含义 | 说明 |
| --- | --- | --- | --- |
| `success` | Boolean | 是否执行成功 | 见下文「结果判定规则」 |
| `result` | String | 工具执行结果 | 工具方法返回值的字符串形式。可能是纯文本（如 readFile 的文件内容、listDir 的目录树）、JSON 字符串（如 searchCode 的结果）、YAML 文本（如 nacosConfig 的配置内容），或错误 JSON 字符串（如 `{"error":"..."}`） |
| `duration` | Long | 执行耗时 | 单位毫秒。工具不存在/方法未找到/参数转换失败时为 `0` |
| `errorMsg` | String | 失败时的错误信息 | 仅当 `success=false` 时有值；`success=true` 时为 `null` |

---

## 五、结果判定规则（前端必读）

工具测试接口**始终返回 HTTP 200 + `code=200000`**，工具执行的成功与否由 `data.success` 字段表示。前端**不能**仅靠 `code` 判断工具是否执行成功，必须结合 `success` 和 `result` 综合判断。

### 5.1 `success=false` 的场景（错误信息在 `errorMsg`）

此时 `result` 通常为 `null`，`errorMsg` 描述失败原因：

| 场景 | errorMsg 示例 | duration |
| --- | --- | --- |
| 工具名不存在或未注册 | `工具不存在或未注册：name=xxx` | `0` |
| 工具类中未找到 `@Tool` 方法 | `工具方法未找到：在 XxxTool 中未找到 @Tool 注解方法` | `0` |
| 参数类型转换失败 | `参数 filePath 类型转换失败：123 → String` | `0` |
| 工具执行超时 | `工具执行超时（超过 30 秒）` | 实际耗时 |
| 方法调用抛出未捕获异常（罕见） | `工具执行失败：{异常 message}` | 实际耗时 |

### 5.2 `success=true` 但 `result` 含错误的场景（前端必须解析 result）

工具方法内部捕获异常并返回错误 JSON 字符串，此时 `success=true`、`errorMsg=null`，**错误信息在 `result` 字段中**。常见情况：

- **路径不在白名单**（readFile/listDir/searchInFile）：`result` = `{"error":"路径不在允许范围内: /xxx"}`
- **文件/目录不存在**：`result` = `{"error":"文件不存在: /xxx"}`
- **参数为空**（如 searchCode 的 keyword 为空）：`result` = `{"error":"搜索关键词不能为空"}`
- **Nacos 配置不存在**：`result` = `{"error":"配置不存在: share-redis-dev.yaml"}`
- **业务无结果**（非错误）：如 `{"results":[],"message":"未找到匹配的代码"}`

> **重要**：`ResultCode` 中定义的 `500025 工具调用失败`、`500026 路径不在白名单` **不会**由本接口返回。路径白名单违规等工具内部错误全部以 `success=true` + `result` 内嵌错误 JSON 的形式返回。前端若需识别这类错误，应尝试 `JSON.parse(result)`，若解析出含 `error` 字段的对象，则视为工具执行出错。

### 5.3 前端判定流程建议

```
1. 若 HTTP code !== 200000 → 网络/鉴权/系统错误，按通用错误处理
2. 若 data.success === false → 工具未执行成功，展示 data.errorMsg
3. 若 data.success === true：
   a. 尝试 JSON.parse(data.result)
   b. 若解析成功且含 error 字段 → 工具内部错误，展示 error 值
   c. 否则 → 工具执行成功，按工具类型渲染 result（文本/JSON/YAML）
```

## 六、请求参数类型转换规则

后端 `buildMethodArgs` 从 `params` Map 中按参数名取值，并通过 `convertArgValue` 转换为目标 Java 类型。支持类型：

| Java 类型 | 取值说明 |
| --- | --- |
| `String` | 调用 `value.toString()`；`null` 时返回 `null` |
| `Integer` / `int` | `Integer.parseInt(strValue)`；`null` 时基本类型返回 `0`，包装类型返回 `null` |
| `Long` / `long` | `Long.parseLong(strValue)`；`null` 时基本类型返回 `0`，包装类型返回 `null` |
| `Boolean` / `boolean` | `Boolean.parseBoolean(strValue)`；`null` 时基本类型返回 `false`，包装类型返回 `null` |
| `Double` / `double` | `Double.parseDouble(strValue)`；`null` 时基本类型返回 `0.0`，包装类型返回 `null` |
| `Float` / `float` | `Float.parseFloat(strValue)`；`null` 时基本类型返回 `0.0`，包装类型返回 `null` |

> - 数值类型参数（如 `limit`、`env` 等）既可传 JSON 数字（`5`）也可传数字字符串（`"5"`），后端统一 `toString()` 后解析。
> - 类型转换失败会抛 `ServiceException`，返回 `success=false`、`errorMsg="参数 {参数名} 类型转换失败：{原值} → {目标类型}"`。

## 八、错误码说明

| code | errMsg | 触发场景 | HTTP 状态码 |
| --- | --- | --- | --- |
| `200000` | 操作成功 | 接口正常返回（工具是否执行成功看 `data.success`） | 200 |
| `500000` | 测试工具调用失败 / 服务繁忙请稍后重试 | admin-service 与 chat-service 之间 Feign 调用失败，或 chat-service 内部异常（非工具执行本身的问题） | 500 |

> **本接口不会返回 AI 模块专属错误码**：
> - `500025 工具调用失败`：定义于 `ResultCode`，但本接口不抛出。工具执行失败统一封装在 `data.success=false` + `data.errorMsg` 中返回。
> - `500026 路径不在白名单`：定义于 `ResultCode`，但本接口不抛出。路径白名单违规由工具方法内部捕获，以 `data.success=true` + `data.result` 内嵌 `{"error":"路径不在允许范围内: ..."}` 形式返回。
>
> 前端只需处理 `200000`（接口正常）和 `500000`（接口异常）两种 code，工具执行结果全部通过 `data` 判定。

## 十、前端注意事项

1. **参数名必须完全匹配**：`params` 的键名要与本文档第三部分列出的参数名完全一致（区分大小写）。例如 `readFile` 的参数是 `filePath`（非 `path`/`filepath`），`searchCode` 是 `keyword` + `limit`，`compareConfig` 是 `env1`/`env2`（非 `envOne`/`envTwo`）。
2. **不同工具渲染方式不同**：`result` 可能是纯文本（readFile/listDir）、JSON 字符串（searchCode/searchInFile/compareConfig/preDeployCheck）、YAML 文本（nacosConfig），前端应根据工具名选择渲染方式：
   - `readFile`：代码/文本预览
   - `listDir`：等宽字体显示目录树
   - `searchCode`/`searchInFile`/`compareConfig`/`preDeployCheck`：解析 JSON 后表格/列表展示
   - `nacosConfig`：YAML 代码高亮
3. **必须区分「接口成功」与「工具执行成功」**：接口 `code=200000` 只代表请求被正常处理，工具是否真的执行成功要看 `data.success`，且 `success=true` 时仍可能存在工具内部错误（在 `result` 中）。
4. **超时处理**：工具执行默认 30 秒超时，超时返回 `success=false` + `errorMsg="工具执行超时（超过 30 秒）"`。前端请求超时建议设置 ≥ 35 秒，并在测试期间显示 loading。
5. **路径白名单限制**：`readFile`/`listDir`/`searchInFile` 的路径必须在 Nacos `knowledge.allowed-paths` 白名单内，且不能命中 `knowledge.path-blacklist` 黑名单（如 `.git`、`/etc`、`/tmp`）和隐藏文件。前端可在测试表单旁提示用户「路径需在白名单内」。
6. **`searchCode` 仅搜索 `.java` 文件**，且搜索范围由 `knowledge.allowed-paths` 决定，返回路径为相对路径（相对于 `knowledge.base-path`）。
7. **`nacosConfig` 的 dataId 不带环境后缀**：传 `share-redis` 而非 `share-redis-dev.yaml`，环境后缀由 `env` 参数自动拼接。
8. **`compareConfig` 的 env1 和 env2 不能相同**，相同会返回错误 JSON。
9. 建议前端测试页面采用「选择工具 → 动态渲染该工具的参数表单 → 提交测试 → 按工具类型渲染结果」的交互模式，参数表单依据本文档第三部分定义。
